## Ключові терміни

**stdin (standard input)** — стандартний потік введення. Зазвичай — клавіатура.

**stdout (standard output)** — стандартний потік виведення. Зазвичай — екран.

**stderr (standard error)** — потік для повідомлень про помилки. Також екран, але окремий від stdout.

**Blocking I/O (блокуючий ввід-вивід)** — операція, що зупиняє програму до завершення (наприклад, очікування вводу).

**Buffer (буфер)** — тимчасове сховище, де накопичуються дані перед обробкою.

**Result\<T, E\>** — тип для операцій, що можуть провалитись: `Ok(value)` або `Err(error)`.

**parse()** — метод перетворення рядка у значення іншого типу.

**REPL (Read-Eval-Print Loop)** — цикл "зчитати → обчислити → вивести".

## Мотиваційний кейс

Наземна станція управління БПЛА — це інтерактивна програма. Оператор вводить: "зліт на висоту 120", "курс 270", "повернення". Програма зчитує команду, аналізує, виконує дію, виводить результат. Цей цикл повторюється безперервно. Усі інтерактивні програми — від bash до чатів — побудовані на тому самому принципі: зчитати → проаналізувати → виконати → відповісти → повторити. І починається все з фундаментального питання: як програма отримує дані від людини?

## 10.1. Потоки: як програма спілкується зі світом

Кожна програма, запущена в терміналі, отримує від операційної системи три канали зв'язку із зовнішнім світом. Ці канали називаються стандартними потоками.

```mermaid
graph LR
    USER["Користувач\n(клавіатура)"]
    PROG["Програма"]
    SCREEN["Екран\n(термінал)"]

    USER -->|"stdin"| PROG
    PROG -->|"stdout"| SCREEN
    PROG -->|"stderr"| SCREEN
```

**stdin** — вхідний потік. Коли програма "читає з stdin", вона звертається до клавіатури (або до файлу, якщо ввід перенаправлений). Це єдиний спосіб отримати дані від користувача в консольній програмі.

**stdout** — вихідний потік для нормального виводу. Кожен `println!`, що ви писали з Розділу 3, виводив текст саме сюди.

**stderr** — вихідний потік для помилок. `eprintln!` (який ми згадували у Розділі 3) пише сюди.

Навіщо два вихідних потоки? Здавалося б, і stdout, і stderr виводять текст на той самий екран. Різниця проявляється при перенаправленні. Команда `./program > output.txt` перенаправляє stdout у файл, а stderr залишається на екрані. Результат: нормальний вивід зберігається у файл, а повідомлення про помилки оператор бачить одразу. Для БПЛА: телеметрія пишеться в лог (stdout → файл), а аварійні попередження видимі негайно (stderr → екран).

Ще одна важлива деталь: потоки буферизовані. stdout у Rust накопичує текст у внутрішньому буфері і виводить на екран порціями — зазвичай при зустрічі символу `\n` (тобто при `println!`). `print!` без `\n` може не вивести текст одразу — він залишиться в буфері. Для примусового виводу використовується `flush()`. stderr, навпаки, не буферизований — `eprintln!` виводить одразу, без затримки. Для аварійних повідомлень це критично: оператор має побачити "БАТАРЕЯ КРИТИЧНА" негайно, а не після того, як буфер заповниться.

## 10.2. Зчитування рядка: read_line

Зчитування тексту з клавіатури — це blocking operation (блокуюча операція). Коли програма викликає `read_line()`, вона повністю зупиняється і чекає, поки користувач натисне Enter. Ніякий інший код не виконується, поки ввід не отриманий. Це як телефонний дзвінок: ви не продовжуєте розмову, поки співрозмовник не відповів.

Процес зчитування працює так. Користувач друкує текст — символи накопичуються в буфері операційної системи. Коли натискає Enter — ОС додає символ нового рядка `\n` і передає весь буфер програмі. `read_line()` записує отримані байти (включно з `\n`) у `String`, яку ви надали, і повертає `Result` з кількістю прочитаних байтів.

Три нюанси, які потрібно розуміти до написання коду.

Перший: `read_line()` не створює новий рядок — він дописує до існуючого. Тому потрібно заздалегідь створити порожній `String::new()` і передати його як мутабельне посилання `&mut`.

Другий: прочитаний текст містить `\n` наприкінці. Якщо користувач набрав "ЗЛІТ" і натиснув Enter, у буфері буде "ЗЛІТ\n". При порівнянні `input == "ЗЛІТ"` результат буде `false`. Тому потрібен `trim()`.

Третій: `read_line()` повертає `Result`, а не просто число. Зчитування може провалитись (наприклад, якщо stdin закритий або перенаправлений з пошкодженого файлу). Rust вимагає обробити цю можливість.

Найпростіший приклад — програма, що запитує ім'я:

```rust
use std::io;

fn main() {
    println!("Як вас звати?");

    let mut name = String::new();
    io::stdin().read_line(&mut name).expect("Помилка зчитування");

    let name = name.trim();
    println!("Привіт, {}!", name);
}
```

Розберемо кожен елемент. `use std::io` — імпорт модуля введення-виведення. `String::new()` — порожній мутабельний рядок, контейнер для вводу. `io::stdin()` — доступ до потоку stdin. `.read_line(&mut name)` — зчитування рядка в `name` через мутабельне посилання (`&mut` — preview з Частини II, поки використовуйте як шаблон). `.expect("...")` — обробка Result: якщо Ok — продовжити, якщо Err — panic з повідомленням. `name.trim()` — видалення `\n` та пробілів (shadowing: нова `name` типу `&str` затіняє стару `String`).

## 10.3. Чому trim() критичний

Ця деталь заслуговує окремого пояснення, бо забутий `trim()` — одна з найчастіших помилок при роботі з вводом. Проблема невидима: ви дивитесь на екран, бачите "ЗЛІТ", і не розумієте, чому порівняння дає `false`.

Справа в тому, що символ `\n` — невидимий. Він не відображається як символ, а лише переміщує курсор на новий рядок. Тому `"ЗЛІТ\n"` на екрані виглядає як `"ЗЛІТ"` — але для порівняння це різні рядки.

```rust
use std::io;

fn main() {
    let mut input = String::new();
    println!("Введіть 'так':");
    io::stdin().read_line(&mut input).expect("помилка");

    // БЕЗ trim — не працює!
    println!("Без trim: введено '{}', == 'так'? {}", input, input == "так");

    // З trim — працює
    let clean = input.trim();
    println!("З trim:  введено '{}', == 'так'? {}", clean, clean == "так");
}
```

Якщо користувач ввів "так", вивід буде:

```
Без trim: введено 'так
', == 'так'? false
З trim:  введено 'так', == 'так'? true
```

Зверніть увагу на "Без trim": після слова "так" закриваюча лапка з'їхала на новий рядок — це `\n` в дії. Правило без винятків: завжди `trim()` перед порівнянням або парсингом.

## 10.4. Перетворення тексту в число: parse та Result

### Проблема

Користувач завжди вводить текст. Навіть коли набирає "120" — для `read_line` це рядок із символів '1', '2', '0', а не число 120. Щоб отримати число, потрібно перетворення — парсинг.

Але парсинг може провалитись. Що робити, якщо користувач ввів "abc" замість числа? Або "12.5" коли очікувалось ціле? Або порожній рядок?

Різні мови вирішують це по-різному. C: функція `atoi("abc")` мовчки поверне 0 — жодного попередження, що ввід хибний. Python: виняток `ValueError`, який "вилетить" і зупинить програму, якщо не перехоплений. Rust: тип `Result`, що змушує програміста явно вирішити, що робити при помилці.

### Result: перше знайомство

`Result` — це тип з двома варіантами (enum, який ви детально вивчите у Розділі 17):

- `Ok(value)` — операція успішна, ось результат
- `Err(error)` — операція провалилась, ось опис помилки

Коли `.parse::<i32>()` отримує "120" — повертає `Ok(120)`. Коли отримує "abc" — повертає `Err(ParseIntError)`. Rust не дає "просто число" — він дає Result, і ви мусите вирішити, що робити з обома варіантами.

На цьому етапі курсу ми використовуватимемо два підходи.

**expect("повідомлення")** — простий підхід для прототипів. Витягує значення з Ok. Якщо Err — panic з вашим повідомленням. Програма аварійно завершується, але з зрозумілим повідомленням.

```rust
use std::io;

fn main() {
    println!("Введіть висоту (метри):");
    let mut input = String::new();
    io::stdin().read_line(&mut input).expect("Помилка зчитування");

    let altitude: f64 = input.trim().parse().expect("Це не число!");
    println!("Висота: {:.1} м", altitude);
}
```

При введенні "120.5" — працює: `parse` повертає `Ok(120.5)`, `expect` витягує 120.5. При введенні "abc" — panic: `parse` повертає `Err(...)`, `expect` зупиняє програму з "Це не число!".

**match** — правильний підхід для інтерактивних програм. Перевіряє обидва варіанти без panic:

```rust
use std::io;

fn main() {
    println!("Введіть висоту (метри):");
    let mut input = String::new();
    io::stdin().read_line(&mut input).expect("Помилка зчитування");

    match input.trim().parse::<f64>() {
        Ok(altitude) => println!("Висота: {:.1} м", altitude),
        Err(_) => println!("Помилка: '{}' — не число", input.trim()),
    }
}
```

`match` перевіряє, який варіант Result: якщо `Ok` — витягує значення в змінну `altitude` і використовує. Якщо `Err` — виводить повідомлення і програма продовжує працювати. Жодного panic. Це те, що потрібно для бортового комп'ютера БПЛА: неправильна команда оператора не повинна "валити" всю систему.

Повноцінну обробку помилок ми вивчимо у Частині III (Розділи 22–25). Поки запам'ятайте: `expect` — для прототипів та навчання, `match` — для інтерактивних програм.

### Синтаксис parse

`parse()` може перетворити рядок у різні типи. Тип потрібно вказати — або через "turbofish" `::<T>`, або через анотацію змінної:

```rust
fn main() {
    let text = "42";
    let as_i32 = text.parse::<i32>().expect("не i32");       // turbofish
    let as_f64: f64 = text.parse().expect("не f64");          // анотація
    let as_u8 = text.parse::<u8>().expect("не u8");           // turbofish
    println!("i32={}, f64={}, u8={}", as_i32, as_f64, as_u8);
}
```

## 10.5. Цикл команд: побудова REPL крок за кроком

Інтерактивна програма — це не одноразове зчитування, а безперервний цикл. Патерн REPL (Read-Eval-Print Loop) повторює: зчитати ввід → проаналізувати → виконати → показати результат → повторити. Побудуємо його поступово.

### Крок 1: нескінченний цикл зчитування

Найпростіший REPL — цикл, що читає команди і виводить їх назад (echo):

```rust
use std::io;
use std::io::Write;

fn main() {
    println!("Введіть команду (або 'вихід'):");
    loop {
        print!("> ");
        io::stdout().flush().expect("flush");

        let mut input = String::new();
        io::stdin().read_line(&mut input).expect("помилка");
        let command = input.trim();

        if command == "вихід" {
            println!("До побачення!");
            break;
        }

        println!("Отримано: '{}'", command);
    }
}
```

Зверніть увагу на `flush()`. `print!("> ")` не містить `\n`, тому текст залишається в буфері stdout і не з'являється на екрані. `flush()` примусово виводить буфер — символ `>` стає видимим до того, як програма зупиниться на `read_line`. Без `flush` користувач бачитиме порожній екран і не знатиме, що програма чекає ввід.

Також зверніть увагу: `String::new()` створюється всередині циклу. Якби він був зовні — кожен `read_line` дописував би до попереднього вводу, і після трьох команд у `input` було б три рядки.

### Крок 2: аналіз команд

Додаємо розпізнавання команд через `if`/`else if`:

```rust
use std::io;
use std::io::Write;

fn main() {
    let mut is_flying = false;
    let mut battery: u8 = 100;

    println!("Команди: зліт, посадка, статус, вихід");

    loop {
        print!("> ");
        io::stdout().flush().expect("flush");
        let mut input = String::new();
        io::stdin().read_line(&mut input).expect("помилка");
        let cmd = input.trim().to_lowercase();

        if cmd.is_empty() { continue; }
        if cmd == "вихід" { break; }

        if cmd == "зліт" {
            if is_flying { println!("Вже у повітрі!"); }
            else {
                is_flying = true;
                battery = battery.saturating_sub(10);
                println!("Зліт! Батарея: {}%", battery);
            }
        } else if cmd == "посадка" {
            if !is_flying { println!("Вже на землі!"); }
            else {
                is_flying = false;
                battery = battery.saturating_sub(5);
                println!("Посадка. Батарея: {}%", battery);
            }
        } else if cmd == "статус" {
            println!("Літаємо: {}, батарея: {}%", is_flying, battery);
        } else {
            println!("Невідома команда: '{}'", cmd);
        }
    }
}
```

Нормалізація `to_lowercase()` дозволяє приймати "ЗЛІТ", "Зліт", "зліт" однаково. `is_empty()` + `continue` пропускає порожній ввід (коли користувач просто натиснув Enter).

### Крок 3: команди з параметрами

Для команд із числовим параметром ("висота 120") потрібен парсинг. Ось як обробити одну таку команду:

```rust
// Всередині циклу, замість простого else if:
if cmd.starts_with("висота ") {
    let value_str = cmd.replace("висота ", "");
    match value_str.trim().parse::<f64>() {
        Ok(alt) => {
            if alt < 0.0 || alt > 400.0 {
                println!("Висота має бути 0–400 м!");
            } else {
                println!("Нова висота: {:.0} м", alt);
            }
        }
        Err(_) => println!("Невірний формат. Приклад: висота 120"),
    }
}
```

`starts_with("висота ")` перевіряє початок рядка. `replace("висота ", "")` видаляє команду, залишаючи число. `match parse` обробляє і успіх, і помилку без panic. Якщо число валідне — перевіряємо діапазон. Якщо ні — повідомлення і `continue`.

Цей патерн (starts_with → replace → parse → match) — простий, але ефективний спосіб парсингу текстових команд із параметрами. Для складніших парсерів існують бібліотеки (clap, structopt), але для нашого рівня це достатньо.

## Troubleshooting

**Програма не чекає вводу — одразу завершується.**

Перевірте, що `read_line` викликається з `&mut`. Без нього — помилка компіляції, без виклику — програма не зупиняється.

**Введена команда не розпізнається.**

Забутий `trim()`. Додайте `println!("Debug: '{:?}'", input);` — побачите невидимий `\n`.

**`error: use of moved value` при повторному read_line.**

Створюйте `String::new()` всередині циклу. Або додайте `input.clear()` перед `read_line`.

**Символ `>` не з'являється перед вводом.**

`print!` без `\n` буферизується. Додайте `use std::io::Write;` та `io::stdout().flush().expect("flush");`.

**parse повертає помилку для правильного числа.**

Забутий `trim()`. "120\n" не парситься — потрібен "120".

**`error[E0308]: mismatched types` при parse.**

Вкажіть тип: `.parse::<f64>()` або `let x: f64 = ....parse()...`.

## Додатково

### Перенаправлення потоків

У терміналі потоки можна перенаправляти. `./drone > log.txt` — stdout у файл. `./drone 2> errors.txt` — stderr у файл. `echo "зліт" | ./drone` — stdin з рядка замість клавіатури. `./drone < commands.txt` — stdin з файлу. Це означає: ваша інтерактивна програма може працювати і без людини — подаючи команди з файлу. Це основа автоматизованого тестування.

### unwrap vs expect vs match

Три рівні обробки Result, від найпростішого до найправильнішого:

`unwrap()` — panic без пояснення. Ніколи не використовуйте в продакшені. Навіть для прототипів — expect краще.

`expect("повідомлення")` — panic з вашим текстом. Прийнятно для навчання та прототипів. Неприйнятно для бортового ПЗ.

`match Ok/Err` — повноцінна обробка без panic. Правильний підхід для інтерактивних програм. Стане стандартним з Частини III.

## Резюме

Три потоки: stdin (ввід), stdout (вивід), stderr (помилки). `println!` → stdout, `eprintln!` → stderr. stdout буферизований (потрібен `\n` або `flush`), stderr — ні.

`read_line(&mut buffer)` зчитує рядок із `\n`. Завжди `trim()` перед порівнянням чи парсингом. Створюйте `String::new()` всередині циклу.

`.parse::<T>()` повертає `Result`: `Ok(value)` або `Err(error)`. `expect` — panic з повідомленням (для прототипів). `match Ok/Err` — обробка без panic (для інтерактивних програм).

REPL: `loop` + `read_line` + `trim` + аналіз + `break`/`continue`.

## Що далі

Агент тепер інтерактивний. Але код став великим: один `fn main()` на десятки рядків із ланцюжком `else if`. Додавання нової команди вимагає зміни одного блоку. У Розділі 11 ми навчимось розбивати код на функції — іменовані блоки, що роблять одну конкретну річ і викликаються з будь-якого місця.

# Розділ 22. Колекції: HashMap та HashSet

## Анотація

У Розділі 21 вектор дав агенту динамічну пам'ять — списки довільної довжини. Але вектор організований як послідовність з числовим індексом: "елемент номер 0", "елемент номер 7". Коли потрібно знайти агента за ідентифікатором "SCOUT-07" або перевірити, чи відвідана зона з координатами (5, 3) — доведеться перебирати весь вектор, порівнюючи кожен елемент. Для десятка записів це непомітно. Для тисяч — це різниця між миттєвою відповіддю та помітною затримкою.

`HashMap<K, V>` вирішує цю задачу: зберігає пари "ключ → значення" з доступом за ключем за амортизований O(1). `HashSet<T>` — спрощений варіант, що зберігає лише ключі (без значень) і дає миттєву перевірку наявності. Ці дві колекції разом із Vec покривають більшість потреб у зберіганні даних.

## Цілі навчання

Після опрацювання цього розділу студент зможе:

1. Створити HashMap, додати, прочитати, оновити та видалити пари ключ-значення.
2. Використати `entry()` API для умовної вставки та модифікації.
3. Пояснити вимоги до ключів: `Eq + Hash`.
4. Ітерувати по HashMap трьома способами.
5. Використати HashSet для множинних операцій: перетин, об'єднання, різниця.
6. Пояснити ownership ключів та значень у HashMap.
7. Обрати між HashMap, BTreeMap та Vec для конкретної задачі.

## Ключові терміни

**HashMap\<K, V\>** — колекція пар "ключ-значення", де кожен ключ унікальний, з доступом за ключем за O(1).

**HashSet\<T\>** — множина унікальних елементів з перевіркою наявності за O(1).

**Hash function (хеш-функція)** — функція, що перетворює ключ довільного розміру у число фіксованого розміру (хеш), яке визначає позицію у таблиці.

**Entry API** — інтерфейс HashMap для атомарних операцій "перевірити наявність → вставити або модифікувати".

**BTreeMap\<K, V\>** — впорядкована мапа на основі B-дерева, де ключі зберігаються у відсортованому порядку.

---

## Мотиваційний кейс

Під час пошуково-рятувальної операції рій БПЛА сканує зону стихійного лиха. Кожен дрон фіксує координати виявлених постраждалих, зруйнованих будівель, заблокованих доріг. Координатор рою повинен миттєво відповісти на запити: "Чи хтось вже перевірив будівлю на вулиці Шевченка, 15?" (перевірка наявності за ключем), "Скільки постраждалих виявлено в секторі B4?" (лічильник за ключем), "Який дрон останнім сканував зону (47.5, 35.2)?" (значення за ключем-координатами).

З вектором кожен запит потребує лінійного перебору. При 50 000 записів за годину роботи рою — це відчутне навантаження. HashMap дає відповідь за одну операцію, незалежно від кількості записів. Різниця між O(N) та O(1) стає критичною, коли від швидкості відповіді залежать людські життя.

## 22.1. Концепція: ключ → значення

Масив та вектор зберігають елементи, доступні за числовим індексом. HashMap зберігає пари, де ключ може бути будь-яким типом, що реалізує `Eq` та `Hash`: рядок, число, кортеж, власний тип. Значення — будь-який тип.

Аналогія: телефонна книга. Ви шукаєте номер (значення) за ім'ям (ключ). Не за позицією "третій запис зверху" — а саме за ім'ям. Два записи не можуть мати однакове ім'я — ключ унікальний. Якщо додати новий номер для існуючого імені — старий перезаписується.

Як це працює всередині? HashMap обчислює хеш ключа — число, що визначає "кошик" (bucket) у внутрішньому масиві. Замість перебору всіх елементів, функція одразу знає, де шукати. Для рядка "SCOUT-07" хеш-функція може повернути число 42381. HashMap бере залишок від ділення на кількість кошиків (наприклад, 42381 % 128 = 77) і дивиться у кошик 77. Там — або шуканий елемент, або ланцюжок з кількох елементів (колізія), який потрібно перебрати. Але зазвичай у кошику 1–2 елементи, тому пошук — фактично O(1).

Вам не потрібно розуміти деталі хеш-функції, щоб використовувати HashMap. Достатньо знати: пошук, вставка та видалення за ключем працюють за O(1) в середньому — незалежно від того, чи у мапі 10 записів чи мільйон.

## 22.2. Створення та базові операції

HashMap живе у модулі `std::collections`, тому потрібен `use`:

```rust
use std::collections::HashMap;

fn main() {
    // Створення порожньої мапи
    let mut altitudes: HashMap<String, f64> = HashMap::new();
    
    // Вставка пар ключ-значення
    altitudes.insert(String::from("SCOUT-01"), 120.0);
    altitudes.insert(String::from("SCOUT-02"), 85.5);
    altitudes.insert(String::from("RECON-01"), 200.0);
    
    println!("Кількість агентів: {}", altitudes.len()); // 3
    
    // Читання за ключем — повертає Option<&V>
    match altitudes.get("SCOUT-01") {
        Some(alt) => println!("SCOUT-01 на висоті {:.0} м", alt),
        None => println!("Агент не знайдений"),
    }
    
    // Перезапис: insert з існуючим ключем замінює значення
    altitudes.insert(String::from("SCOUT-01"), 150.0);
    println!("Нова висота SCOUT-01: {:.0}", altitudes["SCOUT-01"]);
}
```

Розберемо ключові моменти. `HashMap::new()` створює порожню мапу. Тип ключа та значення виводиться з першого `insert`. `insert` приймає ключ та значення за значенням (ownership переходить до мапи). `get` приймає посилання на ключ і повертає `Option<&V>` — None якщо ключ відсутній, Some з посиланням на значення якщо присутній.

Зверніть увагу: `get("SCOUT-01")` приймає `&str`, хоча ключі — `String`. Це працює завдяки тому, що `String` реалізує `Borrow<str>`, і HashMap вміє шукати String-ключі за &str-запитом. Не потрібно створювати String лише для пошуку.

Індексація `altitudes["SCOUT-01"]` теж працює, але панікує при відсутньому ключі — як `vec[i]` при невалідному індексі. Для безпечного коду завжди використовуйте `get`.

Видалення через `remove`:

```rust
use std::collections::HashMap;

fn main() {
    let mut registry: HashMap<String, String> = HashMap::new();
    registry.insert(String::from("D-001"), String::from("Розвідник"));
    registry.insert(String::from("D-002"), String::from("Транспорт"));
    registry.insert(String::from("D-003"), String::from("Штурмовик"));
    
    // remove повертає Option<V> — значення видаленого елемента
    if let Some(role) = registry.remove("D-002") {
        println!("Видалено агент D-002 з роллю: {}", role);
    }
    
    println!("Залишилось: {}", registry.len()); // 2
    println!("D-002 є? {}", registry.contains_key("D-002")); // false
}
```

`remove` повертає `Option<V>` — значення видаленого елемента за наявності. `contains_key` — швидка перевірка наявності ключа без отримання значення.

## 22.3. Entry API: умовна вставка та модифікація

Типова задача: "якщо ключ є — оновити значення, якщо немає — вставити початкове". Наприклад, підрахунок кількості виявлень у кожній зоні. Наївний підхід:

```rust
use std::collections::HashMap;

fn main() {
    let zones = vec!["A1", "B2", "A1", "C3", "A1", "B2"];
    let mut counts: HashMap<&str, u32> = HashMap::new();
    
    // Наївний підхід: get + insert
    for zone in &zones {
        let count = match counts.get(zone) {
            Some(&c) => c + 1,
            None => 1,
        };
        counts.insert(zone, count);
    }
    
    println!("{:?}", counts); // {"A1": 3, "B2": 2, "C3": 1}
}
```

Це працює, але виконує два пошуки за ключем: один у get, другий у insert. Entry API об'єднує їх в одну операцію:

```rust
use std::collections::HashMap;

fn main() {
    let zones = vec!["A1", "B2", "A1", "C3", "A1", "B2"];
    let mut counts: HashMap<&str, u32> = HashMap::new();
    
    for zone in &zones {
        let counter = counts.entry(zone).or_insert(0);
        *counter += 1;
    }
    
    println!("{:?}", counts); // {"A1": 3, "B2": 2, "C3": 1}
}
```

`entry(key)` повертає enum `Entry`, що представляє або зайнятий слот (`Occupied`), або вільний (`Vacant`). `or_insert(default)` — якщо слот вільний, вставляє значення за замовчуванням; якщо зайнятий — нічого не робить. В обох випадках повертає `&mut V` — мутабельне посилання на значення, яке можна одразу модифікувати.

Цей патерн — підрахунок частоти елементів — настільки поширений, що варто запам'ятати його як ідіому: `*map.entry(key).or_insert(0) += 1`.

Є кілька варіантів Entry API:

```rust
use std::collections::HashMap;

fn main() {
    let mut config: HashMap<String, String> = HashMap::new();
    
    // or_insert: вставити конкретне значення, якщо ключ відсутній
    config.entry(String::from("mode")).or_insert(String::from("patrol"));
    
    // or_insert_with: вставити результат функції (ледача ініціалізація)
    config.entry(String::from("id")).or_insert_with(|| {
        // Виконується лише якщо ключ відсутній
        format!("DRONE-{}", rand_id())
    });
    
    // or_default: вставити Default::default() (для String — порожній рядок)
    config.entry(String::from("notes")).or_default();
    
    println!("{:?}", config);
}

fn rand_id() -> u32 { 42 } // заглушка
```

`or_insert_with` приймає замикання, яке викликається лише якщо ключ відсутній. Це важливо, коли обчислення значення за замовчуванням дороге: з `or_insert(expensive_function())` функція виконається завжди, навіть якщо ключ вже є і результат не потрібен. З `or_insert_with(|| expensive_function())` — лише коли дійсно треба.

## 22.4. Ітерація по HashMap

HashMap підтримує три способи ітерації, аналогічно до Vec:

```rust
use std::collections::HashMap;

fn main() {
    let mut stats: HashMap<&str, u32> = HashMap::new();
    stats.insert("виявлено", 15);
    stats.insert("скановано", 42);
    stats.insert("пропущено", 3);
    
    // Іммутабельна ітерація: (&K, &V)
    println!("Статистика місії:");
    for (key, value) in &stats {
        println!("  {}: {}", key, value);
    }
    
    // Мутабельна ітерація: (&K, &mut V) — ключі не змінюються!
    for (_, value) in &mut stats {
        *value *= 2; // подвоїти всі значення
    }
    
    // Споживаюча ітерація: (K, V) — HashMap знищується
    for (key, value) in stats {
        println!("{} = {}", key, value);
    }
    // stats тут вже не існує
}
```

Важлива деталь: при мутабельній ітерації змінювати можна лише значення, не ключі. Зміна ключа зіпсувала б хеш-таблицю — елемент опинився б у неправильному кошику і його не можна було б знайти.

Порядок ітерації у HashMap не визначений і може відрізнятись між запусками програми. Якщо потрібен конкретний порядок — відсортуйте ключі окремо або використайте BTreeMap.

## 22.5. Ownership ключів та значень

HashMap володіє і ключами, і значеннями. Для типів із ownership (String, Vec) це означає переміщення при insert:

```rust
use std::collections::HashMap;

fn main() {
    let name = String::from("SCOUT-01");
    let role = String::from("Розвідник");
    
    let mut registry: HashMap<String, String> = HashMap::new();
    registry.insert(name, role);
    
    // println!("{}", name);  // ПОМИЛКА: name переміщено в HashMap
    // println!("{}", role);  // ПОМИЛКА: role теж переміщено
    
    // Але можна дістати через get:
    if let Some(r) = registry.get("SCOUT-01") {
        println!("Роль: {}", r);
    }
}
```

Після `insert(name, role)` обидва String належать HashMap. Оригінальні змінні `name` та `role` невалідні. Це логічно: якщо б name залишалась доступною і хтось змінив рядок — ключ у HashMap став би некоректним (хеш уже обчислений за старим значенням).

Для Copy-типів (i32, f64, bool) insert копіює значення, оригінал залишається доступним:

```rust
use std::collections::HashMap;

fn main() {
    let zone_id = 42_u32;
    let altitude = 150.0_f64;
    
    let mut data: HashMap<u32, f64> = HashMap::new();
    data.insert(zone_id, altitude);
    
    println!("zone_id = {}", zone_id);     // працює — u32 скопійовано
    println!("altitude = {}", altitude);   // працює — f64 скопійовано
}
```

Якщо ключами мають бути рядки, але ви не хочете передавати ownership — використовуйте `&str` як ключ. Але тоді рядки повинні жити довше за HashMap (lifetime зв'язок, Розділ 32).

## 22.6. Вимоги до ключів: Eq + Hash

Не кожен тип може бути ключем HashMap. Ключ повинен реалізувати два traits: `Eq` (точне порівняння на рівність) та `Hash` (обчислення хешу).

Типи, що задовольняють ці вимоги "з коробки": всі цілі числа (i32, u64 тощо), bool, char, String, &str, кортежі з Eq+Hash елементів, масиви з Eq+Hash елементів.

Типи, що НЕ задовольняють: `f32` та `f64`. Причина — NaN (Not a Number). За стандартом IEEE 754 `NaN != NaN`, що порушує вимогу Eq (кожне значення повинно дорівнювати собі). Крім того, різні бітові представлення NaN повинні мати однаковий хеш, але порівнюватися як нерівні — це ламає контракт Hash.

Якщо потрібна f64 як ключ (наприклад, координати) — є кілька рішень. Найпростіше: перетворити на цілі числа з фіксованою точністю:

```rust
use std::collections::HashMap;

fn coord_key(x: f64, y: f64) -> (i64, i64) {
    // Множимо на 1000 для точності до третього знаку
    ((x * 1000.0) as i64, (y * 1000.0) as i64)
}

fn main() {
    let mut zone_map: HashMap<(i64, i64), &str> = HashMap::new();
    
    zone_map.insert(coord_key(47.123, 35.456), "Зона пошуку");
    zone_map.insert(coord_key(47.200, 35.500), "База");
    
    // Пошук
    let key = coord_key(47.123, 35.456);
    if let Some(zone) = zone_map.get(&key) {
        println!("Знайдено: {}", zone);
    }
}
```

Для власних структур потрібно реалізувати (або derive) обидва traits:

```rust
use std::collections::HashMap;

#[derive(Hash, Eq, PartialEq, Debug)]
struct ZoneId {
    sector: char,
    number: u32,
}

fn main() {
    let mut zones: HashMap<ZoneId, u32> = HashMap::new();
    zones.insert(ZoneId { sector: 'A', number: 1 }, 15);
    zones.insert(ZoneId { sector: 'B', number: 3 }, 7);
    
    let key = ZoneId { sector: 'A', number: 1 };
    println!("Зона A1: {} об'єктів", zones[&key]);
}
```

`#[derive(Hash, Eq, PartialEq)]` автоматично реалізує потрібні traits, якщо всі поля структури їх підтримують. Це працює для більшості простих структур.

## 22.7. HashSet: множина унікальних елементів

HashSet — це, по суті, HashMap без значень. Він зберігає лише ключі та забезпечує миттєву перевірку "чи є цей елемент у множині?".

```rust
use std::collections::HashSet;

fn main() {
    let mut visited: HashSet<(i32, i32)> = HashSet::new();
    
    // Агент відвідує зони
    visited.insert((0, 0));
    visited.insert((1, 0));
    visited.insert((1, 1));
    visited.insert((0, 0)); // дублікат — ігнорується
    
    println!("Відвідано {} зон", visited.len()); // 3, не 4
    
    // Перевірка наявності — O(1)
    if visited.contains(&(1, 1)) {
        println!("Зона (1,1) вже відвідана");
    }
    if !visited.contains(&(2, 2)) {
        println!("Зона (2,2) ще не відвідана");
    }
}
```

`insert` повертає `bool`: `true` якщо елемент додано (його не було), `false` якщо він вже був (дублікат проігноровано). `contains` — перевірка наявності за O(1).

HashSet підтримує теоретико-множинні операції — перетин, об'єднання, різницю, симетричну різницю:

```rust
use std::collections::HashSet;

fn main() {
    let drone_a: HashSet<(i32, i32)> = [(0,0), (1,0), (2,0), (1,1)].into();
    let drone_b: HashSet<(i32, i32)> = [(1,0), (1,1), (1,2), (2,2)].into();
    
    // Перетин: зони, відвідані обома
    let both: HashSet<_> = drone_a.intersection(&drone_b).cloned().collect();
    println!("Обидва відвідали: {:?}", both); // {(1,0), (1,1)}
    
    // Об'єднання: всі відвідані зони
    let all: HashSet<_> = drone_a.union(&drone_b).cloned().collect();
    println!("Всього зон: {}", all.len()); // 6
    
    // Різниця: зони, які відвідав A, але не B
    let only_a: HashSet<_> = drone_a.difference(&drone_b).cloned().collect();
    println!("Тільки A: {:?}", only_a); // {(0,0), (2,0)}
}
```

Ці операції корисні для координації рою: знайти зони, які ще жоден дрон не перевірив (різниця між усіма зонами та об'єднанням відвіданих), або зони, де потрібне повторне сканування (перетин зон із виявленнями).

## 22.8. BTreeMap: коли потрібен порядок

HashMap не гарантує порядок ітерації. Якщо потрібні ключі у відсортованому порядку — використовуйте `BTreeMap`:

```rust
use std::collections::BTreeMap;

fn main() {
    let mut timeline: BTreeMap<u32, &str> = BTreeMap::new();
    timeline.insert(30, "Повернення на базу");
    timeline.insert(5, "Зліт");
    timeline.insert(12, "Сканування зони A");
    timeline.insert(20, "Сканування зони B");
    
    // Ітерація завжди у порядку зростання ключів
    println!("Хронологія місії:");
    for (time, event) in &timeline {
        println!("  t={:3}с: {}", time, event);
    }
    // t=  5с: Зліт
    // t= 12с: Сканування зони A
    // t= 20с: Сканування зони B
    // t= 30с: Повернення на базу
}
```

BTreeMap дає доступ за O(log N) замість O(1) у HashMap — повільніше для пошуку, але ітерація завжди у порядку ключів. Вимоги до ключів: `Ord` (повний порядок) замість `Hash`. Це означає, що f64 теж не підходить (через NaN), але String, числа та кортежі — працюють.

Коли що обирати? HashMap — за замовчуванням, коли порядок не важливий (95% випадків). BTreeMap — коли потрібна ітерація за порядком ключів (хронологія, діапазонні запити) або коли тип ключа реалізує Ord, але не Hash. BTreeSet — аналог HashSet із впорядкованими елементами.

## 22.9. Практика: HashMap та HashSet у БПЛА-агенті

Перша задача — реєстр виявлених об'єктів. Координатор рою зберігає інформацію про кожен виявлений об'єкт із миттєвим доступом за ідентифікатором:

```rust
use std::collections::HashMap;

fn main() {
    let mut objects: HashMap<String, (f64, f64, &str)> = HashMap::new();
    
    // Дрони повідомляють про виявлені об'єкти
    objects.insert(String::from("OBJ-001"), (47.12, 35.45, "будівля"));
    objects.insert(String::from("OBJ-002"), (47.15, 35.50, "транспорт"));
    objects.insert(String::from("OBJ-003"), (47.11, 35.44, "людина"));
    
    // Швидкий пошук за id
    if let Some(&(x, y, kind)) = objects.get("OBJ-003") {
        println!("OBJ-003: {} на ({:.2}, {:.2})", kind, x, y);
    }
    
    // Статистика за типами
    let mut type_counts: HashMap<&str, u32> = HashMap::new();
    for (_, &(_, _, kind)) in &objects {
        *type_counts.entry(kind).or_insert(0) += 1;
    }
    println!("За типами: {:?}", type_counts);
}
```

Тут entry API підраховує кількість об'єктів кожного типу. Замість ручної перевірки "чи є вже такий тип у лічильнику" — один рядок: `*type_counts.entry(kind).or_insert(0) += 1`.

Друга задача — карта відвіданих зон із HashSet. Агент позначає кожну клітинку сітки як відвідану та може миттєво перевірити, чи потрібно туди летіти:

```rust
use std::collections::HashSet;

fn main() {
    let mut visited: HashSet<(i32, i32)> = HashSet::new();
    
    // Симуляція патрулювання — сітка 5x5
    let patrol_route = vec![(0,0), (1,0), (2,0), (2,1), (2,2), (1,2), (0,2)];
    
    for pos in &patrol_route {
        visited.insert(*pos);
    }
    
    // Знайти зони, що ще не відвідані
    let grid_size = 3;
    let mut unvisited = Vec::new();
    for x in 0..grid_size {
        for y in 0..grid_size {
            if !visited.contains(&(x, y)) {
                unvisited.push((x, y));
            }
        }
    }
    
    println!("Відвідано: {} зон", visited.len());
    println!("Не відвідано: {:?}", unvisited);
    // Не відвідано: [(0, 1), (1, 1)]
    
    // Покриття у відсотках
    let total = (grid_size * grid_size) as f64;
    let coverage = visited.len() as f64 / total * 100.0;
    println!("Покриття: {:.0}%", coverage); // 78%
}
```

Перевірка `visited.contains(&(x, y))` працює за O(1) — для сітки 1000x1000 це мільйон перевірок, кожна миттєва. З вектором та лінійним пошуком кожна перевірка потребувала б перебору всіх записів.

Третя задача — агрегація даних місії через HashMap. Кожен дрон рою надсилає статистику, координатор збирає підсумки:

```rust
use std::collections::HashMap;

fn main() {
    // Звіти від дронів: (id_дрона, зона, кількість_виявлень)
    let reports = vec![
        ("D-01", "A1", 3), ("D-02", "A1", 5), ("D-01", "B2", 2),
        ("D-03", "A1", 1), ("D-02", "B2", 4), ("D-03", "C3", 7),
    ];
    
    // Агрегація: загальна кількість виявлень на зону
    let mut zone_totals: HashMap<&str, u32> = HashMap::new();
    for &(_, zone, count) in &reports {
        *zone_totals.entry(zone).or_insert(0) += count;
    }
    
    // Агрегація: кількість зон на дрон
    let mut drone_zones: HashMap<&str, HashSet<&str>> = HashMap::new();
    for &(drone, zone, _) in &reports {
        drone_zones.entry(drone).or_insert_with(HashSet::new).insert(zone);
    }
    
    println!("Виявлення за зонами:");
    for (zone, total) in &zone_totals {
        println!("  {}: {} об'єктів", zone, total);
    }
    
    println!("Зони за дронами:");
    for (drone, zones) in &drone_zones {
        println!("  {}: {:?}", drone, zones);
    }
}

use std::collections::HashSet;
```

Тут показано комбінацію HashMap та HashSet: `drone_zones` — це HashMap, де значення — HashSet (множина зон, які відвідав кожен дрон). `or_insert_with(HashSet::new)` створює порожню множину лише для нового дрона.

## Troubleshooting

### Помилка 1: `the trait bound f64: Hash is not satisfied`

Спроба використати f64 як ключ HashMap.

```rust
let mut map: HashMap<f64, String> = HashMap::new(); // ПОМИЛКА
```

Виправлення: перетворити на цілі числа з фіксованою точністю `(x * 1000.0) as i64`, або використати обгортку з ручною реалізацією Hash. Причина: NaN порушує контракт Eq та Hash.

### Помилка 2: `cannot borrow as mutable more than once`

Одночасне читання та запис у HashMap:

```rust
let val = map.get("key").unwrap(); // іммутабельне позичання
map.insert(String::from("other"), 5); // мутабельне позичання — ПОМИЛКА
println!("{}", val);
```

Виправлення: скопіювати значення перед мутацією (`let val = *map.get("key").unwrap();` для Copy-типів), або перебудувати логіку.

### Помилка 3: `key not found` при індексації map[key]

Паніка при зверненні до неіснуючого ключа через `[]`.

Виправлення: використовуйте `get` замість `[]` — він повертає Option замість паніки.

### Помилка 4: `cannot move out of borrowed content`

Спроба витягти String з HashMap через get:

```rust
let name: String = *map.get("key").unwrap(); // ПОМИЛКА для String
```

Виправлення: `get` повертає `&V`, а не `V`. Використовуйте посилання (`let name: &String = ...`) або клонування (`let name = map.get("key").unwrap().clone()`), або `remove("key")` якщо елемент більше не потрібен у мапі.

## Додатково

### HashMap vs Vec: коли що обирати

Просте правило: якщо доступ переважно за числовим індексом або послідовний обхід — Vec. Якщо доступ за довільним ключем (рядок, id, координати) — HashMap.

Конкретніше: Vec кращий коли елементів мало (до ~50) — лінійний пошук по кешованим послідовним даним може бути швидшим за хешування. Також Vec кращий коли порядок елементів важливий, або коли потрібен доступ за числовим індексом.

HashMap кращий коли елементів багато і потрібен пошук за ключем, коли потрібна перевірка наявності, коли дані природно описуються парами "ключ-значення".

### Внутрішня реалізація HashMap у Rust

Rust використовує hashbrown — реалізацію на основі Swiss Table від Google. Ця структура зберігає метадані (один байт на слот: порожній, видалений, або старші біти хешу) окремо від даних, що дозволяє ефективно використовувати SIMD-інструкції процесора для паралельної перевірки кількох слотів одночасно. На практиці це означає, що навіть при колізіях (кілька ключів потрапили в один кошик) пошук залишається дуже швидким.

## Контрольні запитання

### Рівень 1

1. Чим HashMap відрізняється від Vec?

Відповідь: Vec зберігає елементи послідовно з доступом за числовим індексом. HashMap зберігає пари ключ-значення з доступом за довільним ключем за O(1).

2. Що повертає `get` і чому не просто значення?

Відповідь: `Option<&V>` — None якщо ключ відсутній, Some(&значення) якщо присутній. Тому що відсутність ключа — нормальна ситуація, не помилка.

### Рівень 2

3. Чому f64 не може бути ключем HashMap?

Відповідь: f64 не реалізує Eq та Hash через NaN. NaN != NaN порушує вимогу рефлексивності Eq. Різні бітові NaN мали б давати різні хеші, але порівнюватись як нерівні — це ламає контракт Hash.

4. Поясніть рядок `*counts.entry(zone).or_insert(0) += 1`.

Відповідь: `entry(zone)` знаходить або створює слот для ключа zone. `or_insert(0)` вставляє 0 якщо слот порожній. Обидва повертають `&mut u32`. `*` розіменовує посилання. `+= 1` збільшує лічильник.

### Рівень 3

5. Напишіть функцію `most_frequent(words: &[&str]) -> &str` — слово з найбільшою частотою.

Відповідь:

```rust
fn most_frequent<'a>(words: &[&'a str]) -> &'a str {
    use std::collections::HashMap;
    let mut counts: HashMap<&str, u32> = HashMap::new();
    for &word in words {
        *counts.entry(word).or_insert(0) += 1;
    }
    counts.into_iter()
        .max_by_key(|&(_, count)| count)
        .map(|(word, _)| word)
        .unwrap_or("")
}
```

### Рівень 4

6. Чому HashMap не гарантує порядок ітерації? Коли це проблема?

Відповідь: порядок залежить від хеш-функції та розміру таблиці, які можуть змінюватись при реалокації або між запусками (Rust використовує рандомізований SipHash для захисту від HashDoS-атак). Це проблема, коли потрібен детерміністичний порядок: логи, звіти, серіалізація. Рішення: BTreeMap або окремо відсортувати ключі.

---

## Резюме

`HashMap<K, V>` зберігає пари ключ-значення з доступом за O(1). Ключ повинен реалізувати `Eq + Hash` (числа, рядки, кортежі — так; f64 — ні). Entry API (`entry().or_insert()`) об'єднує перевірку наявності та вставку в одну операцію — ідіома для лічильників та групування.

`HashSet<T>` — множина унікальних елементів із миттєвою перевіркою наявності. Підтримує теоретико-множинні операції: перетин, об'єднання, різниця.

HashMap володіє ключами та значеннями. String-ключ переміщується при insert; Copy-типи копіюються. Порядок ітерації не визначений — для впорядкованих даних використовуйте BTreeMap.

Вибір колекції: Vec — послідовні дані з числовим індексом. HashMap — доступ за довільним ключем. BTreeMap — доступ за ключем із гарантованим порядком. HashSet — перевірка наявності та множинні операції.

## Що далі

HashMap та Vec зберігають дані, але обробляють їх поелементно: цикл for, ручне накопичення результату, вкладені умови. Це працює, але код виходить багатослівним — 10 рядків там, де могло б бути 2.

У Розділі 23 ви вивчите ітератори та функціональний стиль: ланцюжки `map`, `filter`, `fold`, `collect`, що перетворюють обробку колекцій на потік трансформацій. `vec.iter().filter(|x| x > &0).count()` замість циклу з лічильником. Код стає декларативним — ви описуєте що зробити, а не як. І компілятор Rust оптимізує ланцюжки ітераторів до рівня ручних циклів — без втрати продуктивності.

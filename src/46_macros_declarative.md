# Розділ 46. Макроси: declarative (macro_rules!)

## Анотація

Створення нового актора у v3.0 — це десятки рядків boilerplate: enum Message з варіантами, struct Actor зі станом та receiver, impl з run() та match, struct Handle з sender та методами, fn new(). Для 10 типів акторів — 10 разів один і той самий каркас з різними полями. Макроси вирішують цю проблему: ви пишете шаблон один раз, і компілятор генерує код для кожного використання. `macro_rules!` — система декларативних макросів Rust: "якщо вхід виглядає ось так — згенеруй ось такий код". Це метапрограмування — код, що пише код.

---

## Цілі навчання

Після опрацювання цього розділу студент зможе:

1. Пояснити, чим макрос відрізняється від функції.
2. Написати `macro_rules!` з patterns та repetition.
3. Використати повторення: `$(...)*`, `$(...)+`, `$(...),*`.
4. Пояснити macro hygiene та `$crate`.
5. Дебажити макроси через `cargo expand`.
6. Обрати між макросом, функцією та trait для конкретної задачі.

---

## Ключові терміни

**macro_rules!** — система декларативних макросів Rust. Визначає правила "вхідний патерн → згенерований код".

**Pattern (патерн)** — шаблон, що макрос намагається зіставити з вхідними токенами. Містить метазмінні ($name:тип).

**Repetition (повторення)** — `$(...)*` (нуль або більше), `$(...)+` (один або більше) — для генерації змінної кількості елементів.

**Macro hygiene (гігієна макросів)** — властивість, що ідентифікатори всередині макросу не конфліктують з ідентифікаторами у коді, де макрос використовується.

**cargo expand** — інструмент для перегляду коду після розгортання макросів.

## 46.1. Макрос vs функція: коли що

Функція працює зі значеннями у runtime. Макрос працює з кодом (токенами) при компіляції. Функція приймає параметри конкретних типів. Макрос приймає довільні фрагменти коду: вирази, типи, ідентифікатори, блоки.

Конкретні випадки, де функція не працює, а макрос — так:

Змінна кількість аргументів: `println!("{} + {} = {}", a, b, c)` — функція не може прийняти довільну кількість аргументів різних типів. Макрос — може, бо він генерує конкретний код для кожного виклику.

Генерація нового коду: "створити struct з полями X, Y, Z" — функція не може створити тип. Макрос — може, бо він генерує код перед компіляцією.

DSL (Domain Specific Language): `vec![1, 2, 3]`, `println!`, `select!` — спеціальний синтаксис, неможливий через функції.

Правило: якщо задача вирішується функцією або trait — використовуйте їх. Макроси — лише коли альтернативи неможливі (змінна кількість аргументів, кодогенерація, DSL). Макроси складніші для читання, дебагу та підтримки.

## 46.2. Синтаксис macro_rules!

Макрос визначається через `macro_rules!` з одним або кількома правилами "патерн → код":

```rust
macro_rules! say_hello {
    () => {
        println!("Привіт, світ!");
    };
    ($name:expr) => {
        println!("Привіт, {}!", $name);
    };
}

fn main() {
    say_hello!();             // Привіт, світ!
    say_hello!("SCOUT-01");  // Привіт, SCOUT-01!
}
```

Кожне правило: `(патерн) => { код };`. Макрос пробує правила зверху вниз — перше що зіставляється, виконується. `$name:expr` — метазмінна: `$name` — ім'я, `expr` — тип фрагменту (вираз). При виклику `say_hello!("SCOUT-01")` — `$name` зіставляється з `"SCOUT-01"`, і генерується `println!("Привіт, {}!", "SCOUT-01")`.

Типи фрагментів (fragment specifiers):

| Тип | Що зіставляє | Приклад |
|-----|-------------|---------|
| `expr` | Вираз | `42`, `x + 1`, `foo()` |
| `ident` | Ідентифікатор | `my_var`, `Agent` |
| `ty` | Тип | `i32`, `String`, `Vec<f64>` |
| `pat` | Патерн | `Some(x)`, `_`, `(a, b)` |
| `stmt` | Інструкція | `let x = 5;` |
| `block` | Блок | `{ ... }` |
| `tt` | Один токен-дерево | Будь-що |
| `literal` | Літерал | `42`, `"hello"`, `3.14` |

## 46.3. Повторення: змінна кількість елементів

Головна сила макросів — повторення. `$(...)*` генерує код для кожного елемента:

```rust
macro_rules! create_struct {
    ($name:ident { $($field:ident : $ty:ty),* }) => {
        #[derive(Debug)]
        struct $name {
            $($field: $ty),*
        }
    };
}

create_struct!(Position { x: f64, y: f64 });
create_struct!(AgentInfo { id: String, battery: u8, active: bool });

fn main() {
    let pos = Position { x: 10.0, y: 20.0 };
    let agent = AgentInfo { id: String::from("S-01"), battery: 85, active: true };
    println!("{:?}", pos);
    println!("{:?}", agent);
}
```

`$($field:ident : $ty:ty),*` — повторення: нуль або більше пар "ідентифікатор : тип", розділених комами. Для кожної пари генерується `$field: $ty` у тілі struct.

Практичніший приклад — макрос для реєстрації обробників повідомлень:

```rust
macro_rules! match_message {
    ($msg:expr, { $($variant:pat => $handler:expr),* $(,)? }) => {
        match $msg {
            $($variant => $handler),*
        }
    };
}

enum Command {
    Start,
    Stop,
    Move(f64, f64),
}

fn handle(cmd: Command) {
    match_message!(cmd, {
        Command::Start => println!("Запуск"),
        Command::Stop => println!("Зупинка"),
        Command::Move(x, y) => println!("Рух до ({}, {})", x, y),
    });
}
```

`$(,)?` — опціональна trailing comma (нуль або одна кома в кінці). Це дозволяє писати останній елемент як з комою, так і без — зручність для користувача макросу.

## 46.4. Практичний макрос: генерація акторів

Макрос, що генерує каркас актора з Handle:

```rust
macro_rules! define_actor {
    (
        name: $actor_name:ident,
        message: $msg_type:ident,
        handle: $handle_name:ident,
        buffer: $buf_size:expr
    ) => {
        pub struct $actor_name {
            receiver: tokio::sync::mpsc::Receiver<$msg_type>,
        }

        impl $actor_name {
            pub async fn run(mut self) {
                while let Some(msg) = self.receiver.recv().await {
                    self.handle_message(msg).await;
                }
            }
        }

        #[derive(Clone)]
        pub struct $handle_name {
            sender: tokio::sync::mpsc::Sender<$msg_type>,
        }

        impl $handle_name {
            pub fn new() -> (Self, $actor_name) {
                let (sender, receiver) = tokio::sync::mpsc::channel($buf_size);
                let handle = $handle_name { sender };
                let actor = $actor_name { receiver };
                (handle, actor)
            }

            pub async fn send(&self, msg: $msg_type) {
                let _ = self.sender.send(msg).await;
            }
        }
    };
}

// Використання:
enum LogMessage {
    Info(String),
    Error(String),
}

define_actor!(
    name: LoggerActor,
    message: LogMessage,
    handle: LoggerHandle,
    buffer: 64
);

impl LoggerActor {
    async fn handle_message(&self, msg: LogMessage) {
        match msg {
            LogMessage::Info(text) => println!("[INFO] {}", text),
            LogMessage::Error(text) => println!("[ERROR] {}", text),
        }
    }
}
```

Макрос генерує struct Actor, struct Handle та fn new(). Користувач лише визначає enum Message та реалізує handle_message. Для нового актора — один виклик define_actor! замість 30+ рядків boilerplate.

## 46.5. Дебаг макросів: cargo expand

Макроси генерують код невидимо. Коли щось не працює — потрібно побачити згенерований код:

```bash
# Встановити
cargo install cargo-expand

# Переглянути розгорнутий код
cargo expand
```

cargo expand показує файл після розгортання всіх макросів — ви бачите конкретний код, що компілюється. Це ключовий інструмент для дебагу: якщо макрос генерує некоректний код — ви побачите його у expand та зрозумієте проблему.

## 46.6. Macro hygiene

Rust макроси є "гігієнічними" — ідентифікатори всередині макросу не конфліктують із зовнішніми:

```rust
macro_rules! create_var {
    () => {
        let x = 42; // цей x — "всередині макросу"
    };
}

fn main() {
    let x = 10;     // зовнішній x
    create_var!();   // створює свій x, не впливає на зовнішній
    println!("{}", x); // 10, не 42
}
```

Гігієна запобігає випадковим конфліктам імен. Але іноді потрібно "пробити" гігієну — для цього є патерни з $crate та передачею ідентифікаторів як параметрів.

## Troubleshooting

### Помилка 1: `no rules expected the token`

Вхідні токени не зіставляються з жодним правилом макросу. Перевірте: правильний тип фрагменту (expr vs ident vs ty), правильні роздільники (коми, крапки з комою).

### Помилка 2: `local ambiguity`

Компілятор не може однозначно зіставити вхід з правилом — два можливих зіставлення. Виправлення: додайте роздільники між метазмінними або змініть порядок правил.

### Помилка 3: макрос не видимий в іншому модулі

За замовчуванням макрос видимий лише у поточному модулі. Виправлення: `#[macro_export]` перед macro_rules!, або `pub(crate) use macro_name;`.

## Додатково

### Коли макрос, коли функція, коли trait

| Задача | Рішення |
|--------|---------|
| Обчислення зі значеннями | Функція |
| Спільна поведінка для різних типів | Trait |
| Змінна кількість аргументів | Макрос |
| Генерація нових типів/структур | Макрос |
| DSL (спеціальний синтаксис) | Макрос |
| Derive для користувацьких типів | Процедурний макрос (Р.47) |

### Відомі макроси стандартної бібліотеки

`vec![]` — створення Vec з елементів. `println!`/`format!` — форматований вивід. `assert!`/`assert_eq!` — перевірки у тестах. `cfg!` — умовна компіляція. `include_str!` — включення файлу як рядка. Кожен — macro_rules! (або compiler built-in).

## Контрольні запитання

### Рівень 1

1. Чим макрос відрізняється від функції?

Відповідь: функція працює зі значеннями у runtime. Макрос працює з токенами (кодом) при компіляції. Макрос може приймати довільні фрагменти коду (типи, ідентифікатори, блоки) та генерувати нові типи та функції.

2. Що означає `$name:expr` у макросі?

Відповідь: метазмінна $name, що зіставляється з виразом (expr). При використанні макросу конкретний вираз підставляється замість $name.

### Рівень 2

3. Чим `$(...)*` відрізняється від `$(...)+`?

Відповідь: `*` — нуль або більше повторень. `+` — один або більше (мінімум одне). `$($x:expr),*` зіставить порожній вхід або `a, b, c`. `$($x:expr),+` вимагає хоча б один елемент.

4. Що таке macro hygiene?

Відповідь: ідентифікатори створені всередині макросу не конфліктують із зовнішніми. let x = 42 всередині макросу — не перезаписує зовнішній x. Запобігає випадковим побічним ефектам.

### Рівень 3

5. Напишіть макрос `max!` що приймає 2+ аргументів і повертає найбільший.

Відповідь:

```rust
macro_rules! max {
    ($x:expr) => ($x);
    ($x:expr, $($rest:expr),+) => {
        {
            let a = $x;
            let b = max!($($rest),+);
            if a > b { a } else { b }
        }
    };
}
```

### Рівень 4

6. Чому macro_rules! обмежений і коли потрібні процедурні макроси?

Відповідь: macro_rules! працює з патернами токенів — не може "розуміти" Rust-код (аналізувати поля struct, виводити типи). Процедурні макроси (Р.47) отримують TokenStream, можуть його парсити через syn та генерувати довільний код через quote. Derive macros (#[derive(Agent)]) — лише через процедурні. macro_rules! — для простих підстановок та DSL.

## Резюме

macro_rules! — система декларативних макросів: "патерн → код". Метазмінні ($name:expr, $name:ident, $name:ty) зіставляють фрагменти коду. Повторення ($(...)*) генерують змінну кількість елементів.

Макроси — для кодогенерації, DSL, змінної кількості аргументів. Не для звичайної логіки — функції та traits простіші та зрозуміліші.

cargo expand — ключовий інструмент дебагу: показує код після розгортання макросів.

Macro hygiene запобігає конфліктам імен між макросом та зовнішнім кодом.

---

## Що далі

macro_rules! працює з патернами токенів — потужно, але обмежено. Не можна парсити struct, аналізувати поля, генерувати код на основі типу. Розділ 47 представляє процедурні макроси: derive macros (`#[derive(Agent)]` автоматично генерує impl Agent), attribute macros (`#[instrument]` — ви вже використовували), function-like macros. Процедурні макроси отримують повний TokenStream, парсять його через крейт syn, і генерують новий код через quote. Це повноцінне метапрограмування.

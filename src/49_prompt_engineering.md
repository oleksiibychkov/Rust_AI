# Розділ 49. Prompt Engineering: просунуті техніки та професійний workflow

## Анотація

Протягом усього підручника PE-секції навчали конкретних прийомів: "дай AI код + помилку для пояснення", "попроси переписати цикл через ітератори", "попроси порівняти enum vs trait object". Це тактичні навички — вирішення конкретної задачі конкретним промптом. Цей розділ піднімає рівень до стратегічного: як вибудувати професійний workflow з AI-асистентом для всього циклу розробки — від архітектурного проєктування до code review, від генерації тестів до документації. Ви вивчите просунуті техніки промптингу (chain-of-thought, few-shot, системні промпти), навчитесь працювати з великим контекстом, розумітимете обмеження AI (галюцинації, застарілі знання), та створите персональну бібліотеку промпт-шаблонів.

---

## Цілі навчання

Після опрацювання цього розділу студент зможе:

1. Застосовувати chain-of-thought для складних задач.
2. Використовувати few-shot промптинг для контролю формату відповіді.
3. Налаштовувати системні промпти та персони.
4. Вести ітеративний діалог для складних задач.
5. Працювати з великим контекстом (передача codebase).
6. Розпізнавати галюцинації та обмеження AI.
7. Створити бібліотеку промпт-шаблонів для Rust-розробки.

---

## Ключові терміни

**Chain-of-thought (ланцюжок міркувань)** — техніка, де AI просять "думати вголос" — показати покрокові міркування перед фінальною відповіддю. Підвищує якість для складних задач.

**Few-shot prompting** — техніка, де промпт містить 2–3 приклади бажаного результату. AI "бачить" патерн і генерує відповідь у тому ж форматі.

**System prompt (системний промпт)** — інструкція, що визначає поведінку AI для всієї сесії: роль, стиль, обмеження, контекст.

**Hallucination (галюцинація)** — ситуація, коли AI генерує впевнену, але фактично неправильну відповідь. Неіснуючі функції, неправильні сигнатури, вигадані крейти.

**Prompt template (шаблон промпту)** — заготовка промпту з плейсхолдерами для конкретної задачі. Перевірений та відточений шаблон дає стабільно якісні результати.

## 49.1. Chain-of-thought: змусити AI думати

Для простих задач ("що робить `?` оператор?") AI відповідає добре без додаткових інструкцій. Для складних ("спроєктуй архітектуру actor-based рою з 5 типами повідомлень та graceful shutdown") якість різко зростає, якщо попросити AI "думати вголос":

Без chain-of-thought:

```
Спроєктуй архітектуру рою БПЛА на Tokio з акторами.
```

AI може одразу видати код — але пропустити важливі рішення, не пояснити trade-offs, обрати перший варіант замість оптимального.

З chain-of-thought:

```
Спроєктуй архітектуру рою БПЛА на Tokio з акторами.

Перед тим як давати фінальну архітектуру:
1. Перелічи компоненти системи та їхні відповідальності
2. Для кожної пари компонентів визнач: чи потрібна комунікація, 
   якщо так — який тип (mpsc/broadcast/watch/oneshot)?
3. Визнач потенційні проблеми (deadlock, bottleneck, memory)
4. Запропонуй 2 варіанти архітектури з trade-offs
5. Обери один та поясни чому

Потім — фінальна архітектура з діаграмою та кодом.
```

Результат значно кращий: AI розглядає альтернативи, ідентифікує проблеми, обґрунтовує вибір. Покрокова інструкція не дозволяє "перескочити" до коду, ігноруючи дизайн-рішення.

Коли використовувати: архітектурні рішення, дебаг складних помилок, порівняння підходів, оптимізація. Не потрібно для простих задач ("напиши функцію сортування").

## 49.2. Few-shot: приклади замість пояснень

Замість довгого опису бажаного формату — покажіть 2–3 приклади:

```
Мені потрібна документація для Rust-функцій у такому форматі:

Приклад:
/// Обчислює відстань між двома точками.
///
/// # Arguments
/// * `a` - Перша точка (x, y)
/// * `b` - Друга точка (x, y)
///
/// # Returns
/// Евклідова відстань як f64
///
/// # Example
/// ```
/// let d = distance((0.0, 0.0), (3.0, 4.0));
/// assert_eq!(d, 5.0);
/// ```
fn distance(a: (f64, f64), b: (f64, f64)) -> f64

Тепер згенеруй документацію у цьому форматі для:
fn find_nearest<T: HasPosition>(items: &[T], target: (f64, f64)) -> Option<&T>
fn calculate_route(from: Position, to: Position, obstacles: &[Position]) -> Vec<Position>
```

AI "розуміє" формат з прикладу і генерує документацію в ідентичному стилі. Це ефективніше за текстовий опис "напиши doc comments з Arguments, Returns та Example секціями у стилі rustdoc".

Few-shot особливо корисний для: генерації коду у конкретному стилі (ваша code convention), форматування (логи, звіти, таблиці), трансформації (один формат → інший).

## 49.3. Системні промпти та персони

Системний промпт визначає "хто" AI для всієї сесії:

```
Ти — senior Rust developer з досвідом у embedded та robotics.
Відповідаєш лаконічно, з фокусом на продуктивність та безпеку.
Для кожного рішення вказуєш: runtime overhead, memory usage, 
thread safety.
Якщо є unsafe — пояснюєш чому необхідний та чи можна уникнути.
Код завжди з #[deny(clippy::all)] та повними типами без unwrap.
```

Або вужча роль:

```
Ти — code reviewer. Для кожного блоку коду:
1. Назви 3 сильних сторони
2. Назви 3 проблеми (bugs, performance, style)
3. Для кожної проблеми — конкретний fix з кодом
Не хвали загально — лише конкретика.
```

Персона змінює стиль, глибину та фокус відповідей. "Senior Rust developer" зверне увагу на unsafe та performance. "Rust teacher for beginners" пояснить кожен крок детально. "Security auditor" шукатиме вразливості.

## 49.4. Ітеративний промптинг: складні задачі за кілька кроків

Складну задачу не варто вирішувати одним промптом. Краще — серія кроків, де кожен будує на попередньому:

Крок 1 — архітектура:
```
Опиши архітектуру async рою: які актори, які канали, які повідомлення.
Лише дизайн, без коду.
```

Крок 2 — типи:
```
На основі архітектури вище, створи enum для кожного типу повідомлень 
та struct для стану кожного актора.
```

Крок 3 — каркас:
```
На основі типів вище, створи скелет кожного актора: 
struct, new(), async fn run() з порожнім match.
```

Крок 4 — логіка:
```
Заповни логіку handle_report у координаторі. 
Ось вимоги: [конкретні вимоги].
```

Крок 5 — review:
```
Переглянь весь код. Знайди: потенційні deadlocks, 
зайві clone, пропущені error handling, неправильні рівні tracing.
```

Кожен крок — конкретна задача з контекстом з попередніх. AI "бачить" еволюцію та відповідає в контексті. Одним промптом отримати якісний результат для складної системи — маловірогідно.

## 49.5. Робота з великим контекстом

Реальний проєкт — десятки файлів. Як передати достатньо контексту AI?

Стратегія "релевантний мінімум": передавайте лише файли, релевантні до поточної задачі. Для дебагу — файл з помилкою + типи, що використовуються. Для review — файл, що переглядається + trait/struct визначення, що він використовує.

Стратегія "структура + деталі": спершу дайте AI огляд (дерево файлів, опис модулів, ключові типи), потім — конкретний файл для роботи.

```
Структура проєкту:
src/
  main.rs — точка входу, ініціалізація акторів
  coordinator.rs — CoordinatorActor, обробка звітів
  agent.rs — AgentActor, select! loop
  messages.rs — enum AgentReport, Command, MissionConfig
  error.rs — AgentError з thiserror

Ключові типи:
- AgentReport: Position, Detection, StatusUpdate, MissionComplete
- Command: ChangeTarget, ReturnToBase, Shutdown
- CoordinatorHandle: report(), get_status(), shutdown()

Зараз працюю над agent.rs. Ось повний файл:
[вставити agent.rs]

Задача: додати обробку Command::ChangeAltitude.
```

AI має достатньо контексту (типи, архітектура, поточний файл) без передачі всього проєкту.

## 49.6. Обмеження AI: галюцинації та як їх розпізнати

AI може генерувати впевнені, але неправильні відповіді. Типові випадки у Rust-контексті:

Неіснуючі функції та методи: AI "вигадує" метод, якого немає у стандартній бібліотеці або крейті. Наприклад, `vec.remove_all(predicate)` — такого методу немає (є retain). Ознака: метод "звучить логічно" але ви не знаходите його в документації.

Неправильні сигнатури: AI показує `HashMap::get` що повертає `&V` замість `Option<&V>`. Або плутає порядок параметрів. Ознака: код "виглядає правильно" але не компілюється.

Застарілий API: AI навчений на коді до певної дати. Може пропонувати deprecated функції або старі версії крейтів. Ознака: deprecation warnings при компіляції.

Вигадані крейти: AI може запропонувати крейт, що не існує на crates.io. Ознака: `cargo add` не знаходить крейт.

Як захиститись: завжди компілюйте та тестуйте код від AI. Перевіряйте сигнатури у документації (docs.rs). Не довіряйте коду, який "виглядає правильно" — запускайте його. AI — помічник, не авторитет.

## 49.7. AI для різних етапів розробки

AI корисний на кожному етапі, але по-різному:

Проєктування: "порівняй 3 архітектури для цієї задачі" — AI пропонує варіанти з trade-offs. Ви обираєте та адаптуєте.

Реалізація: "напиши функцію за цією специфікацією" — AI генерує каркас. Ви доповнюєте бізнес-логіку та edge cases.

Дебаг: "ось помилка компілятора + код, поясни та виправ" — найефективніший use case AI для Rust. Помилки borrow checker часто незрозумілі для початківців, AI пояснює покроково.

Code review: "знайди баги, performance проблеми, стиль" — AI знаходить те, що людське око пропускає: пропущений error handling, зайві clone, можливі паніки.

Тести: "згенеруй тести для цієї функції, включаючи edge cases" — AI добре генерує happy path та типові edge cases. Але може пропустити тонкі граничні випадки — доповніть ручними тестами.

Документація: "напиши rustdoc для цих функцій" — AI генерує якісну документацію, особливо з few-shot прикладом формату.

Рефакторинг: "перепиши match на if let де доречно" або "заміни unwrap на Result" — AI добре справляється з механічними трансформаціями.

## 49.8. Бібліотека промпт-шаблонів

За час курсу ви використали десятки промптів. Найефективніші — зберігайте як шаблони:

Архітектурний промпт:
```
Проєктую систему [опис]. Компоненти: [список].
Для кожного: відповідальність, стан, вхідні/вихідні повідомлення.
Запропонуй 2 архітектури з trade-offs. 
Обери одну та поясни чому.
```

Дебаг промпт:
```
Код: [код]
Помилка: [повний текст помилки]
Поясни: 1) чому виникає 2) як виправити 3) як уникнути у майбутньому.
```

Review промпт:
```
Код: [код]
Перевір: bugs, performance, error handling, style, unsafe, 
пропущені edge cases. Для кожної проблеми — fix з кодом.
```

Рефакторинг промпт:
```
Код: [код]
Перетвори: [що зробити: unwrap→Result, цикл→ітератор, struct→actor].
Поясни кожну зміну.
```

Тести промпт:
```
Функція: [код]
Згенеруй тести: happy path, edge cases (порожній вхід, 
максимальне значення, невалідний вхід), error cases.
```

## 49.9. Етика та відповідальність

Код, згенерований AI, — ваша відповідальність. AI не "автор" — ви. Якщо AI-код містить баг, вразливість, або порушення ліцензії — відповідаєте ви.

Практичні правила: завжди компілюйте та тестуйте. Розумійте кожен рядок — не вставляйте код, який не розумієте. Перевіряйте ліцензії крейтів, запропонованих AI. Не передавайте конфіденційний код у публічний AI без дозволу. AI — інструмент, як компілятор або IDE. Фінальне рішення та відповідальність — завжди на програмісті.

## Контрольні запитання

### Рівень 1

1. Що таке chain-of-thought?

Відповідь: техніка, де AI просять "думати вголос" — показати покрокові міркування перед відповіддю. Підвищує якість для складних задач: архітектура, дебаг, порівняння.

2. Що таке галюцинація AI?

Відповідь: впевнена, але фактично неправильна відповідь. Неіснуючі функції, неправильні сигнатури, вигадані крейти. Захист: компілювати, тестувати, перевіряти документацію.

### Рівень 2

3. Чим few-shot відрізняється від інструкції?

Відповідь: інструкція — текстовий опис ("напиши у такому форматі"). Few-shot — приклади бажаного результату. AI "бачить" патерн і відтворює. Few-shot ефективніший для контролю формату та стилю.

4. Коли ітеративний промптинг кращий за один промпт?

Відповідь: для складних задач із кількома етапами: архітектура → типи → каркас → логіка → review. Кожен крок будує на попередньому. Одним промптом AI може пропустити деталі або зробити некоректні припущення.

### Рівень 4

5. Як оцінити, чи AI-відповідь якісна для Rust-коду?

Відповідь: компілюється без warnings (cargo clippy), тести проходять, жодних unwrap у production коді, правильні типи помилок, ідіоматичний стиль (ітератори де доречно, ownership без зайвих clone), документовані unsafe блоки. Якщо будь-що з цього порушено — відповідь потребує доопрацювання.

---

## Резюме

Chain-of-thought — "думай вголос" для складних задач. Few-shot — приклади замість пояснень для контролю формату. Системні промпти — налаштування ролі та стилю AI.

Ітеративний промптинг — складні задачі за кілька кроків: архітектура → типи → каркас → логіка → review. Кожен крок — конкретна задача з контекстом.

Галюцинації — неіснуючі функції, неправильні сигнатури. Захист: компілювати, тестувати, перевіряти docs.rs.

AI для кожного етапу: проєктування (варіанти + trade-offs), реалізація (каркас), дебаг (пояснення помилок), review (баги + performance), тести (edge cases), документація (rustdoc).

Бібліотека шаблонів — перевірені промпти для типових задач. AI — інструмент. Відповідальність — на програмісті.

# Розділ 11. Функції та параметри

## Анотація

Код нашого БПЛА-агента з Розділу 10 виріс до десятків рядків в одному `fn main()`, з довгим ланцюжком `else if` для обробки команд. Додавання кожної нової команди ускладнює цей моноліт. Якщо потрібно обчислити відстань до бази в двох різних місцях — доведеться скопіювати формулу двічі, а при виправленні помилки — знайти та виправити обидві копії. Функції вирішують ці проблеми через декомпозицію (розбиття великої задачі на менші) та повторне використання (написати один раз — викликати скільки завгодно). Цей розділ — один із найважливіших у курсі: він пояснює не лише синтаксис, а й глибинні концепції передачі даних між частинами програми, повернення результатів, та ізоляції змінних.

## Цілі навчання

Після опрацювання цього розділу студент зможе:

1. Оголосити функцію з параметрами та типом повернення, і викликати її з аргументами.
2. Пояснити різницю між параметром та аргументом, і правильно використовувати терміни.
3. Пояснити, що відбувається в пам'яті при передачі за значенням, за посиланням `&T` та `&mut T`.
4. Пояснити різницю між виразом та інструкцією, та роль крапки з комою.
5. Повернути значення з функції неявно та явно, пояснити різницю між функцією та процедурою.
6. Описати стек викликів (call stack) та пояснити, як процесор відстежує вкладені виклики.
7. Пояснити область дії параметрів та локальних змінних.

## Ключові терміни

**Function (функція)** — іменований блок коду з визначеними входами та виходом.

**Procedure (процедура)** — функція, що виконує дію без змістовного результату (повертає `()`).

**Parameter (параметр)** — змінна в оголошенні функції; визначає тип вхідних даних.

**Argument (аргумент)** — конкретне значення, передане при виклику функції.

**Pass by value (передача за значенням)** — створення копії аргументу для функції.

**Pass by reference (передача за посиланням)** — передача адреси даних замість копії.

**Return value (повернене значення)** — результат, що функція передає у місце виклику.

**Expression (вираз)** — конструкція, що обчислюється у значення.

**Statement (інструкція)** — конструкція, що виконує дію без повернення значення.

**Call stack (стек викликів)** — область пам'яті, де зберігаються локальні змінні кожної активної функції.

**Stack frame (кадр стеку)** — блок пам'яті на стеку, що належить одному виклику функції.

**Signature (сигнатура)** — ім'я функції, параметри з типами, тип повернення.

## 11.1. Навіщо потрібні функції

### Проблема складності

Людський мозок одночасно утримує 5–9 елементів інформації (закон Міллера). Функція на 200 рядків має десятки змінних, умов і залежностей — далеко за межами когнітивного навантаження. Ви не можете "побачити" таку функцію цілком: коли читаєте рядок 150, ви вже забули, що було на рядку 20.

Подумайте про код з попередніх розділів. У Розділі 10 ми зчитували команди користувача, обробляли їх через ланцюжок `else if`, оновлювали координати, перевіряли заряд батареї, виводили статус — все в одному `main()`. Кожне нове доповнення робило код ще довшим та менш зрозумілим. Ми дійшли до межі, де моноліт стає некерованим.

### Декомпозиція

Декомпозиція — це розбиття великої задачі на менші, кожна з яких вирішується окремою функцією. Замість одного `main()` на 200 рядків — десять функцій по 20. Кожна має одну відповідальність: `calculate_distance()` — відстань, `check_battery()` — заряд, `process_command()` — команда.

Аналогія з реальним світом: авіадиспетчер не проєктує двигун літака. Він працює з абстракцією "літак летить зі швидкістю X на висоті Y". Деталі двигуна — відповідальність інженера. Деталі навігації — відповідальність навігатора. Кожен спеціаліст — це "функція" з чіткою відповідальністю.

### Повторне використання

Якщо формула обчислення відстані потрібна в трьох місцях програми — без функцій доведеться скопіювати її тричі. При виправленні помилки — шукати та виправляти три копії. З функцією: формула існує в одному місці, виклик — у трьох. Виправлення — одне, ефект — скрізь.

Для БПЛА: `takeoff()` — зліт, `land()` — посадка, `get_status()` — статус. Кожна функція — як кнопка на пульті. Натиснув — виконалось. Не потрібно знати деталі, щоб користуватись.

### Тестованість

Окрема функція може бути протестована незалежно від решти програми. `calculate_distance(0.0, 0.0, 3.0, 4.0)` має повернути 5.0 — і ви можете перевірити це одним рядком. Тестувати 200-рядковий `main()` — значно складніше. Ми повернемось до тестування у Розділі 19, і функції будуть основою кожного тесту.

### До і після: монолітний код vs декомпозиція

Порівняємо два підходи до тієї самої задачі — обчислити, чи може БПЛА долетіти до цілі та повернутись на базу.

Без функцій (монолітний код):

```rust
fn main() {
    let agent_x: f64 = 10.0;
    let agent_y: f64 = 20.0;
    let target_x: f64 = 40.0;
    let target_y: f64 = 60.0;
    let base_x: f64 = 0.0;
    let base_y: f64 = 0.0;
    let battery: u8 = 80;
    let consumption_per_km: f64 = 1.5;

    // Відстань до цілі
    let dx1 = target_x - agent_x;
    let dy1 = target_y - agent_y;
    let dist_to_target = (dx1 * dx1 + dy1 * dy1).sqrt();

    // Відстань від цілі до бази
    let dx2 = base_x - target_x;
    let dy2 = base_y - target_y;
    let dist_to_base = (dx2 * dx2 + dy2 * dy2).sqrt();

    // Загальне пальне
    let total_dist = dist_to_target + dist_to_base;
    let fuel_needed = total_dist * consumption_per_km;

    // Перевірка
    if (battery as f64) >= fuel_needed {
        println!("Місія можлива! Потрібно {:.1} одиниць пального", fuel_needed);
    } else {
        println!("Недостатньо пального: є {}, потрібно {:.1}", battery, fuel_needed);
    }
}
```

З функціями:

```rust
fn distance(x1: f64, y1: f64, x2: f64, y2: f64) -> f64 {
    let dx = x2 - x1;
    let dy = y2 - y1;
    (dx * dx + dy * dy).sqrt()
}

fn fuel_for_route(dist: f64, rate: f64) -> f64 {
    dist * rate
}

fn can_complete_mission(battery: u8, fuel_needed: f64) -> bool {
    (battery as f64) >= fuel_needed
}

fn main() {
    let (ax, ay) = (10.0, 20.0);
    let (tx, ty) = (40.0, 60.0);
    let (bx, by) = (0.0, 0.0);
    let battery: u8 = 80;

    let to_target = distance(ax, ay, tx, ty);
    let to_base = distance(tx, ty, bx, by);
    let fuel = fuel_for_route(to_target + to_base, 1.5);

    if can_complete_mission(battery, fuel) {
        println!("Місія можлива! Потрібно {:.1} одиниць пального", fuel);
    } else {
        println!("Недостатньо пального: є {}, потрібно {:.1}", battery, fuel);
    }
}
```

`main` у другому варіанті — 8 рядків логіки замість 20. Формула відстані написана один раз і використана двічі. Кожну функцію можна протестувати окремо: `assert_eq!(distance(0.0, 0.0, 3.0, 4.0), 5.0)`. Якщо потрібно обчислити відстань ще в десяти місцях програми — один виклик `distance(...)` замість копіювання формули.

## 11.2. Синтаксис оголошення та виклику

Функція в Rust оголошується ключовим словом `fn`:

```rust
fn greet() {
    println!("Привіт від БПЛА!");
}

fn main() {
    greet();
    greet();
}
```

Розберемо елементи оголошення. `fn` — "оголосити функцію". `greet` — ім'я (snake_case: `calculate_distance`, `check_battery_level`). `()` — порожні дужки: функція не приймає даних. `{ ... }` — тіло: код, що виконується при виклику.

Виклик `greet()` — це інструкція для процесора: "стрибни до тіла greet, виконай його, повернись назад до наступного рядка". Виклик можна робити скільки завгодно разів.

На відміну від C, порядок оголошення функцій у файлі не має значення. `main` може бути першою або останньою — компілятор Rust аналізує весь файл перед компіляцією і знаходить усі функції.

Правило іменування: ім'я функції описує дію, тому зазвичай починається з дієслова: `calculate_distance`, `check_battery`, `print_status`, `parse_command`. Іменники на початку (`distance_calculate`) — нетиповий стиль для функцій.

### Функція з параметрами та поверненням

Більшість корисних функцій приймають дані та повертають результат. Ось функція, що обчислює квадрат числа:

```rust
fn square(x: i32) -> i32 {
    x * x
}

fn main() {
    let a = square(5);
    let b = square(3);
    println!("5² = {}, 3² = {}", a, b);
    println!("5² + 3² = {}", square(5) + square(3));
}
```

Вивід:

```
5² = 25, 3² = 9
5² + 3² = 34
```

Розберемо сигнатуру `fn square(x: i32) -> i32`. `x: i32` — параметр: функція очікує одне ціле число. `-> i32` — тип повернення: функція гарантовано повертає ціле число. Тіло `x * x` — вираз (без `;`), значення якого автоматично повертається.

Виклик `square(5)` — це вираз, що має значення 25. Тому його можна використати в будь-якому місці, де очікується i32: присвоїти змінній, передати в іншу функцію, додати до іншого числа.

## 11.3. Параметр та аргумент: два різних поняття

Ця термінологія плутається навіть у досвідчених програмістів. Зупинимось на ній детально.

**Параметр** (parameter) — це змінна в *оголошенні* функції. Параметр описує, яке значення функція очікує: тип, ім'я, призначення. Параметр — це "порожнє місце", "гніздо", що буде заповнене при виклику. Параметр існує тільки в сигнатурі та тілі функції.

**Аргумент** (argument) — це конкретне значення, передане при *виклику* функції. Аргумент заповнює "гніздо" параметра. Аргумент існує в коді, що викликає функцію.

Аналогія: форма замовлення в ресторані має поле "Назва страви" — це параметр. Коли ви пишете "борщ" у цьому полі — це аргумент. Поле визначає, що сюди можна вписати (назву страви, а не номер телефону). Запис визначає конкретне значення.

```rust
//                   два параметри
//                   ↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓
fn print_altitude(altitude: f64, unit: &str) {
    println!("Висота: {:.1} {}", altitude, unit);
}

fn main() {
    let h = 120.5;
    //                   два аргументи
    //                   ↓↓↓↓↓↓↓↓↓↓↓
    print_altitude(h, "м");
    print_altitude(395.0, "футів");
}
```

У рядку `fn print_altitude(altitude: f64, unit: &str)` — `altitude` та `unit` це параметри. Вони визначають контракт: "ця функція приймає дробове число та рядок."

У рядку `print_altitude(h, "м")` — `h` та `"м"` це аргументи. Це конкретні значення, що заповнюють параметри при виклику.

Аргументи підставляються за позицією: перший аргумент йде у перший параметр, другий — у другий. Порядок має значення: `print_altitude("м", h)` не скомпілюється, бо `"м"` — це `&str`, а перший параметр очікує `f64`.

В Rust тип кожного параметра вказується обов'язково. `fn f(x: i32)` — правильно. `fn f(x)` — помилка. Це свідоме рішення: сигнатура функції — це повний контракт. Побачивши `fn calculate_distance(x1: f64, y1: f64, x2: f64, y2: f64) -> f64`, будь-хто знає: чотири дробових на вході, одне на виході. Не потрібно читати тіло.

Коли у розмові чи документації ви чуєте "передай параметр у функцію" — це технічно некоректно (передають аргумент). Але на практиці цю різницю часто ігнорують, і слово "параметр" використовують в обох значеннях. У цьому підручнику ми дотримуємось точної термінології: параметр — в оголошенні, аргумент — при виклику.

## 11.4. Стек викликів: як процесор відстежує функції

Перш ніж говорити про передачу даних, потрібно зрозуміти механізм, що стоїть за кожним викликом функції: call stack (стек викликів).

### Що таке стек і як він влаштований

У Розділі 9 ви дізнались, що пам'ять програми поділяється на стек та купу (heap). Стек використовується для локальних змінних, параметрів та адрес повернення. "Стек" як структура даних означає "останній прийшов — перший вийшов" (LIFO: Last In, First Out). Як стопка тарілок у кафетерії: берете верхню, кладете зверху.

Кожен виклик функції створює на стеку stack frame (кадр стеку) — блок пам'яті, що містить усі локальні дані цього виклику. Коли функція завершується — кадр знімається зі стеку, а вся його пам'ять миттєво стає вільною.

Уявіть стек як стопку тарілок. Кожна тарілка — один виклик функції. Коли функція викликається — нова тарілка кладеться зверху. Коли функція завершується — верхня тарілка знімається. Завжди працюємо з верхньою тарілкою — тобто з тією функцією, що виконується зараз.

### Що зберігається в кадрі стеку

Кожен stack frame містить кілька важливих речей. Адреса повернення — номер інструкції в коді, до якої процесор повинен повернутись після завершення функції. Коли `main` викликає `add(5, 3)`, процесор запам'ятовує "після add повернутись на рядок, де присвоюється sum". Ця адреса зберігається у кадрі, і коли `add` завершується, процесор читає адресу та стрибає назад у потрібне місце.

Далі — параметри функції. Значення аргументів (для Copy-типів — копії значень) розміщуються в кадрі як локальні змінні. Параметр `a: i32` у функції `add(a: i32, b: i32)` — це комірка пам'яті всередині кадру `add`, заповнена копією переданого значення.

Локальні змінні — все, що оголошено через `let` всередині функції. `let result = a + b;` створює ще одну комірку `result` у кадрі.

Нарешті — збережені значення регістрів процесора. Це технічна деталь, яку ми тут не розглядатимемо, але важливо знати: процесор зберігає свій робочий стан перед входом у нову функцію, щоб відновити його після повернення.

### Як працює Stack Pointer

Процесор має спеціальний регістр — stack pointer (SP, вказівник стеку). Він завжди вказує на поточну вершину стеку. Коли функція викликається, SP зсувається вниз (на більшості архітектур стек росте від вищих адрес до нижчих), виділяючи місце для нового кадру. Коли функція повертається, SP зсувається вгору, ефективно звільняючи кадр.

Ніякого "прибирання" не потрібно — дані залишаються у пам'яті, але вони вже за межами стеку і будуть перезаписані наступним викликом. Тому стек надзвичайно швидкий: виділення пам'яті — це зсув одного числа. Порівняйте з купою (heap), де потрібен пошук вільного блоку, оновлення списків, можливе дефрагментування.

### Покрокова ілюстрація

Прослідкуємо виконання конкретної програми крок за кроком:

```rust
fn add(a: i32, b: i32) -> i32 {
    let result = a + b;
    result
}

fn multiply(x: i32, y: i32) -> i32 {
    let product = x * y;
    product
}

fn main() {
    let val = 5;
    let sum = add(val, 3);
    let prod = multiply(sum, 2);
    println!("Результат: {}", prod);
}
```

**Крок 1: Початок main.** На стеку створюється кадр для `main()`. У ньому — місце для `val`, `sum`, `prod`.

```
Стек:
┌──────────────────┐
│ main()           │
│   val = 5        │
│   sum = ?        │
│   prod = ?       │
└──────────────────┘ ← вершина
```

**Крок 2: Виклик add(5, 3).** На стеку зверху додається кадр для `add()`. Аргумент `val` (5) копіюється у параметр `a`. Літерал `3` копіюється у `b`. Крім того, у кадрі зберігається адреса повернення — рядок у main, де присвоюється `sum`.

```
Стек:
┌──────────────────┐
│ add()            │
│   a = 5 (копія)  │
│   b = 3 (копія)  │
│   result = ?     │
│   [return → main]│
└──────────────────┘ ← вершина
┌──────────────────┐
│ main()           │
│   val = 5        │
│   sum = ?        │
│   prod = ?       │
└──────────────────┘
```

Зверніть увагу: `a` у кадрі `add` — це копія `val` з кадру `main`. Це два різних місця у пам'яті з однаковим значенням.

**Крок 3: add обчислює result = 8, повертає 8.** Кадр `add` знімається зі стеку. Значення 8 передається у `main`, де присвоюється `sum`.

```
Стек:
┌──────────────────┐
│ main()           │
│   val = 5        │
│   sum = 8        │
│   prod = ?       │
└──────────────────┘ ← вершина
```

Після повернення з `add` усі її локальні змінні (`a`, `b`, `result`) знищені — їхня пам'ять звільнена зі стеку.

**Крок 4: Виклик multiply(8, 2).** Новий кадр на стеку. Цікаво, що він може зайняти ту саму фізичну пам'ять, де раніше був кадр `add` — стек перевикористовує простір.

```
Стек:
┌──────────────────┐
│ multiply()       │
│   x = 8 (копія)  │
│   y = 2 (копія)  │
│   product = ?    │
│   [return → main]│
└──────────────────┘ ← вершина
┌──────────────────┐
│ main()           │
│   val = 5        │
│   sum = 8        │
│   prod = ?       │
└──────────────────┘
```

**Крок 5: multiply повертає 16.** Кадр знімається, `prod = 16` у main.

### Вкладені виклики

Функції можуть викликати інші функції, створюючи ланцюжок кадрів на стеку. Розглянемо програму з трьома рівнями:

```rust
fn inner() -> i32 {
    let x = 42;
    x
}

fn middle() -> i32 {
    let temp = inner();
    temp + 1
}

fn main() {
    let result = middle();
    println!("{}", result); // 43
}
```

У момент виконання `inner()` стек має три кадри одночасно:

```
┌─────────────┐
│ inner()     │  ← виконується зараз
│   x = 42    │
└─────────────┘
┌─────────────┐
│ middle()    │  ← чекає повернення inner
│   temp = ?  │
└─────────────┘
┌─────────────┐
│ main()      │  ← чекає повернення middle
│   result = ?│
└─────────────┘
```

Коли `inner` повертається — залишається два кадри. Коли `middle` повертається — один. Це і є "стек" викликів: кожен виклик додає рівень, кожне повернення — знімає.

### Переповнення стеку та рекурсія

Стек має обмежений розмір — зазвичай від 1 до 8 мегабайт. Що станеться, якщо функція викликає себе без зупинки?

```rust
fn infinite() {
    infinite(); // кожен виклик додає кадр
}

fn main() {
    infinite(); // stack overflow!
}
```

Кожен виклик `infinite()` додає кадр. Через мільйони викликів стек вичерпується, і програма аварійно завершується з помилкою "thread 'main' has overflowed its stack". Це називається stack overflow (переповнення стеку).

Рекурсія (коли функція викликає себе) — корисний інструмент, але вона потребує обов'язкової умови зупинки (base case):

```rust
fn factorial(n: u64) -> u64 {
    if n <= 1 { 1 } else { n * factorial(n - 1) }
}
```

При `factorial(5)` на стеку з'являється лише 5 кадрів — це безпечно. Але `factorial(100000)` потребує 100 000 кадрів, що може перевищити розмір стеку. Для великих чисел краще використовувати цикли замість рекурсії.

### Чому call stack пояснює поведінку функцій

Ця модель пояснює кілька речей. Чому локальні змінні функції не видимі ззовні — вони існують тільки у кадрі, який знищується при поверненні. Чому передача за значенням не змінює оригінал — бо значення копіюється в окремий кадр. Чому рекурсія може переповнити стек — кожен рекурсивний виклик додає новий кадр. Чому стек такий швидкий — виділення пам'яті це лише зсув одного числа (stack pointer).

### Спостереження за стеком через println

Щоб побачити, як стек працює на практиці, додамо `println!` на вході та виході кожної функції:

```rust
fn inner() -> i32 {
    println!("  → увійшли в inner");
    let x = 42;
    println!("  ← виходимо з inner, повертаємо {}", x);
    x
}

fn middle() -> i32 {
    println!(" → увійшли в middle");
    let temp = inner();
    let result = temp + 1;
    println!(" ← виходимо з middle, повертаємо {}", result);
    result
}

fn main() {
    println!("→ увійшли в main");
    let result = middle();
    println!("← main: result = {}", result);
}
```

Вивід:

```
→ увійшли в main
 → увійшли в middle
  → увійшли в inner
  ← виходимо з inner, повертаємо 42
 ← виходимо з middle, повертаємо 43
← main: result = 43
```

Бачите, як відступи показують глибину стеку? `main` викликає `middle` (один відступ), `middle` викликає `inner` (два відступи). Повернення — у зворотному порядку: спершу `inner`, потім `middle`, потім `main` продовжує. Це LIFO в дії: `inner` увійшла останньою — вийшла першою.

## 11.5. Передача за значенням: копія у кадрі функції

Тепер, маючи модель call stack, розглянемо передачу за значенням детально.

### Механізм

Коли ви передаєте аргумент у функцію, Rust (для Copy-типів: числа, bool, char) створює побітову копію значення у кадрі викликаної функції. Функція працює зі своєю копією. Оригінал у кадрі функції, що викликала, залишається незмінним.

Що означає "побітова копія"? Для `i32` — це 4 байти. Процесор бере ці 4 байти і записує ідентичну послідовність в інше місце — у кадр нової функції. Як ксерокопія документа: оригінал і копія виглядають однаково, але це два окремих фізичних об'єкти.

Аналогія: ви фотографуєте документ і надсилаєте фото колезі. Колега може домалювати що завгодно на фото — ваш оригінал не постраждає. Передача за значенням — це "надіслати фото".

### Простий приклад

Ця програма демонструє, що зміна параметра всередині функції не впливає на оригінальну змінну:

```rust
fn drain_battery(mut level: u8) {
    level = level.saturating_sub(30);
    println!("  Всередині: {}%", level);
}

fn main() {
    let battery: u8 = 100;
    println!("До виклику: {}%", battery);
    drain_battery(battery);
    println!("Після виклику: {}%", battery);
}
```

Вивід:

```
До виклику: 100%
  Всередині: 70%
Після виклику: 100%
```

Як це працює на рівні стеку:

```
Виклик drain_battery(100):
┌─────────────────────┐
│ drain_battery()     │
│   level = 100→70    │  ← копія, змінюється
└─────────────────────┘
┌─────────────────────┐
│ main()              │
│   battery = 100     │  ← оригінал, НЕ змінюється
└─────────────────────┘
```

`level` у кадрі `drain_battery` — це окрема комірка пам'яті, ініціалізована копією значення `battery`. Зміна `level` змінює лише цю комірку. Після повернення кадр `drain_battery` знищується разом із `level`.

`mut` перед параметром `level` означає: "ця локальна копія може бути змінена всередині функції". Це не впливає на оригінал — це дозвіл змінювати копію.

### Коли працює побітове копіювання

Copy-типи — це типи, для яких побітове копіювання є коректним і дешевим: `i8`, `i16`, `i32`, `i64`, `i128`, `u8`, `u16`, `u32`, `u64`, `u128`, `f32`, `f64`, `bool`, `char`, а також кортежі та масиви, де всі елементи — Copy. Для цих типів копіювання автоматичне і непомітне.

Для `String`, `Vec` та інших не-Copy типів працює інший механізм — move (переміщення). При переміщенні оригінал стає недоступним після передачі у функцію. Ми детально розглянемо це у Частині II (Розділ 13, Ownership). Поки що просто запам'ятайте: числа, bool, char копіюються автоматично; String та Vec — ні.

### Коли передача за значенням найкраща

Для малих типів (i32 — 4 байти, f64 — 8 байт) копіювання дешевше за створення посилання. Посилання саме по собі займає 8 байт (на 64-бітній системі), плюс процесор має додатково "розіменовувати" його при кожному зверненні. Тому для чисел та bool передача за значенням — це стандарт і найефективніший варіант.

### Кілька параметрів: кожен копіюється окремо

Коли функція має кілька параметрів, кожен аргумент копіюється незалежно:

```rust
fn move_agent(x: f64, y: f64, dx: f64, dy: f64) -> (f64, f64) {
    (x + dx, y + dy)  // повертаємо нову позицію
}

fn main() {
    let pos_x = 10.0;
    let pos_y = 20.0;
    let (new_x, new_y) = move_agent(pos_x, pos_y, 5.0, -3.0);
    println!("Було: ({}, {})", pos_x, pos_y);     // 10.0, 20.0 — не змінились
    println!("Стало: ({}, {})", new_x, new_y);     // 15.0, 17.0
}
```

Функція `move_agent` отримує чотири копії. Вона обчислює нову позицію і повертає її як кортеж. Оригінальні `pos_x` та `pos_y` залишаються незмінними. Цей підхід — "обчислити нове значення замість зміни старого" — є ідіоматичним для Rust і функціонального програмування загалом.

Зверніть увагу на повернення кортежу `(f64, f64)`. Коли функція має повернути кілька значень, кортеж — найпростіший спосіб. У місці виклику використовується деструктуризація: `let (new_x, new_y) = move_agent(...)` розпаковує кортеж у дві окремі змінні.

## 11.6. Передача за посиланням: &T — доступ без копіювання

### Навіщо потрібні посилання

Іноді функція потребує доступу до даних, але копіювати їх дорого або непотрібно. Функція `print_sensor_data` виводить масив з 1000 показань — копіювати 8000 байтів (1000 * f64) лише для виводу — марнотратство. Замість цього передаємо посилання — адресу даних у пам'яті.

Посилання записується як `&T`. `&f64` — посилання на дробове число. `&[f64; 100]` — посилання на масив. `&str` — посилання на рядок. Посилання передає не самі дані, а лише їхню адресу (8 байт на 64-бітній системі, незалежно від розміру даних).

### Як це виглядає у пам'яті

```
Передача за значенням (копія):
┌─────────────────┐     ┌─────────────────┐
│ main()          │     │ func()          │
│ data: [f64;100] │ ──→ │ data: [f64;100] │  ← 800 байт скопійовано
│ (800 байт)      │     │ (800 байт)      │
└─────────────────┘     └─────────────────┘

Передача за посиланням (адреса):
┌─────────────────┐     ┌─────────────────┐
│ main()          │     │ func()          │
│ data: [f64;100] │ ←── │ data: &[f64;100]│  ← 8 байт (лише адреса)
│ (800 байт)      │     │ (8 байт)        │
└─────────────────┘     └─────────────────┘
```

При передачі за посиланням дані залишаються у кадрі `main`. Функція `func` отримує лише адресу — 8 байтів замість 800. Коли функція звертається до `data[5]`, процесор йде за адресою в кадр `main` і читає значення звідти.

### Приклад з масивом

Ця програма демонструє, як передати масив у функцію для обчислення середнього значення без копіювання:

```rust
fn print_average(data: &[f64; 5]) {
    let mut sum = 0.0;
    for value in data {
        sum = sum + value;
    }
    let avg = sum / data.len() as f64;
    println!("Середнє: {:.2}", avg);
}

fn main() {
    let readings = [10.0, 20.5, 30.3, 25.1, 18.7];
    print_average(&readings);
    println!("readings доступні: {:?}", readings);
}
```

Як працює цей код: виклик `print_average(&readings)` — амперсанд `&` створює посилання на масив. Параметр `data: &[f64; 5]` — приймає посилання, а не копію. Оригінальний масив залишається у main і доступний після виклику. Функція не може випадково змінити масив — `&T` дає лише право на читання.

### Приклад з рядками

Для рядків передача за посиланням — стандарт. Функції приймають `&str`, а не `String`:

```rust
fn greet_drone(name: &str) {
    println!("Агент '{}' на зв'язку.", name);
}

fn main() {
    let id = String::from("БПЛА-07");
    greet_drone(&id);       // &String → &str автоматично
    greet_drone("БПЛА-12"); // літерал &str напряму
    println!("id доступний: {}", id);
}
```

`&String` автоматично конвертується у `&str` (deref coercion). Тому функція з `&str` параметром приймає і String (через &), і літерали. Це найгнучкіший підхід.

### Гарантія `&T` — тільки читання

`&T` дає лише право на читання. Спроба змінити дані через `&T` — помилка компіляції. Це гарантія: якщо функція приймає `&T`, вона точно не змінить ваші дані. Ви можете передати `&readings` десяти різним функціям, і жодна не зможе зіпсувати масив.

### Слайси: гнучкіший спосіб передачі масивів

У прикладі вище параметр `data: &[f64; 5]` приймає посилання на масив рівно з 5 елементів. Але що, якщо у вас масив з 10 елементів? Або з 3? Доведеться писати окрему функцію для кожного розміру?

Рішення — слайси (slices). Слайс `&[f64]` (без зазначення розміру) — це посилання на послідовність елементів довільної довжини:

```rust
fn average(data: &[f64]) -> f64 {
    if data.is_empty() {
        return 0.0;
    }
    let mut sum = 0.0;
    for value in data {
        sum += value;
    }
    sum / data.len() as f64
}

fn main() {
    let small = [1.0, 2.0, 3.0];
    let big = [10.0, 20.0, 30.0, 40.0, 50.0, 60.0, 70.0];
    
    println!("Середнє (3 елементи): {:.1}", average(&small));
    println!("Середнє (7 елементів): {:.1}", average(&big));
}
```

Вивід:

```
Середнє (3 елементи): 2.0
Середнє (7 елементів): 40.0
```

Одна функція `average` працює з масивами будь-якого розміру. `&small` автоматично конвертується з `&[f64; 3]` у `&[f64]`. Це стандартний підхід у Rust: якщо функція обробляє послідовність елементів — використовуйте `&[T]`, а не `&[T; N]`.

Всередині слайс — це два значення: адреса першого елемента та кількість елементів. Разом — 16 байт (8 для адреси + 8 для довжини), незалежно від кількості елементів. Так само ефективно, як звичайне посилання, але з інформацією про довжину.

## 11.7. Мутабельне посилання: &mut T — зміна оригіналу

### Навіщо потрібне &mut T

`&T` дає читання без зміни. Але іноді функція повинна змінити оригінальні дані: `recharge_battery` збільшує заряд, `update_position` змінює координати. Для цього — `&mut T` (мутабельне посилання).

`&mut T` дає функції право і читати, і записувати за адресою. Оригінальна змінна у caller буде змінена.

### Приклад

Ця програма демонструє зміну змінної через мутабельне посилання:

```rust
fn recharge(battery: &mut u8, amount: u8) {
    *battery = battery.saturating_add(amount);
    println!("  Зарядка: тепер {}%", battery);
}

fn main() {
    let mut charge: u8 = 45;
    println!("До: {}%", charge);
    recharge(&mut charge, 30);
    println!("Після: {}%", charge); // ЗМІНИЛОСЬ!
}
```

Вивід:

```
До: 45%
  Зарядка: тепер 75%
Після: 75%
```

Як працює цей код: три обов'язкових елементи — параметр `battery: &mut u8`, аргумент `&mut charge`, та змінна `let mut charge`. Якщо будь-який з них відсутній — помилка компіляції. Rust вимагає явної згоди на зміну на кожному рівні.

Символ `*` у `*battery = ...` — розіменування: "запиши значення за адресою, на яку вказує посилання". Для методів (`.saturating_add()`) Rust робить розіменування автоматично.

### Як це виглядає у стеку

```
┌─────────────────────┐
│ recharge()          │
│   battery: &mut u8 ─┼──→ charge у main (змінює!)
│   amount: 30        │
└─────────────────────┘
┌─────────────────────┐
│ main()              │
│   charge: 45 → 75   │  ← значення ЗМІНИЛОСЬ
└─────────────────────┘
```

На відміну від передачі за значенням, де `level` і `battery` — різні комірки, тут `battery` у `recharge` — це стрілка (адреса), що вказує на `charge` у `main`. Запис через `*battery = ...` змінює не локальну комірку, а комірку в кадрі `main`.

### Правила безпеки посилань (preview Частини II)

Rust гарантує відсутність data races через три правила:

Правило 1: Може існувати скільки завгодно іммутабельних посилань `&T` одночасно. Читати можуть всі.

Правило 2: Може існувати лише одне мутабельне посилання `&mut T`. Змінювати може тільки один.

Правило 3: Не можна мати `&T` та `&mut T` одночасно. Якщо хтось змінює — ніхто не може читати, і навпаки.

Ці правила перевіряються компілятором. Код, що їх порушує, не скомпілюється. Ми детально розберемо ці правила в Розділі 16 (Borrowing).

### Практичний патерн: послідовні модифікації через &mut

У реальних програмах часто потрібно викликати кілька функцій, що модифікують одні й ті самі дані. Це працює, бо кожен виклик — послідовний, а не одночасний:

```rust
fn accelerate(speed: &mut f64, delta: f64) {
    *speed += delta;
    if *speed < 0.0 { *speed = 0.0; }
}

fn turn(heading: &mut u16, degrees: i16) {
    let new = (*heading as i16 + degrees) % 360;
    *heading = if new < 0 { (new + 360) as u16 } else { new as u16 };
}

fn consume_fuel(battery: &mut u8, amount: u8) {
    *battery = battery.saturating_sub(amount);
}

fn main() {
    let mut speed: f64 = 0.0;
    let mut heading: u16 = 0;
    let mut battery: u8 = 100;

    // Послідовні виклики — кожен &mut існує лише під час виклику
    accelerate(&mut speed, 15.0);
    turn(&mut heading, 90);
    consume_fuel(&mut battery, 5);

    println!("Швидкість: {:.1}, курс: {}°, батарея: {}%", speed, heading, battery);
    // Швидкість: 15.0, курс: 90°, батарея: 95%
}
```

Кожен `&mut` існує лише на час виклику відповідної функції. Коли `accelerate` повертається, `&mut speed` зникає, і `speed` знову доступна для наступного `&mut` у `turn` або `consume_fuel`. Rust дозволяє це, бо мутабельні посилання не перекриваються у часі.

### Порівняння трьох способів передачі

| Спосіб | Параметр | Аргумент | Оригінал змінюється? | Копіювання | Типові випадки |
|--------|----------|----------|---------------------|------------|----------------|
| За значенням | `x: T` | `value` | Ні | Повна копія | i32, f64, bool, char |
| За посиланням | `x: &T` | `&value` | Ні | Лише адреса (8 байт) | Масиви, рядки — для читання |
| За &mut | `x: &mut T` | `&mut value` | Так | Лише адреса (8 байт) | Коли функція повинна змінити дані |

Як обирати? Починайте з найбільш обмежувального варіанту і послаблюйте за потреби. Спершу спробуйте передати за значенням (для малих типів) або за `&T` (для великих). Якщо функція повинна змінити оригінал — тоді `&mut T`. Якщо можна повернути нове значення замість зміни оригіналу — це часто ще краще.

## 11.8. Повернення значень: функція vs процедура

### Процедура: дія без результату

Функція, що виконує дію, але не повертає змістовного значення — це процедура. У Rust процедура не має `->` у сигнатурі і неявно повертає `()` (unit type):

```rust
fn report_status(battery: u8) {
    println!("Батарея: {}%", battery);
} // неявно повертає ()
```

`()` — це не "нічого". Це повноцінний тип з одним можливим значенням: `()`. Аналог `void` у C, але в Rust це тип, а не "відсутність типу".

### Функція: обчислення результату

Функція з `->` обчислює та повертає значення:

```rust
fn calculate_distance(x1: f64, y1: f64, x2: f64, y2: f64) -> f64 {
    let dx = x2 - x1;
    let dy = y2 - y1;
    (dx * dx + dy * dy).sqrt()
}
```

Сигнатура `-> f64` — контракт: "ця функція гарантовано повертає f64". Компілятор перевіряє, що кожен можливий шлях виконання дійсно повертає f64.

Повернене значення — це звичайне значення, яке можна присвоїти змінній (`let d = calculate_distance(...)`), передати як аргумент (`print_distance(calculate_distance(...))`), або використати у виразі (`if calculate_distance(...) > 100.0`).

### Повернення кількох значень через кортеж

Іноді функція має повернути більше ніж одне значення. Наприклад, функція `minmax` повертає і мінімум, і максимум. Для цього використовується кортеж:

```rust
fn minmax(a: f64, b: f64, c: f64) -> (f64, f64) {
    let mut min = a;
    let mut max = a;
    if b < min { min = b; }
    if b > max { max = b; }
    if c < min { min = c; }
    if c > max { max = c; }
    (min, max) // повертаємо кортеж з двох значень
}

fn main() {
    let (lo, hi) = minmax(25.0, 10.0, 42.0);
    println!("Мін: {}, Макс: {}", lo, hi); // Мін: 10, Макс: 42
}
```

`(f64, f64)` у сигнатурі означає: функція повертає кортеж з двох дробових чисел. У місці виклику використовується деструктуризація: `let (lo, hi) = minmax(...)` розпаковує кортеж у дві окремі змінні.

Інший приклад — функція, що розбиває координати на компоненти:

```rust
fn polar_to_cartesian(distance: f64, angle_deg: f64) -> (f64, f64) {
    let angle_rad = angle_deg * std::f64::consts::PI / 180.0;
    let x = distance * angle_rad.cos();
    let y = distance * angle_rad.sin();
    (x, y)
}

fn main() {
    let (dx, dy) = polar_to_cartesian(100.0, 45.0);
    println!("Зміщення: ({:.1}, {:.1})", dx, dy);
    // Зміщення: (70.7, 70.7)
}
```

Кортежі — найпростіший спосіб повернути кілька значень. Для двох-трьох значень це зручно. Для більшої кількості краще створити структуру (Розділ 17).

### Guard clauses: ранній вихід з return

Типовий патерн — перевірити "погані" випадки на початку функції та повернутись рано, а "щасливий шлях" залишити в основному тілі:

```rust
fn navigate_to(x: f64, y: f64, battery: u8) -> &'static str {
    if battery < 10 {
        return "Батарея критична! Неможливо летіти.";
    }
    if x < 0.0 || y < 0.0 {
        return "Некоректні координати!";
    }
    if x == 0.0 && y == 0.0 {
        return "Вже на місці.";
    }
    // Основна логіка — тільки для валідних випадків
    "Навігація розпочата."
}
```

Кожен `return` — ранній вихід при виявленні проблеми. Без guard clauses цей код мав би глибоко вкладені `if`/`else`, які важко читати. Guard clauses роблять код плоским та зрозумілим: спершу перевірки, потім основна логіка.

### Що краще: змінити через &mut чи повернути нове значення?

Порівняємо два підходи до задачі "обмежити висоту":

```rust
// Підхід 1: зміна через &mut
fn clamp_altitude_mut(alt: &mut f64, max: f64) {
    if *alt > max { *alt = max; }
    if *alt < 0.0 { *alt = 0.0; }
}

// Підхід 2: повернення нового значення
fn clamp_altitude(alt: f64, max: f64) -> f64 {
    if alt > max { max }
    else if alt < 0.0 { 0.0 }
    else { alt }
}
```

Підхід 2 (повернення) зазвичай кращий з кількох причин. Він не потребує `mut` — ані в параметрі, ані в caller. Менше mut — менше місць, де дані можуть несподівано змінитися. Він чистий (pure): результат залежить тільки від аргументів, без побічних ефектів. Чисті функції простіше тестувати та міркувати про їхню поведінку. Він композабельний: `let alt = clamp_altitude(requested, MAX);` — одна зрозуміла операція.

Коли `&mut` все ж потрібен? Коли функція змінює великий об'єкт і копіювання було б дорогим. Або коли потрібно змінити кілька полів одночасно. Ми побачимо це у Частині II, коли з'являться структури.

## 11.9. Вираз та інструкція: фундаментальна відмінність

Ця тема стає критичною при поверненні значень з функцій. У Rust більшість конструкцій є виразами — вони обчислюються у значення. Це одна з фундаментальних особливостей мови, яка впливає на стиль написання коду.

### Що таке вираз (expression)

Вираз — це будь-яка конструкція, що обчислюється у значення певного типу. Кожен вираз має тип і повертає значення. Приклади:

- `5` — літерал, тип `i32`, значення 5
- `2 + 3` — арифметичний вираз, тип `i32`, значення 5
- `x * y` — вираз зі змінними, тип залежить від типів `x` та `y`
- `calculate_distance(0.0, 0.0, 3.0, 4.0)` — виклик функції, тип `f64`, значення 5.0
- `if a > b { a } else { b }` — if-вираз, тип `i32` (якщо a та b — i32)
- `{ let x = 5; x + 1 }` — блок, тип `i32`, значення 6

Зверніть увагу на останні два приклади. В більшості мов програмування `if` — це лише інструкція для розгалуження. В Rust `if` — це вираз, що повертає значення. Так само блок `{ ... }` — це вираз, значення якого визначається останнім рядком.

### Що таке інструкція (statement)

Інструкція — це конструкція, що виконує дію, але не має значення. Інструкція не обчислюється ні в що — вона просто "робить щось". Приклади:

- `let x = 5;` — оголошення змінної
- `println!("text");` — вивід на екран
- `x = 10;` — присвоєння

Ви не можете написати `let y = (let x = 5);` — `let` це інструкція, і вона не має значення, яке можна було б присвоїти `y`. Компілятор видасть помилку: "expected expression, found `let` statement".

### Крапка з комою: перемикач між виразом та інструкцією

Ось ключова деталь Rust, яка відрізняє його від більшості мов: крапка з комою `;` перетворює вираз на інструкцію.

`5 + 3` — вираз зі значенням 8.
`5 + 3;` — інструкція: обчислює 8 і відкидає результат, "повертаючи" `()`.

Це як різниця між "порахуй і скажи мені відповідь" (вираз) та "порахуй, але тримай при собі" (інструкція).

Чому це критично для функцій?

```rust
// ПРАВИЛЬНО: останній рядок — вираз, повертає i32
fn add_ok(a: i32, b: i32) -> i32 {
    a + b    // вираз — значення повертається
}

// НЕПРАВИЛЬНО: крапка з комою перетворила вираз на інструкцію
// fn add_bad(a: i32, b: i32) -> i32 {
//     a + b;   // інструкція — повертає (), а не i32
// }
```

Якщо розкоментувати `add_bad`, компілятор скаже:

```
error: mismatched types — expected i32, found ()
help: consider removing this semicolon
```

Компілятор навіть підказує: "видаліть крапку з комою". Це одна з найчастіших помилок початківців у Rust, і компілятор чудово її діагностує.

### Таблиця: що є виразом, а що інструкцією

| Конструкція | Вираз чи інструкція? | Тип / Значення |
|-------------|---------------------|---------------|
| `42` | Вираз | `i32`, значення 42 |
| `x + y` | Вираз | залежить від x та y |
| `func(arg)` | Вираз | тип повернення func |
| `if c { a } else { b }` | Вираз | тип a (= тип b) |
| `{ let a = 1; a + 2 }` | Вираз | `i32`, значення 3 |
| `let x = 5;` | Інструкція | — |
| `println!("...");` | Інструкція | — (повертає ()) |
| `x + y;` | Інструкція | — (значення відкинуто) |

### Блоки як вирази

У Rust блок `{ ... }` — це вираз. Значення блоку — це значення останнього виразу в ньому (без `;`):

```rust
fn main() {
    let result = {
        let a = 10;
        let b = 20;
        a + b       // без ; — значення блоку = 30
    };
    println!("Результат: {}", result); // 30
}
```

Блок `{ let a = 10; let b = 20; a + b }` обчислюється у 30. `let` та `println!` — інструкції всередині блоку. `a + b` — останній вираз, його значення стає значенням блоку.

Якщо останній рядок блоку закінчується на `;`, блок повертає `()`:

```rust
let result = {
    let a = 10;
    let b = 20;
    a + b;          // з ; — це інструкція, блок повертає ()
};
// result має тип (), а не i32
```

### if/else як вираз

`if`/`else` — теж вираз (ми вже бачили це в Розділі 6):

```rust
fn categorize(level: u8) -> &'static str {
    if level < 15 { "критичний" }
    else if level < 30 { "низький" }
    else if level < 80 { "нормальний" }
    else { "повний" }
}
```

Весь `if`/`else if`/`else` — один вираз. Жоден рядок не має `;` (бо кожен — вираз, що повертає значення). Результат — рядок, обраний відповідною гілкою.

Оскільки `if` це вираз, його можна використати для ініціалізації змінної:

```rust
let status = if battery > 50 { "OK" } else { "LOW" };
```

Це аналог тернарного оператора `? :` з C та Java. В Rust немає окремого тернарного оператора — його роль виконує `if` як вираз.

Важливе обмеження: обидві гілки `if` та `else` повинні повертати однаковий тип. `if true { 5 } else { "text" }` не скомпілюється — `i32` та `&str` це різні типи.

### Як expression-oriented підхід змінює стиль коду

Замість тимчасової змінної — ініціалізація через вираз:

```rust
// Стиль із тимчасовою mut-змінною
let mut result = 0;
if condition { result = 10; } else { result = 20; }

// Ідіоматичний Rust: if як вираз
let result = if condition { 10 } else { 20 };
```

Другий варіант кращий: `result` можна зробити іммутабельним (`let`, а не `let mut`), бо він ініціалізується рівно один раз. Менше `mut` — менше місць, де значення може несподівано змінитися.

## 11.10. Неявне та явне повернення

### Неявне повернення (implicit return)

Останній вираз у тілі функції (без `;`) автоматично стає поверненим значенням. Це ідіоматичний спосіб у Rust:

```rust
fn fuel_for_distance(km: f64, consumption: f64) -> f64 {
    km * consumption  // implicit return
}
```

### Явне повернення (explicit return)

Ключове слово `return` з наступним значенням:

```rust
fn fuel_for_distance(km: f64, consumption: f64) -> f64 {
    return km * consumption; // explicit return — з ;
}
```

`return` в основному використовується для раннього виходу з функції — коли потрібно перервати виконання в середині тіла:

```rust
fn safe_divide(a: f64, b: f64) -> f64 {
    if b == 0.0 {
        return 0.0; // ранній вихід
    }
    // Дійшли сюди = b != 0.0
    a / b  // implicit return
}
```

Без `return` тут не обійтись: якщо `b == 0.0`, ми хочемо повернути 0.0 одразу, не виконуючи решту функції.

Правило стилю: implicit return — за замовчуванням. `return` — тільки для раннього виходу. `cargo clippy` попередить, якщо `return` на останньому рядку — це зайве.

## 11.11. Область дії параметрів та локальних змінних

### Ізоляція між функціями

Кожна функція — ізольований "світ". Вона бачить тільки свої параметри та локальні змінні. Не бачить змінних функції-caller. І caller не бачить локальних змінних функції.

```rust
fn calculate_fuel(distance: f64, rate: f64) -> f64 {
    let fuel = distance * rate; // fuel — локальна
    fuel
}

fn main() {
    let route = 15.0;
    let consumption = 0.5;
    let total = calculate_fuel(route, consumption);
    println!("Пальне: {:.1}", total);
    // println!("{}", fuel);      // ПОМИЛКА: fuel не існує в main
    // println!("{}", distance);  // ПОМИЛКА: distance не існує в main
}
```

`fuel` існує тільки в кадрі `calculate_fuel`. Параметри `distance`, `rate` — теж. `main` бачить лише `route`, `consumption`, `total`.

Якби `calculate_fuel` напряму читала `route` з `main`, будь-яка зміна `route` непередбачувано впливала б на `calculate_fuel`. Параметри роблять залежності явними: функція отримує рівно те, що передано, і нічого більше.

### Параметри — це локальні змінні

Параметр — це, по суті, локальна змінна, ініціалізована значенням аргументу. Він створюється при вході у функцію і знищується при виході:

```rust
fn process(mut value: i32) {
    value = value * 2; // змінюємо локальну копію
    println!("Всередині: {}", value);
} // value знищується тут

fn main() {
    let x = 10;
    process(x);
    println!("Зовні: {}", x); // 10, не 20
}
```

Вивід:

```
Всередині: 20
Зовні: 10
```

`value` у `process` — це нова змінна, що існує тільки протягом виклику. Вона ініціалізована копією `x` (число 10). Зміна `value` на 20 не впливає на `x`, бо це різні комірки пам'яті у різних кадрах стеку. Після повернення з `process` кадр знищується разом із `value`.

### Чому ізоляція — це добре

Якби `calculate_fuel` напряму читала змінну `route` з `main`, будь-яка зміна `route` непередбачувано впливала б на `calculate_fuel`. Програміст, що читає `calculate_fuel`, не зміг би зрозуміти її поведінку без перевірки всіх місць, де `route` змінюється.

Параметри роблять залежності явними: функція отримує рівно те, що передано, і нічого більше. Побачивши `fn calculate_fuel(distance: f64, rate: f64) -> f64`, ви точно знаєте: ця функція залежить від двох чисел і більше ні від чого. Зміна будь-якої іншої змінної в програмі не вплине на її результат. Саме це робить функції передбачуваними та тестованими.

### Вкладені блоки в функціях

Правила scope з Розділу 6 працюють і всередині функцій:

```rust
fn analyze(alt: f64) -> &'static str {
    let threshold = 200.0;
    if alt > threshold {
        let excess = alt - threshold; // лише в if-блоці
        println!("Перевищення: {:.0} м", excess);
        "високо"
    } else {
        // excess тут не існує
        "нормально"
    }
    // excess тут теж не існує
}
```

`threshold` — у всій функції. `excess` — лише в if-блоці.

### Shadowing у функціях

Ім'я параметра може бути затінене локальною змінною:

```rust
fn process_input(input: &str) -> String {
    let input = input.trim();           // shadowing: &str → &str (trimmed)
    let input = input.to_lowercase();   // shadowing: &str → String
    input                               // повертаємо String
}
```

Кожне `let input` створює нову змінну, що затіняє попередню. Тип змінюється: `&str` → `&str` → `String`. Це ідіоматичний патерн для покрокової обробки даних.

## 11.12. Композиція функцій

Сила функцій не лише в ізоляції — а й у можливості комбінувати їх. Результат однієї функції стає аргументом іншої. Прості функції, з'єднані разом, вирішують складні задачі.

### Послідовна композиція

Найпростіший спосіб — зберігати проміжні результати у змінних:

```rust
fn distance(x1: f64, y1: f64, x2: f64, y2: f64) -> f64 {
    let dx = x2 - x1;
    let dy = y2 - y1;
    (dx * dx + dy * dy).sqrt()
}

fn fuel_needed(dist: f64, rate: f64) -> f64 {
    dist * rate
}

fn can_reach(available: f64, required: f64) -> bool {
    available >= required
}

fn main() {
    let dist = distance(0.0, 0.0, 3.0, 4.0);
    let fuel = fuel_needed(dist, 1.5);
    let ok = can_reach(10.0, fuel);
    println!("Відстань: {:.1}, пальне: {:.1}, дістанемось: {}", dist, fuel, ok);
}
```

Кожна функція проста — 1-3 рядки. Разом — вирішують задачу "чи може БПЛА долетіти до точки". Якщо потрібно змінити формулу витрати пального — змінюєте одну функцію `fuel_needed`, а `main` та `can_reach` не зачеплені.

### Вкладені виклики

Замість збереження у змінних можна вкладати виклики один в одний:

```rust
fn main() {
    let ok = can_reach(
        10.0,
        fuel_needed(distance(0.0, 0.0, 3.0, 4.0), 1.5)
    );
    println!("Дістанемось: {}", ok);
}
```

Виконання відбувається зсередини назовні: спершу `distance(...)` повертає 5.0, потім `fuel_needed(5.0, 1.5)` повертає 7.5, потім `can_reach(10.0, 7.5)` повертає true. На рівні стеку кожен виклик додає кадр, і вони розгортаються у зворотному порядку.

Вкладені виклики компактніші, але менш читабельні. Якщо проміжні значення потрібні для виводу або дебагу — використовуйте змінні. Якщо важлива лише кінцева відповідь — вкладені виклики можуть бути зручнішими.

### Правило: функція робить одну річ

Хороша функція має одну відповідальність. `distance` обчислює відстань — і все. Вона не перевіряє батарею, не виводить текст, не змінює координати. Якщо назва функції потребує слова "та" — це сигнал, що функцію варто розбити на дві: `calculate_distance_and_check_battery` → `calculate_distance` + `check_battery`.

Цей принцип називається Single Responsibility Principle (SRP) — принцип єдиної відповідальності. Він робить код простішим для розуміння, тестування та зміни.

## 11.13. Практика: декомпозиція БПЛА-агента

У попередніх розділах (5–10) ми писали весь код в одному `main()`. Координати, батарея, команди — все в одній функції. Зараз покажемо, як той самий функціонал виглядає після декомпозиції на функції. Кожна функція має одну відповідальність, а `main` стає послідовністю зрозумілих викликів.

### Функції агента

Розіб'ємо логіку на п'ять функцій:

`battery_status` — чиста функція, що повертає текстовий опис рівня заряду. Приймає `u8` за значенням (копія), повертає `&'static str`. Не змінює нічого.

`consume` — процедура, що зменшує заряд. Приймає `&mut u8`, бо повинна змінити оригінальну змінну. Використовує `saturating_sub`, щоб не було переповнення при відніманні (якщо заряд менший за amount, результат — 0, а не паніка).

`clamp` — чиста функція, що обмежує значення діапазоном. Приймає три `f64` за значенням (копії), повертає нове `f64`. Не змінює жодного аргументу.

`print_status` — процедура виводу. Приймає дані лише для читання: `&str` для рядка, `f64` та `u8` за значенням (копії), `u16` за значенням. Зверніть увагу: вона сама викликає `battery_status` — це композиція функцій.

`main` — тільки ініціалізація та виклики. Жодної обчислювальної логіки.

```rust
fn battery_status(level: u8) -> &'static str {
    if level < 15 { "КРИТИЧНИЙ" }
    else if level < 30 { "низький" }
    else if level < 80 { "нормальний" }
    else { "повний" }
}

fn consume(battery: &mut u8, amount: u8) {
    *battery = battery.saturating_sub(amount);
}

fn clamp(value: f64, min: f64, max: f64) -> f64 {
    if value < min { min }
    else if value > max { max }
    else { value }
}

fn print_status(id: &str, alt: f64, bat: u8, hdg: u16) {
    println!("--- {} ---", id);
    println!("  Висота: {:.0} м, курс: {}°", alt, hdg);
    println!("  Батарея: {}% [{}]", bat, battery_status(bat));
}

fn main() {
    let id = "БПЛА-07";
    let mut battery: u8 = 100;
    let mut altitude: f64 = 0.0;

    print_status(id, altitude, battery, 0);

    altitude = clamp(350.0, 0.0, 400.0);
    consume(&mut battery, 15);
    print_status(id, altitude, battery, 90);

    altitude = clamp(500.0, 0.0, 400.0); // 500 → обмежено до 400
    consume(&mut battery, 10);
    print_status(id, altitude, battery, 180);
}
```

Вивід програми:

```
--- БПЛА-07 ---
  Висота: 0 м, курс: 0°
  Батарея: 100% [повний]
--- БПЛА-07 ---
  Висота: 350 м, курс: 90°
  Батарея: 85% [нормальний]
--- БПЛА-07 ---
  Висота: 400 м, курс: 180°
  Батарея: 75% [нормальний]
```

### Аналіз декомпозиції

Порівняємо з монолітним підходом. У монолітній версії `main` містив би всю логіку: умови для статусу батареї, математику обмеження висоти, форматування виводу — все перемішане. Зміна формату виводу потребувала б пошуку потрібних рядків серед десятків інших. Зміна правил обмеження висоти — ризик зачепити щось інше.

У декомпозованій версії кожна зміна локалізована. Потрібно змінити пороги батареї? Йдете у `battery_status`, змінюєте числа — решта програми не зачеплена. Потрібно інший алгоритм обмеження? Змінюєте `clamp` — і все, що його використовує, автоматично отримує нову поведінку.

Зверніть увагу на вибір передачі даних у кожній функції:

| Функція | Що приймає | Чому саме так |
|---------|-----------|---------------|
| `battery_status` | `level: u8` (значення) | Маленький Copy-тип, лише читає |
| `consume` | `battery: &mut u8` | Повинна змінити оригінал |
| `clamp` | `value: f64` (значення) | Маленький Copy-тип, повертає нове значення |
| `print_status` | `id: &str`, решта за значенням | Рядок — за посиланням, числа — копії |

Ця таблиця — шпаргалка для вибору способу передачі. Маленький тип, лише читати — за значенням. Рядок або масив, лише читати — `&T`. Потрібно змінити — `&mut T`. Потрібно обчислити нове значення — повернути результат.

### Приклад виконання базового рівня

Фрагмент рішення — функції для обробки трьох команд:

```rust
use std::io;

fn read_command() -> String {
    let mut input = String::new();
    io::stdin().read_line(&mut input).expect("Помилка читання");
    input.trim().to_lowercase()
}

fn cmd_status(id: &str, x: f64, y: f64, battery: u8) {
    println!("=== {} ===", id);
    println!("  Позиція: ({:.1}, {:.1})", x, y);
    println!("  Батарея: {}%", battery);
}

fn cmd_move(x: &mut f64, y: &mut f64, battery: &mut u8, dx: f64, dy: f64) {
    *x += dx;
    *y += dy;
    *battery = battery.saturating_sub(5);
    println!("  Переміщено на ({:.1}, {:.1})", dx, dy);
}

fn cmd_charge(battery: &mut u8) {
    *battery = 100;
    println!("  Батарея заряджена до 100%");
}

fn main() {
    let id = "БПЛА-01";
    let mut x: f64 = 0.0;
    let mut y: f64 = 0.0;
    let mut battery: u8 = 100;

    loop {
        println!("\nВведіть команду (статус/рух/заряд/вихід):");
        let cmd = read_command();

        if cmd == "статус" {
            cmd_status(id, x, y, battery);
        } else if cmd == "рух" {
            cmd_move(&mut x, &mut y, &mut battery, 10.0, 5.0);
        } else if cmd == "заряд" {
            cmd_charge(&mut battery);
        } else if cmd == "вихід" {
            println!("Завершення.");
            break;
        } else {
            println!("Невідома команда: '{}'", cmd);
        }
    }
}
```

Як працює цей код: `read_command` зчитує рядок з клавіатури та повертає очищену lowercase-версію — це чиста функція з одним результатом. `cmd_status` приймає всі дані за значенням або `&str` — вона лише виводить, нічого не змінює. `cmd_move` приймає `&mut` для координат та батареї — бо вона повинна змінити оригінали. `cmd_charge` приймає `&mut u8` — змінює лише батарею. Функція `main` тепер — лише цикл з диспетчеризацією команд. Логіка кожної команди ізольована у відповідній функції.

## Troubleshooting

### Помилка 1: `expected i32, found ()`

Зайва `;` на останньому рядку. Це найчастіша помилка з функціями в Rust.

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b;   // ← зайва крапка з комою!
}
```

```
error[E0308]: mismatched types
  expected `i32`, found `()`
help: consider removing this semicolon
```

Виправлення: видаліть `;` з останнього рядка: `a + b`

Пояснення: крапка з комою перетворює вираз `a + b` (значення типу i32) на інструкцію `a + b;` (значення типу ()). Функція обіцяє повернути i32, а фактично повертає () — звідси помилка.

### Помилка 2: `expected ';', found '}'`

Не вистачає `;` після інструкції, яка не є останнім рядком.

```rust
fn process(x: i32) -> i32 {
    let doubled = x * 2   // ← тут потрібна ;
    doubled
}
```

Виправлення: додайте `;` після `let doubled = x * 2;`

Пояснення: `let` — це завжди інструкція, і вона потребує `;`. Без неї компілятор не розуміє, де закінчується один рядок і починається наступний.

### Помилка 3: `mismatched types` при поверненні з if/else

Гілки повертають різні типи.

```rust
fn describe(level: u8) -> &'static str {
    if level > 50 {
        "високий"
    } else {
        42       // ← тип i32, а не &str!
    }
}
```

Виправлення: обидві гілки повинні повертати однаковий тип — у цьому випадку `&'static str`.

### Помилка 4: `use of moved value`

String передано за значенням, і оригінал став недоступним.

```rust
fn print_id(id: String) {
    println!("ID: {}", id);
}

fn main() {
    let name = String::from("БПЛА-07");
    print_id(name);
    println!("{}", name);  // ← ПОМИЛКА: name переміщено
}
```

Виправлення: змініть параметр на `id: &str` і викликайте `print_id(&name)`. Функції потрібно лише прочитати рядок, не володіти ним.

### Помилка 5: `cannot borrow as mutable`

Передаєте `&mut x`, але `x` оголошено без `mut`.

```rust
fn increment(val: &mut i32) {
    *val += 1;
}

fn main() {
    let x = 10;           // ← немає mut!
    increment(&mut x);    // ← ПОМИЛКА
}
```

Виправлення: `let mut x = 10;`

### Помилка 6: Функція не змінює змінну

Передали за значенням замість `&mut`:

```rust
fn reset_battery(mut battery: u8) {
    battery = 100;  // змінює КОПІЮ
}

fn main() {
    let mut bat = 50;
    reset_battery(bat);
    println!("{}", bat);  // все ще 50!
}
```

Виправлення: змініть на `fn reset_battery(battery: &mut u8)` та виклик `reset_battery(&mut bat)`.

### Помилка 7: `cannot find value in this scope`

Спроба звернутись до локальної змінної іншої функції.

```rust
fn compute() {
    let result = 42;
}

fn main() {
    compute();
    println!("{}", result);  // ← ПОМИЛКА: result не існує тут
}
```

Виправлення: поверніть значення з функції: `fn compute() -> i32 { 42 }` і `let result = compute();`

## Додатково

### Документування функцій

Rust має вбудовану підтримку документаційних коментарів. Замість звичайного `//` використовується `///` (потрійна коса). Ці коментарі автоматично перетворюються на HTML-документацію командою `cargo doc`:

```rust
/// Обчислює евклідову відстань між двома точками на площині.
///
/// # Аргументи
/// * `x1`, `y1` — координати першої точки
/// * `x2`, `y2` — координати другої точки
///
/// # Повертає
/// Відстань як дробове число (завжди >= 0)
fn distance(x1: f64, y1: f64, x2: f64, y2: f64) -> f64 {
    let dx = x2 - x1;
    let dy = y2 - y1;
    (dx * dx + dy * dy).sqrt()
}
```

Документуйте кожну функцію, яка має нетривіальну логіку. Навіть якщо зараз все зрозуміло — через місяць ви забудете, що саме робить `fn normalize_heading`. Документація допоможе і вам, і колегам, і AI-асистенту, якщо ви попросите його рев'юнути код.

### Конвенції іменування функцій

В Rust прийнято називати функції за шаблонами:

- `get_*` — отримати значення: `get_battery_level()`, `get_position()`
- `set_*` — встановити значення: `set_altitude()`, `set_heading()`
- `is_*` / `has_*` — перевірити умову (повертає bool): `is_airborne()`, `has_enough_fuel()`
- `calculate_*` / `compute_*` — обчислити: `calculate_distance()`, `compute_route()`
- `to_*` — конвертувати: `to_radians()`, `to_string()`
- `from_*` — створити з чогось: `from_degrees()`, `from_coordinates()`
- `try_*` — може не вдатися (повертає Result/Option): `try_parse()`, `try_connect()`
- `print_*` / `display_*` — вивести: `print_status()`, `display_route()`

Ці конвенції — не правила компілятора, а домовленість між Rust-програмістами. Дотримуючись їх, ви робите код зрозумілим для будь-кого, хто знає Rust.

### Функції та ownership (preview)

Для String, Vec та інших не-Copy типів передача за значенням означає move — переміщення. Оригінал стає недоступним після виклику. Це тема Частини II (Розділ 13, Ownership). Поки що просто використовуйте `&str` / `&[T]` у параметрах для рядків та масивів.

### Скільки параметрів — забагато?

Більше 5 — сигнал, що потрібна структура (Розділ 17). `print_status(id, alt, bat, hdg, fly)` → `print_status(drone: &DroneState)`. Поки ми не вивчили структури, групуйте пов'язані параметри в кортежі: `fn move_to(pos: (f64, f64), target: (f64, f64))` замість чотирьох окремих f64.

### Рекурсія

Функція може викликати себе. Класичний приклад — факторіал:

```rust
fn factorial(n: u64) -> u64 {
    if n <= 1 { 1 } else { n * factorial(n - 1) }
}

fn main() {
    println!("5! = {}", factorial(5));   // 120
    println!("10! = {}", factorial(10)); // 3628800
}
```

Кожен рекурсивний виклик додає кадр на стек. При `factorial(5)` — це 5 кадрів (безпечно). При `factorial(100000)` — 100 000 кадрів (стек може переповнитись). Для великих значень використовуйте цикли замість рекурсії:

```rust
fn factorial_loop(n: u64) -> u64 {
    let mut result: u64 = 1;
    for i in 2..=n {
        result *= i;
    }
    result
}
```

Цикл використовує один кадр стеку незалежно від `n` — набагато безпечніше.

## Контрольні запитання

### Рівень 1

1. Чим параметр відрізняється від аргументу?

Відповідь: параметр — змінна в оголошенні функції (наприклад, `x: i32`). Аргумент — конкретне значення при виклику (наприклад, `5` у `func(5)`).

2. Що означає `-> f64`?

Відповідь: функція повертає значення типу `f64` (64-бітне дробове число).

3. Чим `&T` відрізняється від `&mut T`?

Відповідь: `&T` — іммутабельне посилання, дає лише право на читання. `&mut T` — мутабельне посилання, дає право і читати, і записувати.

4. Що повертає функція без `->`?

Відповідь: unit type `()` — спеціальний тип з одним можливим значенням. Це аналог "нічого не повертає".

### Рівень 2

5. Чому при передачі i32 за значенням оригінал не змінюється?

Відповідь: i32 є Copy-типом. При виклику функції створюється побітова копія значення у кадрі стеку функції. Функція працює з копією, а не з оригіналом.

6. Чому `a + b;` повертає `()`, а `a + b` — i32?

Відповідь: крапка з комою перетворює вираз (expression) на інструкцію (statement). `a + b` — вираз зі значенням типу i32. Додавання `;` робить його інструкцією — значення обчислюється, але відкидається, а тип стає `()`.

7. Чому Rust вимагає типи параметрів?

Відповідь: сигнатура функції є повним контрактом. Побачивши `fn f(x: f64) -> bool`, ви знаєте типи входу та виходу без читання тіла. Компілятор перевіряє відповідність типів при кожному виклику.

8. Що таке call stack і stack frame?

Відповідь: call stack — область пам'яті, організована за принципом LIFO. Stack frame — блок у стеку, що створюється при виклику функції та містить параметри, локальні змінні та адресу повернення. При поверненні з функції кадр знищується.

### Рівень 3

9. Напишіть `fn warning(level: u8) -> &'static str` — "КРИТИЧНО" при level < 15, "НИЗЬКО" при level < 30, "ОК" інакше. Без `return`.

Відповідь:

```rust
fn warning(level: u8) -> &'static str {
    if level < 15 { "КРИТИЧНО" }
    else if level < 30 { "НИЗЬКО" }
    else { "ОК" }
}
```

10. Виправте: `fn double(x: i32) -> i32 { let r = x * 2; r; }`

Відповідь: прибрати `;` після `r`: `fn double(x: i32) -> i32 { let r = x * 2; r }`. Або коротше: `fn double(x: i32) -> i32 { x * 2 }`.

11. Напишіть `fn minmax(data: &[f64]) -> (f64, f64)` — мінімум та максимум.

Відповідь:

```rust
fn minmax(data: &[f64]) -> (f64, f64) {
    let mut min = data[0];
    let mut max = data[0];
    for &val in &data[1..] {
        if val < min { min = val; }
        if val > max { max = val; }
    }
    (min, max)
}
```

### Рівень 4

12. 7 параметрів — чому погано? Як вирішити без struct?

Відповідь: 7 параметрів перевищують когнітивне навантаження — легко переплутати порядок. Без struct можна: згрупувати пов'язані параметри у кортежі (наприклад, `(x, y)` замість двох окремих), розбити функцію на менші (кожна з меншою кількістю параметрів), або створити ланцюжок викликів, де кожна функція додає свою частину.

13. Порівняйте: (а) глобальна змінна, (б) &mut, (в) повернення нового значення. Безпека, читабельність, тестованість.

Відповідь: (а) Глобальна змінна — найгірший варіант: будь-хто може змінити в будь-який момент, залежності неявні, тестувати складно (стан між тестами). (б) `&mut` — явна залежність у параметрі, компілятор контролює ексклюзивність, тестувати можна. (в) Повернення нового значення — найкращий варіант: чиста функція без побічних ефектів, легко тестувати (вхід → вихід), не потребує mut. Обирайте (в) за замовчуванням, (б) коли зміна на місці ефективніша, (а) — уникайте.

## Резюме

Функція — іменований блок коду з визначеними входами та виходом. Дві головні причини використання функцій: декомпозиція (розбиття складної задачі на прості частини) та повторне використання (одна функція — багато викликів).

Параметр — змінна в оголошенні функції, що визначає контракт. Аргумент — конкретне значення при виклику. Rust вимагає явні типи параметрів — сигнатура є повною документацією.

Call stack: кожен виклик створює кадр (stack frame) із локальними змінними, параметрами та адресою повернення. Stack pointer відстежує вершину стеку. При поверненні кадр миттєво знищується. Рекурсія може переповнити стек при великій глибині.

Три способи передачі даних: за значенням (`T`) — копія для малих типів, оригінал не змінюється; за посиланням (`&T`) — адреса, лише читання, без копіювання; за мутабельним посиланням (`&mut T`) — адреса з правом запису, змінює оригінал.

Процедура — функція без `->`, повертає `()`. Функція з `->` повертає конкретне значення. Повернення нового значення зазвичай краще за зміну через `&mut`.

Вираз обчислюється у значення. Інструкція виконує дію. Крапка з комою перетворює вираз на інструкцію. Implicit return (без `;`) — ідіоматичний спосіб повернення. Explicit `return` — для раннього виходу.

В Rust `if`, блоки `{ }` — це вирази. Це дозволяє ініціалізувати змінні без `mut`: `let x = if condition { 10 } else { 20 };`

Область дії: функція бачить лише свої параметри та локальні змінні. Повна ізоляція забезпечує передбачуваність та тестованість.

# Розділ 32. Box\<T\>: розміщення на Heap

## Анотація

У Розділі 21 ви дізнались, що Vec зберігає дані на купі через внутрішній вказівник. Box\<T\> — це найпростіший smart pointer у Rust: він володіє одним значенням типу T, розміщеним на купі. Коли Box виходить зі scope — значення на купі автоматично звільняється (RAII). Здавалося б, навіщо окремий тип для цього, якщо Vec і String вже працюють з купою? Box потрібен у трьох ситуаціях: рекурсивні типи (дерево, зв'язний список — де розмір типу не відомий без непрямості), великі дані на стеку (масив на 10 МБ краще розмістити на купі), та trait objects (Box\<dyn Trait\> — ви вже використовували це у Розділах 28–31). Цей розділ пояснює механізм Box, trait Deref та deref coercion, і показує, як рекурсивні типи моделюють дерево рішень агента.

---

## Цілі навчання

Після опрацювання цього розділу студент зможе:

1. Створити Box\<T\> та пояснити, що зберігається на стеку, а що на купі.
2. Пояснити, коли Box необхідний: рекурсивні типи, великі дані, trait objects.
3. Створити рекурсивний тип (дерево, зв'язний список) через Box.
4. Пояснити trait Deref та механізм deref coercion.
5. Використати Box\<dyn Trait\> для динамічного вибору стратегії.

---

## Ключові терміни

**Box\<T\> (бокс)** — smart pointer, що володіє значенням типу T на купі. На стеку зберігається лише вказівник (8 байт).

**Smart pointer (розумний вказівник)** — тип, що поводиться як вказівник, але має додаткову семантику (ownership, reference counting тощо). Box, Rc, Arc, RefCell — smart pointers.

**Deref trait** — trait, що дозволяє типу поводитись як посилання. `*box_value` автоматично розіменовує Box до внутрішнього значення.

**Deref coercion** — автоматична конвертація: `&Box<T>` → `&T`, `&String` → `&str`. Компілятор вставляє розіменування неявно.

**Recursive type (рекурсивний тип)** — тип, що містить себе як поле. Без Box має нескінченний розмір; Box розриває рекурсію через непрямість (вказівник фіксованого розміру).

---

## Мотиваційний кейс

БПЛА приймає рішення через дерево (decision tree): "Якщо батарея < 20% → повернутись. Інакше → якщо є ціль у радіусі → слідувати. Інакше → патрулювати." Кожен вузол дерева містить умову та два піддерева — ліве (умова true) та праве (умова false). Піддерево — це знову вузол із умовою та двома піддеревами. Рекурсивна структура. Без Box компілятор запитає: "який розмір DecisionNode?" — нескінченний, бо кожен вузол містить два вузли, кожен з яких містить два вузли... Box розриває цю нескінченність: вузол містить не два вузли, а два вказівники на вузли (8 байт кожен).

## 32.1. Box: найпростіший smart pointer

Box\<T\> створюється через `Box::new(value)`. Значення переміщується на купу, а на стеку залишається лише вказівник:

```rust
fn main() {
    let stack_value = 42;           // 4 байти на стеку
    let heap_value = Box::new(42);  // 4 байти на купі, 8 байт вказівника на стеку
    
    println!("Стек: {}", stack_value);
    println!("Купа: {}", heap_value); // автоматичне розіменування
    println!("Купа (явно): {}", *heap_value); // явне розіменування через *
}
```

Що відбувається у пам'яті:

```
Стек:                    Купа:
┌──────────────────┐     ┌──────┐
│ stack_value: 42  │     │  42  │
├──────────────────┤     └──────┘
│ heap_value: ptr ─┼──────→ ↑
└──────────────────┘
```

`heap_value` на стеку займає 8 байт (вказівник). Значення 42 живе на купі. Коли `heap_value` виходить зі scope — Box звільняє пам'ять на купі автоматично. Не потрібен ніякий `free()` чи garbage collector.

Для i32 Box не має практичного сенсу — 8 байт вказівника більші за 4 байти самого числа. Box стає корисним для великих даних, рекурсивних типів та trait objects.

## 32.2. Коли Box необхідний

Є три основних сценарії, де Box вирішує проблему, яку інакше вирішити неможливо.

Перший сценарій — рекурсивні типи. Тип, що містить себе як поле, має нескінченний розмір. Компілятор не може виділити місце на стеку для нього:

```rust
// НЕ компілюється: нескінченний розмір
// enum List {
//     Cons(i32, List),  // List містить List, яка містить List...
//     Nil,
// }

// Компілюється: Box<List> — фіксований розмір (8 байт)
enum List {
    Cons(i32, Box<List>),
    Nil,
}

fn main() {
    let list = List::Cons(1, 
        Box::new(List::Cons(2, 
            Box::new(List::Cons(3, 
                Box::new(List::Nil))))));
}
```

Без Box: розмір `List` = sizeof(i32) + sizeof(List) = 4 + (4 + (4 + ...)) = нескінченність. З Box: розмір `List` = sizeof(i32) + sizeof(Box<List>) = 4 + 8 = 12 байт. Box розриває рекурсію через непрямість — замість самого значення зберігається вказівник фіксованого розміру.

Другий сценарій — великі дані. Масив `[f64; 100000]` займає 800 КБ на стеку. Стек зазвичай 1–8 МБ. Кілька таких масивів — stack overflow. Box переносить дані на купу:

```rust
fn main() {
    // let huge = [0.0_f64; 100_000]; // 800 КБ на стеку — ризиковано
    let huge = Box::new([0.0_f64; 100_000]); // 800 КБ на купі, 8 байт на стеку
    println!("Перший елемент: {}", huge[0]);
}
```

Третій сценарій — trait objects. Ви вже використовували `Box<dyn Agent>` у Розділі 28:

```rust
fn create_agent(drone: bool) -> Box<dyn Agent> {
    if drone {
        Box::new(Drone { /* ... */ })
    } else {
        Box::new(GroundBot { /* ... */ })
    }
}
```

`dyn Agent` не має відомого розміру (Drone та GroundBot — різні розміри). Box<dyn Agent> — фіксований: вказівник на дані + вказівник на vtable = 16 байт.

## 32.3. Deref та deref coercion

Box реалізує trait `Deref`, що дозволяє йому поводитись як звичайне посилання. Це означає: де очікується `&T`, можна передати `&Box<T>` — компілятор автоматично розіменує Box:

```rust
fn print_name(name: &str) {
    println!("Ім'я: {}", name);
}

fn main() {
    let boxed_string = Box::new(String::from("SCOUT-01"));
    
    // Deref coercion: &Box<String> → &String → &str
    print_name(&boxed_string);
    
    // Явне розіменування
    let inner: &String = &*boxed_string;
    println!("Довжина: {}", inner.len());
    
    // Але зазвичай deref coercion робить це автоматично:
    println!("Довжина: {}", boxed_string.len()); // Box<String>.len() → String.len()
}
```

Ланцюжок deref coercion: `&Box<String>` → `&String` → `&str`. Компілятор автоматично вставляє стільки розіменувань, скільки потрібно, щоб типи збіглися. Це не runtime операція — компілятор вставляє конкретні інструкції при компіляції.

Саме завдяки Deref ви можете викликати методи String на Box<String> без явного розіменування: `boxed_string.len()` замість `(*boxed_string).len()`. Компілятор автоматично додає `*` де потрібно.

DerefMut працює аналогічно для мутабельного доступу:

```rust
fn main() {
    let mut boxed_vec = Box::new(vec![1, 2, 3]);
    boxed_vec.push(4); // DerefMut: Box<Vec<i32>> → Vec<i32>.push()
    println!("{:?}", boxed_vec); // [1, 2, 3, 4]
}
```

## 32.4. Рекурсивні типи: дерево рішень агента

Побудуємо дерево рішень для БПЛА. Кожен вузол — або рішення (leaf), або умова з двома гілками:

```rust
// Дерево рішень
enum Decision {
    // Листок: конкретна дія
    Action(String),
    // Вузол: умова → ліва гілка (true) / права гілка (false)
    Branch {
        condition: String,
        if_true: Box<Decision>,
        if_false: Box<Decision>,
    },
}

impl Decision {
    fn action(name: &str) -> Self {
        Decision::Action(String::from(name))
    }
    
    fn branch(condition: &str, if_true: Decision, if_false: Decision) -> Self {
        Decision::Branch {
            condition: String::from(condition),
            if_true: Box::new(if_true),
            if_false: Box::new(if_false),
        }
    }
    
    fn evaluate(&self, context: &std::collections::HashMap<String, bool>) -> &str {
        match self {
            Decision::Action(action) => action,
            Decision::Branch { condition, if_true, if_false } => {
                let result = context.get(condition).copied().unwrap_or(false);
                if result {
                    if_true.evaluate(context)
                } else {
                    if_false.evaluate(context)
                }
            }
        }
    }
}

fn main() {
    use std::collections::HashMap;
    
    // Дерево:
    //   battery_low?
    //   ├── true → RETURN_TO_BASE
    //   └── false → target_nearby?
    //              ├── true → FOLLOW_TARGET
    //              └── false → PATROL
    
    let tree = Decision::branch("battery_low",
        Decision::action("RETURN_TO_BASE"),
        Decision::branch("target_nearby",
            Decision::action("FOLLOW_TARGET"),
            Decision::action("PATROL"),
        ),
    );
    
    let mut ctx = HashMap::new();
    ctx.insert(String::from("battery_low"), false);
    ctx.insert(String::from("target_nearby"), true);
    
    println!("Рішення: {}", tree.evaluate(&ctx));
    // FOLLOW_TARGET
    
    ctx.insert(String::from("battery_low"), true);
    println!("Рішення: {}", tree.evaluate(&ctx));
    // RETURN_TO_BASE
}
```

`Decision::Branch` містить `Box<Decision>` — вказівник на піддерево. Без Box тип мав би нескінченний розмір. `evaluate` рекурсивно обходить дерево: на Branch — перевіряє умову та спускається у відповідну гілку. На Action — повертає дію. Рекурсія безпечна, бо глибина дерева обмежена його конструкцією (не нескінченна).

Метод `branch` — конструктор, що ховає Box::new від користувача. Замість `Decision::Branch { condition: ..., if_true: Box::new(...), if_false: Box::new(...) }` — зручний `Decision::branch("condition", left, right)`.

## 32.5. Box та ownership

Box володіє даними на купі. Коли Box переміщується — переміщується лише вказівник (8 байт), дані на купі залишаються на місці. Коли Box знищується — дані на купі звільняються через Drop:

```rust
fn main() {
    let b1 = Box::new(String::from("hello"));
    println!("b1: {}", b1);
    
    let b2 = b1; // переміщення: вказівник копіюється, b1 невалідний
    // println!("{}", b1); // ПОМИЛКА: value used after move
    println!("b2: {}", b2);
    
} // b2 виходить зі scope → String на купі звільняється
```

Box<T> реалізує ті самі ownership правила, що й будь-який інший тип: move при присвоєнні та передачі у функцію, clone для явного копіювання (якщо T: Clone), посилання для позичання.

```rust
fn process(data: Box<Vec<i32>>) {
    println!("Обробка {} елементів", data.len());
} // data (і Vec на купі) знищується тут

fn process_ref(data: &Box<Vec<i32>>) {
    println!("Читання {} елементів", data.len());
} // data не знищується — лише позичання

fn main() {
    let data = Box::new(vec![1, 2, 3, 4, 5]);
    
    process_ref(&data); // позичання — data доступна після
    process(data);      // переміщення — data недоступна після
    // process_ref(&data); // ПОМИЛКА: data переміщена
}
```

На практиці функції рідко приймають `&Box<T>` — зазвичай `&T` або `&[T]`, бо deref coercion конвертує автоматично. `&Box<Vec<i32>>` → `&Vec<i32>` → `&[i32]`. Приймайте найменш обмежувальний тип.

## Troubleshooting

### Помилка 1: `recursive type has infinite size`

```rust
enum Tree {
    Node(i32, Tree, Tree), // ПОМИЛКА
    Leaf,
}
```

Виправлення: `Node(i32, Box<Tree>, Box<Tree>)`.

### Помилка 2: `cannot move out of dereference of Box`

Спроба витягти значення з Box без деструктуризації:

```rust
let b = Box::new(String::from("hello"));
let s: String = *b; // Працює! Box<T> підтримує move через *
```

Це насправді працює для Box — на відміну від звичайних посилань, Box дозволяє `*b` для переміщення. Помилка виникає, якщо Box позичений:

```rust
fn take(b: &Box<String>) {
    let s: String = **b; // ПОМИЛКА: cannot move out of borrowed content
}
```

Виправлення: clone або працювати з посиланням.

### Помилка 3: `Box<dyn Trait>` без потрібних bounds

```rust
fn clone_boxed(b: &Box<dyn Agent>) -> Box<dyn Agent> {
    b.clone() // ПОМИЛКА: dyn Agent не реалізує Clone
}
```

Clone не object safe. Виправлення: патерн clone_box (Розділ 28.5) або перепроєктувати без клонування.

## Додатково

### Box vs інші smart pointers

Box — найпростіший: один власник, автоматичне звільнення. Rc<T> (Розділ 33) — кілька власників через reference counting (однопотоковий). Arc<T> (Розділ 37) — Rc для багатопотоковості. RefCell<T> (Розділ 33) — внутрішня мутабельність з runtime перевіркою.

Правило вибору: якщо один власник і потрібна купа — Box. Якщо кілька власників в одному потоці — Rc. Якщо кілька потоків — Arc.

### Box та продуктивність

Box::new виділяє пам'ять на купі — це виклик алокатора, десятки наносекунд. Для одиночних виділень це непомітно. Для мільйонів (наприклад, кожен вузол великого дерева) — може бути помітним. Якщо дерево будується один раз і обходиться багато разів — Box ідеальний. Якщо вузли створюються та знищуються мільйони разів на секунду — розгляньте arena allocation (бібліотека `typed-arena`).

## Контрольні запитання

### Рівень 1

1. Що зберігається на стеку, а що на купі для `Box::new(42)`?

Відповідь: на стеку — вказівник (8 байт). На купі — значення 42 (4 байти для i32).

2. Навіщо Box для рекурсивних типів?

Відповідь: без Box рекурсивний тип має нескінченний розмір (тип містить себе, який містить себе...). Box замінює вкладене значення на вказівник фіксованого розміру (8 байт), розриваючи нескінченність.

### Рівень 2

3. Що таке deref coercion і як це пов'язано з Box?

Відповідь: deref coercion — автоматичне розіменування компілятором. `&Box<String>` → `&String` → `&str`. Це дозволяє передавати `&Box<T>` де очікується `&T`, і викликати методи T на Box<T> без явного `*`.

4. Чим Box<T> відрізняється від &T?

Відповідь: Box<T> володіє даними на купі — при знищенні Box дані звільняються. &T — позичання існуючих даних без володіння. Box можна повернути з функції (owned), &T потребує живого джерела (lifetime).

### Рівень 3

5. Напишіть рекурсивний enum `Expr` для арифметичних виразів з методом `eval`.

Відповідь:

```rust
enum Expr {
    Num(f64),
    Add(Box<Expr>, Box<Expr>),
    Mul(Box<Expr>, Box<Expr>),
}

impl Expr {
    fn eval(&self) -> f64 {
        match self {
            Expr::Num(n) => *n,
            Expr::Add(a, b) => a.eval() + b.eval(),
            Expr::Mul(a, b) => a.eval() * b.eval(),
        }
    }
}
```

### Рівень 4

6. Чому `Box<dyn Trait>` має розмір 16 байт, а `Box<T>` — 8 байт?

Відповідь: `Box<T>` — один вказівник на дані (компілятор знає тип, знає методи). `Box<dyn Trait>` — fat pointer: вказівник на дані (8 байт) + вказівник на vtable (8 байт). Vtable потрібна для dynamic dispatch — знайти конкретну реалізацію методу у runtime.

## Резюме

Box\<T\> — smart pointer, що володіє значенням T на купі. На стеку — лише вказівник (8 байт). При знищенні Box — дані на купі звільняються автоматично (RAII).

Три сценарії: рекурсивні типи (дерево, список — Box розриває нескінченний розмір), великі дані (уникнення переповнення стеку), trait objects (`Box<dyn Trait>` — динамічний поліморфізм).

Deref trait дозволяє Box поводитись як посилання. Deref coercion: `&Box<T>` → `&T` автоматично. Методи T викликаються на Box<T> без явного `*`.

Box<T> має ті самі ownership правила: move при присвоєнні, clone для копіювання (якщо T: Clone), & для позичання. Функції зазвичай приймають `&T`, а не `&Box<T>`.

## Що далі

Box — один власник. Але деякі структури потребують кількох власників: граф, де кілька вузлів вказують на один спільний вузол. Або кілька компонентів агента, що ділять доступ до спільної конфігурації. Для цього потрібен Rc\<T\> — reference-counted pointer, що дозволяє кільком "власникам" ділити одне значення. А RefCell\<T\> додає внутрішню мутабельність — можливість змінювати дані через іммутабельне посилання, з перевіркою правил у runtime. Це тема Розділу 33.

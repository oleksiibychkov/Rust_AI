# Розділ 39. Async/Await: концепції

## Анотація

У Розділі 36 ви використовували потоки для паралелізму: `thread::spawn` створює OS-потік, кожен виконує код незалежно. Це працює для CPU-bound задач (обчислення маршруту, обробка зображень). Але для IO-bound задач (очікування відповіді сенсора, мережевий запит, читання файлу) потоки неефективні: потік більшу частину часу просто чекає, займаючи мегабайти пам'яті. Async/await — альтернативна модель конкурентності, де задачі (tasks) обслуговуються кількома потоками, і коли задача чекає IO — потік переключається на іншу задачу. Одна програма може мати тисячі async tasks на кількох потоках — без мегабайтних стеків, без дорогих context switch ОС.

---

Після опрацювання цього розділу студент зможе:

6. Обрати між threads та async для конкретної задачі.

## Ключові терміни

**Async (асинхронний)** — модель виконання, де задача може "призупинитись" при очікуванні (IO, таймер) і дозволити іншій задачі використати потік.

**Future** — trait, що представляє значення, яке ще не готове. Аналог "обіцянки" (Promise) з JavaScript. Future "опитується" (polled) runtime-ом до завершення.

**await** — оператор, що призупиняє виконання async-функції до завершення Future. Під час очікування потік може виконувати інші задачі.

**Runtime (середовище виконання)** — бібліотека, що керує виконанням async tasks. У Rust — Tokio, async-std. Стандартна бібліотека надає лише trait Future, але не runtime.

**Task (задача)** — легковагий аналог потоку. Тисячі tasks обслуговуються кількома OS-потоками.

**State machine (автомат станів)** — структура, у яку компілятор трансформує async fn. Кожен .await — точка, де функція може призупинитись і відновитись.

## Мотиваційний кейс

БПЛА-координатор обслуговує 100 агентів одночасно. Кожен агент кожні 100мс надсилає телеметрію по мережі. Координатор повинен: прийняти дані від кожного агента, обробити, оновити карту, відправити команди. З потоками: 100 потоків, кожен читає з мережі. 99% часу кожен потік чекає наступний пакет — 100 потоків * 4 МБ стеку = 400 МБ на очікування. З async: 100 tasks, 4 потоки. Коли task-агент-#42 чекає пакет — потік обслуговує task-агент-#17, якому пакет уже прийшов. 100 tasks * 1 КБ overhead = 100 КБ. Різниця — 4000 разів у пам'яті.

## 39.1. Sync vs Async: дві моделі виконання

У синхронному коді кожна операція блокує потік до завершення:

```rust
// Синхронний код (блокуючий)
fn fetch_sensor_data() -> Vec<f64> {
    let data = read_from_network();    // потік чекає 50мс
    let processed = process(data);      // потік обчислює 1мс
    let confirmed = send_confirmation(); // потік чекає 20мс
    processed
}
```

Потік витрачає 71мс, з яких 70мс — очікування (IO). Лише 1мс — корисна робота. Ефективність: 1.4%. Якщо 100 агентів — 100 потоків, кожен ефективний на 1.4%. Решта 98.6% часу кожного потоку — чекання. А кожен потік — мегабайти пам'яті.

В асинхронному коді потік не чекає — він переключається на іншу задачу:

```rust
// Асинхронний код (неблокуючий)
async fn fetch_sensor_data() -> Vec<f64> {
    let data = read_from_network().await;    // task призупиняється, потік бере іншу task
    let processed = process(data);            // потік обчислює
    let confirmed = send_confirmation().await; // знову призупиняється
    processed
}
```

`.await` — точка призупинення. Коли `read_from_network()` ще не завершена — task "засинає", і потік бере іншу task, що готова до виконання. Коли мережа відповість — task "прокидається" і продовжує з того місця, де зупинилась. Один потік обслуговує десятки tasks, переключаючись між ними у точках await.

Аналогія: ресторан. Синхронний підхід — один офіціант на одного клієнта. Поки клієнт обирає страву (чекає) — офіціант стоїть і чекає з ним. 10 клієнтів = 10 офіціантів. Асинхронний — один офіціант на 10 клієнтів. Прийняв замовлення у клієнта #1, поки кухня готує — йде до клієнта #2. Коли страва #1 готова — несе її. Один офіціант обслуговує всіх, бо більшість часу — очікування.

## 39.2. Future trait: обіцянка результату

`async fn` повертає не результат одразу, а Future — "обіцянку" результату. Future — це trait:

```rust
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

enum Poll<T> {
    Ready(T),    // результат готовий
    Pending,     // ще не готовий, розбудіть мене пізніше
}
```

Runtime (Tokio) періодично "опитує" (poll) кожен Future: "ти готовий?". Якщо Ready(значення) — task завершена, результат повертається. Якщо Pending — task спить, runtime перевіряє інші tasks.

Вам не потрібно реалізовувати Future вручну — async fn робить це автоматично. Але розуміння механізму допомагає дебажити та обирати правильні патерни.

### Ледачі futures: async fn без await нічого не робить

Це фундаментальна властивість, що часто дивує початківців:

```rust
async fn compute() -> i32 {
    println!("Обчислюю...");
    42
}

fn main() {
    let future = compute(); // НЕ виконується! Лише створює Future
    // "Обчислюю..." НЕ виведеться
    // future потрібно передати runtime-у або await-ити
}
```

Виклик `compute()` не виконує тіло функції. Він лише створює Future — об'єкт, що описує "що потрібно зробити". Виконання почнеться лише коли Future буде polled (через runtime або .await). Це lazy evaluation — та сама ідея, що й ледачі ітератори з Розділу 23.

Це відрізняє Rust від JavaScript, де `async function compute()` починає виконуватись одразу при виклику (eager). У Rust — lazy, і це має переваги: Future можна створити, передати, зберегти, скомбінувати — і виконати лише коли потрібно.

## 39.3. Як async fn стає state machine

Компілятор трансформує async fn у автомат станів (state machine). Кожен .await — точка, де стан зберігається і виконання може призупинитись.

Для інтуїції, async fn:

```rust
async fn mission_step(agent_id: &str) -> String {
    let sensor_data = read_sensor().await;         // await #1
    let analysis = analyze(sensor_data);
    let confirmation = send_report(analysis).await; // await #2
    format!("{}: {}", agent_id, confirmation)
}
```

Концептуально трансформується приблизно у:

```rust
enum MissionStepState {
    Start { agent_id: String },
    AfterSensor { agent_id: String, sensor_future: SensorFuture },
    AfterReport { agent_id: String, report_future: ReportFuture },
    Complete,
}
```

Три стани: Start (до першого await), AfterSensor (між першим та другим await), AfterReport (після другого await). Кожен стан зберігає локальні змінні, що потрібні для продовження. Runtime опитує цей автомат: якщо SensorFuture повертає Pending — автомат залишається у стані AfterSensor. Якщо Ready — переходить до наступного стану.

Ключова деталь: весь стан async fn зберігається у структурі (автоматі), а не на стеку потоку. Тому async task займає стільки пам'яті, скільки потрібно для її стану — зазвичай сотні байтів. OS-потік завжди тримає мегабайти стеку, навіть якщо використовує кілобайти.

Вам не потрібно знати деталі трансформації для написання async коду. Але розуміння допомагає: "async fn — це структура зі станами, .await — перехід між станами, runtime — диспетчер, що опитує стани".

## 39.4. Pin: чому і коли (інтуїтивно)

Автомат станів async fn може містити самопосилання: локальна змінна, що є посиланням на іншу локальну змінну того ж автомата. Якщо автомат переміститься у пам'яті (move) — посилання стане dangling. `Pin` запобігає цьому: "закріплює" значення у пам'яті, гарантуючи, що воно не буде переміщене.

На практиці вам рідко потрібно працювати з Pin напряму. Tokio та async/await ховають його за зручним API. Але ви побачите Pin у сигнатурах (наприклад, Future::poll) та у помилках компіляції. Знання, що Pin = "значення не може бути переміщене" — достатньо для 95% випадків.

## 39.5. Threads vs Async: коли що обирати

Обидва механізми дають конкурентність. Вибір залежить від характеру задачі:

| Критерій | OS Threads | Async Tasks |
|----------|-----------|-------------|
| Overhead на задачу | ~4 МБ стеку | ~кілобайти |
| Кількість задач | Сотні | Десятки тисяч |
| Переключення | Мікросекунди (OS) | Наносекунди (userspace) |
| Блокування IO | Потік чекає | Task засинає, потік бере іншу |
| CPU-bound задачі | Ефективно (кожен потік = ядро) | Неефективно (блокує runtime) |
| IO-bound задачі | Марнотратство (потік чекає) | Ефективно (потік не чекає) |
| Складність коду | Простіший | Складніший (async/await, Pin) |
| Екосистема | std (стандартна) | Tokio/async-std (зовнішня) |

Правило: CPU-bound (обчислення, шифрування, обробка зображень) — threads або Rayon (Розділ 43). IO-bound (мережа, файли, сенсори, бази даних) — async. Змішані — async для IO, spawn_blocking або Rayon для CPU-heavy частин.

Для БПЛА-рою: координатор обробляє мережеві повідомлення від 100 агентів — IO-bound → async. Обчислення маршруту для кожного агента — CPU-bound → thread/Rayon. Типова архітектура: async для комунікації та координації, потоки для обчислень.

## 39.6. Перший async код (preview Tokio)

Стандартна бібліотека Rust надає trait Future, але не runtime для його виконання. Потрібен зовнішній runtime — Tokio (Розділ 40 детально). Короткий preview:

```rust
// Cargo.toml:
// [dependencies]
// tokio = { version = "1", features = ["full"] }

use tokio::time::{sleep, Duration};

async fn agent_step(id: &str) {
    println!("[{}] Починаю крок", id);
    sleep(Duration::from_millis(100)).await; // неблокуюче очікування
    println!("[{}] Крок завершено", id);
}

#[tokio::main]
async fn main() {
    // Три задачі конкурентно
    let t1 = tokio::spawn(agent_step("SCOUT-01".to_string()));
    let t2 = tokio::spawn(agent_step("SCOUT-02".to_string()));
    let t3 = tokio::spawn(agent_step("RECON-01".to_string()));
    
    // Чекаємо всіх
    t1.await.unwrap();
    t2.await.unwrap();
    t3.await.unwrap();
}

async fn agent_step(id: String) {
    println!("[{}] Починаю крок", id);
    sleep(Duration::from_millis(100)).await;
    println!("[{}] Крок завершено", id);
}
```

Три tasks запущені "одночасно". Під час `sleep(...).await` кожна task засинає, і runtime обслуговує інші. Загальний час — ~100мс (паралельне очікування), а не 300мс (послідовне). І це на одному потоці (current_thread runtime) або кількох (multi_thread).

## Troubleshooting

### Помилка 1: `async fn without runtime`

```rust
async fn hello() { println!("hello"); }
fn main() {
    hello(); // WARNING: unused future — нічого не виконається
}
```

Виправлення: потрібен runtime. `#[tokio::main] async fn main() { hello().await; }`.

### Помилка 2: `future cannot be sent between threads safely`

Async task містить не-Send тип (наприклад, Rc):

```rust
let data = Rc::new(42);
tokio::spawn(async move { println!("{}", data); }); // ПОМИЛКА: Rc не Send
```

Виправлення: замінити Rc на Arc.

### Помилка 3: блокуючий код в async

```rust
async fn bad() {
    std::thread::sleep(Duration::from_secs(5)); // БЛОКУЄ потік runtime!
    // Інші tasks не виконуються 5 секунд
}
```

Виправлення: `tokio::time::sleep(Duration::from_secs(5)).await` — неблокуюче очікування.

### Помилка 4: `the trait Future is not implemented for ()`

Забули `async` перед fn або .await після виклику async fn.

## Додатково

### Порівняння async-моделей у мовах

JavaScript: async/await з event loop (один потік). Eager execution — async fn починає виконуватись одразу.

Go: goroutines — легковагі потоки з runtime у мові. Простіший синтаксис (go func()), але менше контролю. Garbage collector.

Python: asyncio з event loop (один потік). Eager, як JavaScript. GIL обмежує паралелізм.

Rust: async/await з зовнішнім runtime (Tokio). Lazy futures — не виконуються без await/spawn. Zero-cost abstractions — async трансформується у state machine при компіляції, без heap allocation для кожної задачі. Максимальний контроль та продуктивність, але складніший синтаксис та lifetime правила.

### Async та ownership

Async fn у Rust підлягає тим самим правилам ownership та borrowing. Але є нюанс: async fn може "тримати" посилання через .await — це ускладнює lifetime-аналіз. Якщо async fn приймає &str і містить .await — посилання повинно жити до завершення Future, що може бути довільно довго. Тому async fn часто приймають owned типи (String замість &str) для уникнення lifetime-проблем. Це trade-off: менше ефективності (clone), більше простоти.

## Контрольні запитання

### Рівень 1

1. Чим async відрізняється від sync виконання?

Відповідь: sync — потік блокується при очікуванні (IO, sleep) і не робить нічого корисного. Async — task призупиняється, потік переключається на іншу task. Потік завжди працює.

2. Що означає "lazy futures"?

Відповідь: виклик async fn не виконує код — лише створює Future (опис задачі). Виконання почнеться лише коли Future буде polled через runtime або .await.

### Рівень 2

3. Чому async tasks ефективніші за OS-потоки для IO-bound задач?

Відповідь: OS-потік тримає ~4 МБ стеку, навіть чекаючи IO. Async task — кілобайти (лише стан state machine). Переключення потоків — мікросекунди (OS context switch). Переключення tasks — наносекунди (userspace, без ОС). 1000 IO-bound задач: 1000 потоків = 4 ГБ, 1000 tasks = кілька МБ.

4. Коли threads кращі за async?

Відповідь: CPU-bound задачі — обчислення, шифрування, обробка зображень. Async task, що виконує тривале обчислення без .await — блокує потік runtime, і інші tasks не виконуються. Threads кожен мають своє ядро і не блокують один одного.

### Рівень 3

5. Поясніть, чому `std::thread::sleep` всередині async fn — помилка.

Відповідь: `std::thread::sleep` блокує OS-потік. В async runtime один потік обслуговує багато tasks. Блокування потоку на 5 секунд означає: всі tasks на цьому потоці зупинені на 5с. `tokio::time::sleep(...).await` — неблокуючий: task засинає, потік бере іншу task, через 5с task прокидається.

### Рівень 4

6. Порівняйте async моделі Rust, Go та JavaScript.

Відповідь: JavaScript — однопотоковий event loop, eager futures, немає CPU-паралелізму. Go — goroutines з runtime у мові, простий синтаксис, GC overhead. Rust — lazy futures, зовнішній runtime (Tokio), zero-cost state machine, максимальна продуктивність, складніший через ownership та Pin. Rust найефективніший, Go найпростіший, JavaScript найобмеженіший.

## Резюме

Async/await — модель конкурентності, де tasks призупиняються при IO і потік переключається на іншу task. Ефективніше за потоки для IO-bound задач: тисячі tasks на кількох потоках замість тисяч потоків.

`async fn` повертає Future — ледачу "обіцянку" результату. Future не виконується без .await або runtime. Компілятор трансформує async fn у state machine: кожен .await — точка призупинення.

Threads — для CPU-bound. Async — для IO-bound. Комбінація — для реальних систем.

Rust async: lazy futures (на відміну від JS/Python), zero-cost (state machine при компіляції), зовнішній runtime (Tokio). Pin запобігає переміщенню самопосилальних state machines.

## Що далі

Концепції зрозумілі — тепер потрібен інструмент. Розділ 40 присвячений Tokio — найпопулярнішому async runtime для Rust. Ви навчитесь використовувати `#[tokio::main]`, `tokio::spawn` для запуску tasks, `tokio::select!` для одночасного очікування кількох futures, та `tokio::time` для таймерів та таймаутів. Агент стане async: періодична перевірка сенсорів, обробка команд та логування — все конкурентно в одній async fn.

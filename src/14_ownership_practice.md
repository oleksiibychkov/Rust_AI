# Розділ 14. Ownership: практика помилок компілятора

## Анотація

У Розділі 13 ви вивчили три правила ownership та зрозуміли різницю між move та copy на теоретичному рівні. Тепер час перетворити це розуміння у навичку. Цей розділ цілком присвячений практиці: ви будете свідомо створювати код, що порушує правила ownership, читати повідомлення компілятора, аналізувати причини, та знаходити виправлення. Це не "зробити помилку випадково і шукати, що не так". Це цілеспрямований тренінг: ви наперед знаєте, яку помилку створите, бо розумієте правила. Мета — навчитись читати повідомлення Rust-компілятора так само вільно, як ви читаєте текст. Для БПЛА-агента це означає: кожна помилка компілятора — це потенційний баг, який Rust зловив до запуску. Краще витратити хвилину на виправлення, ніж годину на пошук runtime-крашу.

## Ключові терміни

**E0382** — код помилки "borrow of moved value" (використання після переміщення).

**E0505** — код помилки "cannot move out of borrowed content" (спроба забрати позичене).

**E0382 in loop** — використання переміщеного значення у наступній ітерації циклу.

**Compiler diagnostic** — повідомлення компілятора з вказівниками на рядки коду та рекомендаціями.

**rustc --explain** — команда для отримання розширеного пояснення коду помилки.

## Мотиваційний кейс

У 2019 році дослідники Microsoft проаналізували всі вразливості безпеки за 12 років (CVE) і виявили, що приблизно 70% з них — це помилки пам'яті: use-after-free, buffer overflow, double free. Це саме ті помилки, які Rust-компілятор ловить через ownership та borrowing. Кожне повідомлення "borrow of moved value" — це потенційна вразливість, що ніколи не потрапить у ваш код. Навчитись читати ці повідомлення — означає навчитись мови, якою компілятор розмовляє з вами про безпеку.

Google провів аналогічне дослідження для Android і отримав схожі результати: більшість критичних вразливостей — помилки пам'яті у C/C++ коді. Після впровадження Rust для нових компонентів кількість вразливостей пам'яті почала знижуватись. Для БПЛА ставки ще вищі: use-after-free у навігаційній системі може призвести не до витоку даних, а до фізичної аварії.

Тому цей розділ — не "факультативна практика". Це тренування навички, від якої залежить безпека вашого коду. Кожна помилка компілятора, яку ви навчитесь розуміти — це одна вразливість менше у продукті.

## 14.1. Анатомія повідомлення компілятора

Перш ніж створювати помилки, навчимось детально читати повідомлення. Rust-компілятор має одні з найкращих діагностичних повідомлень серед усіх мов програмування. Кожне повідомлення — це міні-урок, що пояснює не лише що пішло не так, а й чому та як виправити.

Це різко відрізняється від досвіду в C або C++, де повідомлення компілятора часто криптичні і не вказують на справжню причину проблеми. У Rust компілятор — ваш союзник. Він витрачає значні обчислювальні ресурси на те, щоб пояснити помилку максимально зрозуміло. Ваша задача — навчитись читати ці пояснення.

Повідомлення Rust-компілятора мають кольорове виділення в терміналі: червоним позначено помилку, синім — місце в коді, зеленим — підказки. Навіть без кольорів структура повідомлення дозволяє швидко знайти потрібну інформацію.

### Структура повідомлення

Розглянемо повідомлення для найтиповішої ownership-помилки:

```rust
fn main() {
    let name = String::from("БПЛА-07");
    let backup = name;
    println!("Агент: {}", name);
}
```

Компілятор видає:

```
error[E0382]: borrow of moved value: `name`
 --> src/main.rs:4:28
  |
2 |     let name = String::from("БПЛА-07");
  |         ---- move occurs because `name` has type `String`,
  |              which does not implement the `Copy` trait
3 |     let backup = name;
  |                  ---- value moved here
4 |     println!("Агент: {}", name);
  |                            ^^^^ value borrowed here after move
  |
help: consider cloning the value if the performance cost is acceptable
  |
3 |     let backup = name.clone();
  |                      ++++++++
```

Повідомлення має шість компонентів:

Рівень та код помилки: `error[E0382]`. Слово `error` означає — код не скомпілюється (на відміну від `warning`, який дозволяє компіляцію). `E0382` — унікальний код, за яким можна знайти розширене пояснення командою `rustc --explain E0382`.

Опис: `borrow of moved value: name`. Стислий опис проблеми: ви намагаєтесь позичити (використати) значення `name`, яке вже було переміщене.

Розташування: `--> src/main.rs:4:28`. Файл, рядок, стовпець, де виявлено проблему.

Вказівники на рядки: компілятор виділяє три ключових рядки коду. Рядок 2 — де створено значення і чому відбувся move (тип String не реалізує Copy). Рядок 3 — де значення переміщено (`value moved here`). Рядок 4 — де ви намагаєтесь його використати (`value borrowed here after move`).

Пояснення причини: `which does not implement the Copy trait`. Компілятор пояснює: move відбувся саме тому, що String не є Copy-типом.

Рекомендація: `help: consider cloning the value`. Компілятор навіть показує конкретну зміну у коді: додати `.clone()`.

### Команда rustc --explain

Для будь-якого коду помилки можна отримати розширене пояснення:

```bash
rustc --explain E0382
```

Ця команда виведе кілька екранів тексту з поясненням помилки, прикладами коду що компілюється та що не компілюється, та рекомендаціями. Це безцінний ресурс, коли повідомлення компілятора недостатньо зрозуміле.

## 14.2. Помилка 1: використання після move (E0382)

Це найчастіша ownership-помилка. Ви переміщуєте значення і потім намагаєтесь його використати.

### Випадок A: move при присвоєнні

Постановка: створюємо String, присвоюємо іншій змінній, намагаємось використати оригінал.

```rust
fn main() {
    let mission = String::from("розвідка периметра");
    let archived = mission;  // Move!
    
    println!("Поточна місія: {}", mission);     // ПОМИЛКА
    println!("Архівована: {}", archived);         // OK
}
```

Помилка: `borrow of moved value: mission`. Рядок 3 — `value moved here`. Рядок 5 — `value borrowed here after move`.

Чому виникає: `let archived = mission` переміщує володіння String з `mission` до `archived`. Після цього `mission` — лише "порожня оболонка", що не вказує на жодні дані. Спроба використати її — помилка.

Виправлення 1 — clone:

```rust
let archived = mission.clone();  // глибока копія
println!("Поточна місія: {}", mission);   // OK — mission не переміщена
println!("Архівована: {}", archived);       // OK — незалежна копія
```

Виправлення 2 — змінити порядок використання:

```rust
println!("Поточна місія: {}", mission);   // використовуємо ДО move
let archived = mission;                    // тепер move
println!("Архівована: {}", archived);       // OK
// mission тепер недійсна, але нам вона вже не потрібна
```

Виправлення 3 — посилання (preview Розділу 15):

```rust
let archived = &mission;  // позичаємо, не переміщуємо
println!("Поточна місія: {}", mission);
println!("Архівована: {}", archived);
```

Порівняння: clone створює копію (дорого для великих даних). Зміна порядку — безкоштовно, але працює не завжди. Посилання — найкращий варіант, якщо не потрібна незалежна копія.

### Випадок B: move при передачі у функцію

Постановка: передаємо String у функцію, потім намагаємось використати у caller.

```rust
fn log_mission(plan: String) {
    println!("[LOG] Місія: {}", plan);
}

fn main() {
    let plan = String::from("патрулювання зони Б");
    log_mission(plan);  // Move у функцію!
    
    println!("Виконуємо: {}", plan);  // ПОМИЛКА
}
```

Помилка: `borrow of moved value: plan`. Рядок 7 — `value moved here`. Рядок 9 — `value borrowed here after move`.

Чому виникає: `log_mission(plan)` переміщує String у параметр `plan` функції. Після повернення з функції String вже знищена (drop у кінці `log_mission`).

Виправлення 1 — clone:

```rust
log_mission(plan.clone());
println!("Виконуємо: {}", plan);  // OK
```

Виправлення 2 — змінити функцію на &str:

```rust
fn log_mission(plan: &str) {  // позичає, не забирає
    println!("[LOG] Місія: {}", plan);
}

fn main() {
    let plan = String::from("патрулювання зони Б");
    log_mission(&plan);  // позичаємо
    println!("Виконуємо: {}", plan);  // OK — plan не переміщений
}
```

Виправлення 3 — повернути String назад:

```rust
fn log_mission(plan: String) -> String {
    println!("[LOG] Місія: {}", plan);
    plan  // повертаємо назад
}

fn main() {
    let plan = String::from("патрулювання зони Б");
    let plan = log_mission(plan);  // shadowing: отримуємо назад
    println!("Виконуємо: {}", plan);
}
```

Порівняння: варіант з `&str` — найкращий (без копіювання, без передачі туди-сюди). Повернення назад — працює, але громіздко. Clone — найпростіший, але найдорожчий.

## 14.3. Помилка 2: move у циклі

Ця помилка підступніша — код виглядає правильно, але ламається на другій ітерації.

### Випадок: передача String у функцію всередині loop

```rust
fn send_alert(msg: String) {
    println!("[ALERT] {}", msg);
}

fn main() {
    let warning = String::from("Низький заряд батареї!");
    
    for i in 0..3 {
        println!("Спроба {}", i + 1);
        send_alert(warning);  // Move на першій ітерації!
    }
}
```

Помилка:

```
error[E0382]: use of moved value: `warning`
 --> src/main.rs:9:20
  |
6 |     let warning = String::from("Низький заряд батареї!");
  |         ------- move occurs because `warning` has type `String`
...
9 |         send_alert(warning);
  |                    ^^^^^^^ value moved here, in previous iteration of loop
```

Зверніть увагу на фразу `in previous iteration of loop` — компілятор розуміє, що move відбувся в попередній ітерації, і на поточній значення вже недійсне.

Чому виникає: на першій ітерації `warning` переміщується у `send_alert`. Коли цикл переходить до другої ітерації, `warning` вже "спорожнена". Але `for` намагається передати її знову — помилка.

Виправлення 1 — clone всередині циклу:

```rust
for i in 0..3 {
    println!("Спроба {}", i + 1);
    send_alert(warning.clone());  // кожна ітерація отримує копію
}
println!("Оригінал: {}", warning);  // OK — warning не переміщена
```

Виправлення 2 (краще) — змінити функцію на &str:

```rust
fn send_alert(msg: &str) {
    println!("[ALERT] {}", msg);
}

fn main() {
    let warning = String::from("Низький заряд батареї!");
    for i in 0..3 {
        println!("Спроба {}", i + 1);
        send_alert(&warning);  // позичаємо на кожній ітерації
    }
}
```

Виправлення 3 — створити String всередині циклу:

```rust
for i in 0..3 {
    let warning = String::from("Низький заряд батареї!");
    send_alert(warning);  // OK: нова String кожну ітерацію
}
```

Порівняння: варіант з `&str` — найкращий. Створення всередині циклу — марнотратство (нова алокація кожної ітерації). Clone — прийнятно, якщо функцію змінити не можна.

## 14.4. Помилка 3: move з контейнера

Ця помилка виникає при спробі "витягнути" не-Copy елемент з вектора або масиву.

```rust
fn main() {
    let drones = vec![
        String::from("Альфа"),
        String::from("Бета"),
        String::from("Гамма"),
    ];
    
    let leader = drones[0];  // ПОМИЛКА: move з вектора
    println!("Лідер: {}", leader);
}
```

Помилка:

```
error[E0507]: cannot move out of index of `Vec<String>`
 --> src/main.rs:8:18
  |
8 |     let leader = drones[0];
  |                  ^^^^^^^^^ move occurs because value has type `String`
  |
help: consider borrowing here
  |
8 |     let leader = &drones[0];
  |                  +
```

Чому виникає: вектор володіє своїми елементами. `drones[0]` намагається перемістити перший елемент назовні, залишаючи "дірку" у векторі. Rust це забороняє — вектор повинен бути цілісним.

Виправлення 1 — позичити:

```rust
let leader = &drones[0];  // позичаємо, не переміщуємо
println!("Лідер: {}", leader);
println!("Усі: {:?}", drones);  // вектор цілісний
```

Виправлення 2 — clone:

```rust
let leader = drones[0].clone();  // незалежна копія
println!("Лідер: {}", leader);
```

Виправлення 3 — використати remove (якщо дійсно потрібно витягнути):

```rust
let mut drones = vec![
    String::from("Альфа"),
    String::from("Бета"),
];
let leader = drones.remove(0);  // витягує і зсуває решту
// drones тепер містить лише "Бета"
```

## 14.5. Помилка 4: часткове переміщення зі структури

Ця помилка стане актуальною після Розділу 16 (Struct), але варто знати заздалегідь:

```rust
struct Agent {
    id: String,
    battery: u8,
}

fn main() {
    let agent = Agent {
        id: String::from("БПЛА-07"),
        battery: 85,
    };
    
    let name = agent.id;  // Move поля id!
    println!("Батарея: {}", agent.battery);  // OK — u8 Copy
    println!("ID: {}", agent.id);            // ПОМИЛКА — id переміщено
}
```

Помилка: `borrow of partially moved value: agent`. Поле `id` переміщене, але `battery` (Copy) — ні.

Чому виникає: `let name = agent.id` переміщує лише поле `id` (String, не-Copy). Поле `battery` (u8, Copy) залишається доступним. Але доступ до `agent.id` або до `agent` як цілого — помилка, бо частина структури "зламана".

Виправлення: `let name = agent.id.clone();` або `let name = &agent.id;`

## 14.6. Практичний тренінг: "Зламай та полагодь"

Виконайте ці вправи без компілятора. Для кожного фрагмента: (1) визначте, чи скомпілюється, (2) якщо ні — яка помилка та на якому рядку, (3) запропонуйте два виправлення.

### Вправа 1

```rust
fn format_status(id: String, battery: u8) -> String {
    format!("{}: {}%", id, battery)
}

fn main() {
    let drone_id = String::from("SCOUT-01");
    let status1 = format_status(drone_id, 85);
    let status2 = format_status(drone_id, 70);
    println!("{}\n{}", status1, status2);
}
```

Відповідь: не скомпілюється. На рядку `let status2 = format_status(drone_id, 70)` — помилка E0382: `drone_id` переміщено у перший виклик `format_status`. Виправлення А: `format_status(drone_id.clone(), 85)`. Виправлення Б (краще): змінити параметр на `id: &str`.

### Вправа 2

```rust
fn main() {
    let coords = vec![(10.0, 20.0), (30.0, 40.0)];
    let first = coords[0];
    println!("{:?}", first);
    println!("{:?}", coords);
}
```

Відповідь: скомпілюється! Кортеж `(f64, f64)` є Copy (обидва елементи Copy). `coords[0]` копіює, а не переміщує. Обидва `println!` працюють.

### Вправа 3

```rust
fn main() {
    let names: Vec<String> = vec![
        String::from("Alpha"),
        String::from("Beta"),
    ];
    
    for name in names {
        println!("Агент: {}", name);
    }
    
    println!("Всього агентів: {}", names.len());
}
```

Відповідь: не скомпілюється. `for name in names` переміщує весь вектор (ітерація за значенням "з'їдає" вектор). Після циклу `names` недійсна. Виправлення А: `for name in &names` (ітерація за посиланнями). Виправлення Б: `for name in names.iter()`.

### Вправа 4

```rust
fn main() {
    let a = String::from("hello");
    let b = a;
    let c = b;
    let a = c;
    println!("{}", a);
}
```

Відповідь: скомпілюється! Ланцюжок move: a → b → c → a (через shadowing). Фінальне `a` — це нова змінна, що отримала володіння від `c`. Попередні `a`, `b`, `c` недійсні, але фінальне `a` дійсне.

### Вправа 5

```rust
fn double_or_nothing(s: String) -> String {
    if s.len() > 5 {
        return String::from("nothing");
    }
    s + &s
}
```

Відповідь: не скомпілюється. `s + &s` — оператор `+` для String переміщує ліву частину (`s`), а права (`&s`) вже позичає `s`, яку щойно переміщено. Помилка: `borrow of moved value: s`. Виправлення: `let result = format!("{}{}", s, s); result` або `let doubled = s.repeat(2); doubled`.

Цей приклад підступний: оператор `+` для String в Rust має сигнатуру `fn add(self, &str) -> String` — він забирає ліву частину за значенням (move). Тому `s + &s` спершу переміщує `s`, потім намагається позичити вже переміщену `s`.

### Вправа 6

```rust
fn select_leader(agents: Vec<String>) -> String {
    let leader = agents[0];
    leader
}

fn main() {
    let team = vec![String::from("Альфа"), String::from("Бета")];
    let boss = select_leader(team);
    println!("Лідер: {}", boss);
}
```

Відповідь: не скомпілюється. `agents[0]` намагається перемістити String з вектора — помилка E0507. Виправлення А: `agents[0].clone()`. Виправлення Б: `agents.into_iter().next().unwrap()` — "споживає" вектор через ітератор.

### Вправа 7: реалістичний сценарій

```rust
fn log_position(id: String, x: f64, y: f64) {
    println!("[LOG] {} at ({:.1}, {:.1})", id, x, y);
}

fn check_boundary(id: String, x: f64, y: f64) -> bool {
    x >= 0.0 && x <= 20.0 && y >= 0.0 && y <= 20.0
}

fn main() {
    let drone = String::from("SCOUT-01");
    log_position(drone, 15.0, 22.0);
    let safe = check_boundary(drone, 15.0, 22.0);
    println!("{} безпечний: {}", drone, safe);
}
```

Відповідь: не скомпілюється у двох місцях. `log_position(drone, ...)` переміщує `drone`. `check_boundary(drone, ...)` — E0382. `println!("{}", drone)` — теж E0382. Виправлення: змінити обидві функції на `id: &str` і викликати з `&drone`.

## 14.7. Clone vs Copy: практичне рішення

У Розділі 13 ви вивчили теоретичну різницю між Clone та Copy. Тепер — практичні рекомендації для щоденного коду.

### Коли clone виправданий

Clone виправданий, коли вам дійсно потрібні два незалежних екземпляри даних. Типовий випадок — зберегти оригінал і модифікувати копію:

```rust
fn main() {
    let original_route = String::from("База → Альфа → Бета → База");
    let mut modified_route = original_route.clone();
    modified_route.push_str(" → Гамма → База");
    
    println!("Стандартний: {}", original_route);
    println!("Розширений:  {}", modified_route);
}
```

Тут clone необхідний: ми хочемо зберегти `original_route` незмінним і створити модифіковану копію.

### Коли clone — антипатерн

Clone стає антипатерном, коли використовується лише для "заткнення" компілятора:

```rust
// Погано: clone лише щоб уникнути move
fn print_info(s: String) {
    println!("{}", s);
}

fn main() {
    let data = String::from("дані місії");
    print_info(data.clone());  // навіщо копіювати, якщо лише читаємо?
    print_info(data.clone());
    print_info(data);           // останній виклик без clone — OK
}
```

Правильне рішення — змінити `print_info` на `fn print_info(s: &str)`. Жодного копіювання, жодних витрат.

### Дерево прийняття рішення

Коли функція потребує доступу до не-Copy даних:

1. Функція лише читає? Передайте `&str` або `&T`. Ніколи не clone для читання.
2. Функція повинна стати власником (зберегти, передати далі)? Передайте за значенням (move).
3. Потрібні дві незалежні копії? Тоді clone виправданий.

## 14.8. Ownership у контексті БПЛА-агента

Тепер застосуємо все вивчене до коду, наближеного до реального агента. Кожен приклад ілюструє типову ситуацію, з якою ви зіткнетесь у Практикумі (Розділ 20).

### Ситуація 1: передача плану місії між модулями

Уявіть систему, де один модуль створює план місії, інший — валідує, третій — виконує. Кожен модуль — це функція:

```rust
fn create_plan(target: &str) -> String {
    format!("Маршрут: База → {} → База", target)
}

fn validate_plan(plan: String) -> String {
    if plan.is_empty() {
        panic!("Порожній план!");
    }
    println!("  [Валідація] План '{}' перевірено", plan);
    plan  // повертаємо назад — передаємо володіння caller
}

fn execute_plan(plan: String) {
    println!("  [Виконання] Запуск: '{}'", plan);
    // ... навігація, сканування, повернення ...
    println!("  [Виконання] Місію '{}' завершено", plan);
}  // plan Drop — план звільнено після виконання

fn archive_result(result: String) {
    println!("  [Архів] Збережено: '{}'", result);
}

fn main() {
    // Ланцюжок передачі володіння:
    let plan = create_plan("точка Альфа");
    let plan = validate_plan(plan);     // move → validate → move назад
    
    // Тут ми маємо вибір: план потрібен і для execute, і для archive.
    // Але execute "з'їсть" план. Рішення — clone або зміна архітектури.
    let result = format!("Результат: {}", plan);
    execute_plan(plan);                  // move — plan більше не доступний
    archive_result(result);              // result створений ДО move plan
}
```

Вивід:

```
  [Валідація] План 'Маршрут: База → точка Альфа → База' перевірено
  [Виконання] Запуск: 'Маршрут: База → точка Альфа → База'
  [Виконання] Місію 'Маршрут: База → точка Альфа → База' завершено
  [Архів] Збережено: 'Результат: Маршрут: База → точка Альфа → База'
```

Зверніть увагу на ключовий момент: ми створили `result` до виклику `execute_plan(plan)`. Якби ми спробували створити `result` після — `plan` вже була б недійсною. Порядок операцій у Rust має значення, бо move — незворотний.

Альтернативний підхід з clone:

```rust
fn main() {
    let plan = create_plan("точка Альфа");
    let plan = validate_plan(plan);
    
    execute_plan(plan.clone());  // передаємо копію
    archive_result(format!("Результат: {}", plan));  // оригінал доступний
}
```

Обидва підходи коректні. Перший — без clone, але потребує обережності з порядком. Другий — з clone, але простіший для розуміння. У реальному коді з посиланнями (Розділ 15) ця проблема зникне.

### Ситуація 2: збирання даних від кількох агентів

Часта задача — зібрати звіти від кількох агентів у єдиний список:

```rust
fn create_report(id: &str, battery: u8) -> String {
    format!("Агент {}: батарея {}%", id, battery)
}

fn compile_reports(reports: Vec<String>) -> String {
    let mut result = String::from("=== Зведений звіт ===\n");
    for report in &reports {  // &reports — позичаємо вектор!
        result.push_str(report);
        result.push('\n');
    }
    result.push_str(&format!("Всього агентів: {}", reports.len()));
    result
}

fn main() {
    let mut reports = Vec::new();
    
    // Кожен create_report повертає новий String → push додає у вектор
    reports.push(create_report("SCOUT-01", 85));
    reports.push(create_report("SCOUT-02", 62));
    reports.push(create_report("SCOUT-03", 91));
    
    let summary = compile_reports(reports);
    // reports переміщено у compile_reports!
    // println!("{}", reports.len());  // ПОМИЛКА
    
    println!("{}", summary);
}
```

Вивід:

```
=== Зведений звіт ===
Агент SCOUT-01: батарея 85%
Агент SCOUT-02: батарея 62%
Агент SCOUT-03: батарея 91%
Всього агентів: 3
```

Зверніть увагу: всередині `compile_reports` ми ітеруємо `for report in &reports` — це позичання кожного елемента. Якби ми написали `for report in reports` — цикл "з'їв" би вектор, і `reports.len()` в кінці функції не працювало б.

Але сам вектор `reports` переміщено у `compile_reports` з `main`. Після виклику `reports` у `main` недійсний. Якщо він потрібен після — передайте `&Vec<String>` або `&[String]`.

### Ситуація 3: ownership при формуванні команди

```rust
fn build_command(action: &str, x: f64, y: f64) -> String {
    format!("CMD:{}:({:.1},{:.1})", action, x, y)
}

fn send_command(cmd: String) -> bool {
    println!("  Надсилаю: {}", cmd);
    // ... логіка передачі ...
    true  // успішно
}

fn main() {
    let cmd = build_command("MOVE", 150.0, 200.0);
    println!("Створено команду: {}", cmd);
    
    let sent = send_command(cmd);  // Move!
    
    if sent {
        // Хочемо залогувати, але cmd переміщено!
        // println!("Лог: команда '{}' надіслана", cmd);  // ПОМИЛКА
        println!("Команда успішно надіслана");
    }
}
```

Тут типова проблема: ми хочемо і надіслати команду (move у `send_command`), і залогувати її текст. Три рішення:

Рішення А — зберегти текст до move:

```rust
let cmd = build_command("MOVE", 150.0, 200.0);
let cmd_text = cmd.clone();  // або format!("{}", cmd)
let sent = send_command(cmd);
if sent {
    println!("Лог: команда '{}' надіслана", cmd_text);
}
```

Рішення Б — send_command повертає String назад:

```rust
fn send_command(cmd: String) -> (bool, String) {
    println!("  Надсилаю: {}", cmd);
    (true, cmd)  // повертаємо і результат, і команду
}
```

Рішення В (найкраще, Розділ 15) — send_command приймає `&str`:

```rust
fn send_command(cmd: &str) -> bool {
    println!("  Надсилаю: {}", cmd);
    true
}
// cmd залишається у caller
```

### Антипатерн: "clone-орієнтоване програмування"

Типова помилка початківців — додавати `.clone()` кожного разу, коли компілятор скаржиться. Подивіться на цей код:

```rust
fn check_battery(id: String) { println!("{}: батарея OK", id); }
fn check_sensors(id: String) { println!("{}: сенсори OK", id); }
fn check_comms(id: String) { println!("{}: зв'язок OK", id); }
fn check_navigation(id: String) { println!("{}: навігація OK", id); }

fn main() {
    let drone_id = String::from("БПЛА-07");
    
    check_battery(drone_id.clone());
    check_sensors(drone_id.clone());
    check_comms(drone_id.clone());
    check_navigation(drone_id);  // останній — без clone
}
```

Цей код працює, але кожен `clone()` виділяє нову пам'ять на купі та копіює всі байти рядка. Чотири виклики — три зайвих алокації. Правильне рішення — змінити всі функції на `fn check_battery(id: &str)`. Тоді жодного clone не потрібно:

```rust
fn check_battery(id: &str) { println!("{}: батарея OK", id); }
fn check_sensors(id: &str) { println!("{}: сенсори OK", id); }
fn check_comms(id: &str) { println!("{}: зв'язок OK", id); }
fn check_navigation(id: &str) { println!("{}: навігація OK", id); }

fn main() {
    let drone_id = String::from("БПЛА-07");
    
    check_battery(&drone_id);
    check_sensors(&drone_id);
    check_comms(&drone_id);
    check_navigation(&drone_id);
    
    println!("Всі перевірки для {} пройдено", drone_id);
}
```

Нуль алокацій, нуль копіювань, `drone_id` доступний до кінця. `cargo clippy` попередить про зайві clone — використовуйте його.

## 14.9. Стратегії виправлення: зведена таблиця

Для кожної ownership-помилки є кілька стратегій. Обирайте від найдешевшої до найдорожчої:

| Стратегія | Вартість | Коли використовувати |
|-----------|----------|---------------------|
| Змінити порядок операцій | Безкоштовно | Коли можна використати дані ДО move |
| Посилання `&T` / `&str` | Безкоштовно | Коли функція лише читає (найкращий варіант) |
| Повернення через return | Безкоштовно | Коли функція має повернути дані назад |
| `clone()` | Алокація + копіювання | Коли дійсно потрібні дві незалежні копії |
| Зміна архітектури | Час розробника | Коли жоден простий варіант не підходить |

Типова еволюція коду початківця:

Етап 1: "Нічого не компілюється!" — clone скрізь, щоб заткнути компілятор.

Етап 2: "Працює, але clippy лається" — усвідомлення, що clone часто зайвий.

Етап 3: "Посилання вирішують 90% проблем" — після вивчення Розділу 15.

Етап 4: "Ownership — це зручно" — розуміння, що правила ownership запобігають цілому класу багів.

## 14.10. Типові помилкові стратегії початківців

Крім антипатерну "clone скрізь" є кілька інших помилкових підходів.

### "Зробити все mut"

Початківець бачить помилку і думає: "може, `mut` допоможе":

```rust
fn main() {
    let mut s = String::from("test");
    let mut t = s;  // все ще move!
    println!("{}", s);  // все ще ПОМИЛКА
}
```

`mut` дозволяє змінювати значення, але не впливає на ownership. `let mut t = s` — все ще move. Ці дві концепції незалежні.

### "Повернути все з функцій"

Кожна функція приймає за значенням і повертає назад:

```rust
fn process(data: String) -> String {
    println!("{}", data);
    data
}

fn validate(data: String) -> String {
    assert!(!data.is_empty());
    data
}

fn main() {
    let mut s = String::from("test");
    s = process(s);
    s = validate(s);
    println!("{}", s);
}
```

Це працює, але заплутано: кожна функція приймає String і повертає String, хоча жодна не змінює дані. Набагато простіше: `fn process(data: &str)`.

### "Використати глобальну змінну"

Для `&str` літералів це працює (вони вкомпільовані і живуть вічно):

```rust
static DRONE_ID: &str = "БПЛА-07";
```

Але для `String` або `Vec` глобальні змінні потребують `static mut` та `unsafe` — це порушує безпеку Rust. Правильний підхід — передавати дані через параметри.

## Troubleshooting

### "Я додав .clone() і все працює — це нормально?"

Технічно — так, програма працює. Але це може бути антипатерн. Запитайте себе: "чи дійсно мені потрібна незалежна копія, чи функція лише читає дані?" Якщо лише читає — посилання (`&str`, `&T`) краще. Clone виправданий тільки коли потрібні дві незалежні копії.

### "Компілятор каже consider cloning — значить clone правильний?"

Компілятор пропонує clone як найпростіше виправлення, не обов'язково найкраще. Це як лікар, що пропонує знеболююче: воно прибирає симптом, але причина може бути глибшою. Часто правильніше змінити сигнатуру функції на посилання.

### "Навіщо ownership, якщо можна просто clone все?"

Так само, як "навіщо бережливість, якщо можна просто купити нове". Clone — це алокація пам'яті на купі + копіювання байтів. Для рядка у 10 символів — непомітно. Для вектора з мільйоном елементів — це мілісекунди затримки. Для БПЛА, що приймає рішення 100 разів на секунду — це катастрофа.

### "Я не розумію повідомлення компілятора"

Зосередьтесь на трьох речах: (1) знайдіть `value moved here` — це місце move, (2) знайдіть `value borrowed/used here after move` — це місце помилки, (3) прочитайте `help:` — компілятор часто пропонує конкретне виправлення. Якщо все одно незрозуміло — скопіюйте код + помилку та попросіть AI пояснити (Розділ PE).

## Додатково

### Інструмент cargo clippy

`cargo clippy` — це лінтер, що знаходить антипатерни у коді. Серед іншого, він попередить про зайві clone:

```bash
cargo clippy
```

Якщо clippy каже `unnecessary clone` — значить, дані можна передати без копіювання. Clippy також знайде невикористані змінні, неідіоматичний код, потенційні помилки логіки. Запускайте clippy після кожної компіляції — це безкоштовний code review.

### Шаблон "створити всередині функції"

Замість передачі String у функцію та повернення назад, часто простіше створити дані всередині:

```rust
fn create_status_report(id: &str, battery: u8) -> String {
    format!("Агент {}: батарея {}%", id, battery)
}
```

Функція приймає `&str` (позичає) та повертає новий `String` (передає володіння caller-у). Жодних move проблем. Цей патерн — "позичити вхідні дані, створити та повернути нові" — найчистіший з точки зору ownership.

### Ownership та продуктивність

Одна з переваг ownership — компілятор може краще оптимізувати код, знаючи, що кожне значення має єдиного власника. Move String — лише 24 байти на стеку (ptr, len, cap), самі дані залишаються на місці. Clone String "hello" — алокація на купі + копіювання = десятки наносекунд. Clone мегабайтного рядка — мікросекунди. Для БПЛА, що приймає рішення 100 разів на секунду, зайві clone — це реальна втрата бюджету часу.

## Контрольні запитання

### Рівень 1

1. Що означає код помилки E0382?

Відповідь: "borrow of moved value" — спроба використати значення, яке вже було переміщене.

2. Як знайти місце move у повідомленні компілятора?

Відповідь: шукайте фразу "value moved here" — компілятор вказує конкретний рядок.

### Рівень 2

3. Чому `for name in names` "з'їдає" вектор, а `for name in &names` — ні?

Відповідь: `for name in names` ітерує за значенням — кожен елемент переміщується з вектора. `for name in &names` ітерує за посиланнями — елементи позичаються, вектор залишається цілісним.

4. Коли clone виправданий, а коли — антипатерн?

Відповідь: виправданий — коли дійсно потрібні дві незалежні копії (оригінал + модифікована версія). Антипатерн — коли використовується лише для уникнення помилки компілятора при читанні даних (краще посилання).

### Рівень 3

5. Виправте двома способами:

```rust
fn process(data: Vec<String>) { for item in data { println!("{}", item); } }
fn main() {
    let items = vec![String::from("a"), String::from("b")];
    process(items);
    println!("{}", items.len());
}
```

Відповідь А: `process(items.clone());`. Відповідь Б: `fn process(data: &[String])` і `process(&items);`.

### Рівень 4

6. У якому порядку Rust знищує значення при виході зі scope? Чому це важливо?

Відповідь: у зворотному порядку створення (LIFO). Це важливо, бо пізніші значення можуть залежати від ранніх (наприклад, посилання на ранню змінну). Зворотний порядок гарантує, що залежні значення знищуються першими.

## Резюме

Повідомлення Rust-компілятора — це не покарання, а навчальний інструмент. Кожне повідомлення має шість компонентів: код помилки, опис, розташування, вказівники на рядки коду, пояснення причини та рекомендацію виправлення. Кольорове виділення у терміналі допомагає орієнтуватись.

Найчастіші ownership-помилки: E0382 — використання після move (при присвоєнні, передачі у функцію, у циклі). E0507 — move з контейнера. Часткове переміщення зі структури.

Для кожної помилки є стратегії виправлення від найдешевшої до найдорожчої: зміна порядку операцій (безкоштовно), посилання &T (безкоштовно), повернення через return (безкоштовно), clone (алокація + копіювання), зміна архітектури.

Clone виправданий тільки коли потрібні дві незалежні копії. "Clone-орієнтоване програмування" — антипатерн. Для читання даних — завжди посилання.

Типові помилкові стратегії: "зробити все mut" (не впливає на ownership), "повернути все з функцій" (громіздко), "глобальні змінні" (небезпечно).

`rustc --explain E0382` — розширене пояснення. `cargo clippy` — знаходить зайві clone.

Ключова навичка: побачивши помилку, знайти "value moved here", зрозуміти причину, обрати найдешевше виправлення.

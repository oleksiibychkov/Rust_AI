# Розділ 19. Модульне тестування

## Анотація

Код агента з Розділів 16–18 має структури, enum, модулі — але як переконатись, що все працює правильно? Ви можете запустити програму і подивитись на вивід. Але що, якщо зміна в одному модулі зламала інший? Що, якщо граничний випадок (батарея рівно 0%, координати на межі сітки) не оброблений? Ручна перевірка не масштабується: при 50 функціях перевірити кожну вручну після кожної зміни — нереально. Модульне тестування автоматизує цей процес: ви пишете код, що перевіряє ваш код. Одна команда `cargo test` запускає всі тести за секунди. Якщо щось зламалось — ви дізнаєтесь миттєво. Для БПЛА тести — це специфікація поведінки: "агент НЕ МОЖЕ злетіти з батареєю менше 10%", "зліт можливий ЛИШЕ зі стану Grounded" — кожне таке твердження стає тестом, що перевіряється автоматично.

---

## Цілі навчання

Після опрацювання цього розділу студент зможе:

1. Написати unit-тест з `#[test]` та запустити його через `cargo test`.
2. Використати `assert!`, `assert_eq!`, `assert_ne!` для перевірки умов.
3. Тестувати паніку через `#[should_panic]`.
4. Організувати тести в модулі `#[cfg(test)]`.
5. Тестувати приватні функції з тестового модуля.
6. Написати тести для інваріантів агента та переходів станів.

---

## Ключові терміни

**Unit test** — тест, що перевіряє одну функцію або метод ізольовано.

**Integration test** — тест, що перевіряє взаємодію кількох модулів.

**Assert (твердження)** — перевірка умови; якщо умова хибна — тест провалюється.

**Test runner** — програма, що знаходить та запускає тести (`cargo test`).

**#[test]** — атрибут, що позначає функцію як тест.

**#[cfg(test)]** — атрибут, що включає код лише при тестуванні.

**#[should_panic]** — атрибут для тестів, що очікують паніку.

---

## Мотиваційний кейс

NASA має правило: кожен рядок коду для космічних місій покритий мінімум трьома тестами. Mars Rover "Curiosity" має більше тестів, ніж рядків production-коду. Це не параноя — це інженерна дисципліна. Помилка у космосі не виправляється "наступним релізом". Для БПЛА ставки аналогічні: баг у навігації може призвести до втрати апарату. Тести — це ваша страховка: якщо тест проходить, поведінка відповідає специфікації.

## 19.1. Перший тест: #[test] та assert

### Найпростіший тест

Тест у Rust — це звичайна функція з атрибутом `#[test]`:

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[test]
fn test_add() {
    assert_eq!(add(2, 3), 5);
}
```

`#[test]` каже cargo: "це тестова функція, запускай її при `cargo test`". `assert_eq!(left, right)` перевіряє рівність. Якщо `add(2, 3)` повертає 5 — тест проходить. Якщо повертає щось інше — тест провалюється з повідомленням, що показує обидва значення.

Запуск:

```bash
cargo test
```

Вивід при успіху:

```
running 1 test
test test_add ... ok

test result: ok. 1 passed; 0 failed; 0 ignored
```

Вивід при провалі (якби add повертав a - b):

```
running 1 test
test test_add ... FAILED

failures:

---- test_add stdout ----
thread 'test_add' panicked at 'assertion failed: `(left == right)`
  left: `-1`,
 right: `5`'

test result: FAILED. 0 passed; 1 failed; 0 ignored
```

Компілятор показує: ліве значення (-1) та праве (5) — одразу видно проблему.

Тест — це звичайна функція без параметрів та без повернення (або з `Result`). Вона не приймає аргументів і не повертає значення. Якщо виконання досягає кінця без паніки — тест пройшов. Якщо `assert!` або `assert_eq!` спрацьовує — це паніка, тест провалюється.

Кожен тест запускається в окремому потоці. Провал одного тесту не зупиняє інші — cargo test запустить всі та покаже зведений результат.

### Що тестувати: правило трьох категорій

Для кожної функції або методу тестуйте три категорії:

Нормальні випадки — типовий вхід, очікуваний результат. `add(2, 3)` повертає 5. Це базова перевірка "функція взагалі працює".

Граничні випадки — крайні значення вхідних даних. `add(0, 0)` повертає 0. `add(i32::MAX, 0)` — чи не буде overflow? Батарея рівно 10% — чи можна злетіти?

Невалідні випадки — некоректний вхід. Від'ємна батарея? Координати за межами сітки? Зліт з повітря? Функція повинна або повернути помилку, або панікувати — не тихо зламатись.

### Три макроси для тверджень

`assert!(condition)` — перевіряє, що умова true:

```rust
#[test]
fn test_positive() {
    let battery = 85;
    assert!(battery > 0);
    assert!(battery <= 100);
}
```

`assert_eq!(left, right)` — перевіряє рівність:

```rust
#[test]
fn test_distance() {
    let dist = distance(0.0, 0.0, 3.0, 4.0);
    assert_eq!(dist, 5.0);
}
```

`assert_ne!(left, right)` — перевіряє нерівність:

```rust
#[test]
fn test_unique_ids() {
    let a = create_agent("SCOUT-01");
    let b = create_agent("SCOUT-02");
    assert_ne!(a.id(), b.id());
}
```

### Кастомні повідомлення

Додайте пояснення до assert — вони з'являться при провалі:

```rust
#[test]
fn test_battery_range() {
    let battery = 85;
    assert!(
        battery <= 100,
        "Батарея {} перевищує максимум 100", battery
    );
}
```

## 19.2. Організація тестів: #[cfg(test)]

### Тестовий модуль у тому ж файлі

Тести зазвичай розміщують у спеціальному модулі в кінці файлу:

```rust
// src/position.rs

pub struct Position {
    pub x: f64,
    pub y: f64,
}

impl Position {
    pub fn new(x: f64, y: f64) -> Self {
        Self { x, y }
    }
    
    pub fn distance_to(&self, other: &Position) -> f64 {
        let dx = other.x - self.x;
        let dy = other.y - self.y;
        (dx * dx + dy * dy).sqrt()
    }
    
    pub fn is_origin(&self) -> bool {
        self.x == 0.0 && self.y == 0.0
    }
}

#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_new() {
        let p = Position::new(3.0, 4.0);
        assert_eq!(p.x, 3.0);
        assert_eq!(p.y, 4.0);
    }
    
    #[test]
    fn test_origin() {
        let p = Position::new(0.0, 0.0);
        assert!(p.is_origin());
    }
    
    #[test]
    fn test_not_origin() {
        let p = Position::new(1.0, 0.0);
        assert!(!p.is_origin());
    }
    
    #[test]
    fn test_distance_pythagorean() {
        let a = Position::new(0.0, 0.0);
        let b = Position::new(3.0, 4.0);
        let dist = a.distance_to(&b);
        assert!((dist - 5.0).abs() < 0.0001, "Очікувалось 5.0, отримано {}", dist);
    }
    
    #[test]
    fn test_distance_same_point() {
        let p = Position::new(5.0, 5.0);
        assert_eq!(p.distance_to(&p), 0.0);
    }
}
```

`#[cfg(test)]` — атрибут умовної компіляції. Цей модуль компілюється ЛИШЕ при `cargo test`. У фінальному бінарнику (при `cargo build`) тестового коду не буде — нуль overhead.

`use super::*` — імпортує все з батьківського модуля (position). Тестовий модуль — це підмодуль position, тому super вказує на position. Завдяки цьому тести мають доступ до всіх елементів, включаючи приватні.

### Тестування приватних функцій

Одна з переваг Rust — тести у тому ж файлі мають доступ до приватних функцій:

```rust
fn internal_helper(x: f64) -> f64 {
    x * x + 1.0
}

pub fn public_api(x: f64) -> f64 {
    internal_helper(x).sqrt()
}

#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_internal() {
        // Тестуємо приватну функцію — це дозволено!
        assert_eq!(internal_helper(3.0), 10.0);
    }
    
    #[test]
    fn test_public() {
        let result = public_api(3.0);
        assert!((result - 10.0_f64.sqrt()).abs() < 0.0001);
    }
}
```

Це зручно для юніт-тестування: ви тестуєте кожну функцію окремо, включаючи внутрішні helper-и.

## 19.3. Тестування паніки: #[should_panic]

Деякі функції повинні панікувати при невалідних вхідних даних. `#[should_panic]` перевіряє, що паніка відбулась:

```rust
fn divide(a: f64, b: f64) -> f64 {
    if b == 0.0 {
        panic!("Ділення на нуль!");
    }
    a / b
}

#[test]
#[should_panic(expected = "Ділення на нуль")]
fn test_divide_by_zero() {
    divide(10.0, 0.0);  // повинна панікувати
}
```

`expected = "..."` — опціональний рядок, що повинен міститись у повідомленні паніки. Якщо паніка не відбулась або повідомлення інше — тест провалюється.

## 19.4. Тестування Result замість паніки

Замість assert, тест може повертати Result:

```rust
#[test]
fn test_parse() -> Result<(), String> {
    let value: i32 = "42".parse().map_err(|e| format!("{}", e))?;
    if value != 42 {
        return Err(format!("Очікувалось 42, отримано {}", value));
    }
    Ok(())
}
```

Тест проходить, якщо повертає `Ok(())`. Провалюється при `Err(...)`. Оператор `?` автоматично повертає Err при помилці — зручно для тестів з кількома операціями, що можуть завершитись помилкою.

## 19.5. Практика: тести для БПЛА-агента

Тепер напишемо тести для Agent з Розділів 16–17. Кожен тест перевіряє один інваріант або один перехід стану:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    
    // --- Тести конструктора ---
    
    #[test]
    fn test_new_agent_is_grounded() {
        let agent = Agent::new("TEST-01");
        assert_eq!(agent.state().name(), "Grounded");
    }
    
    #[test]
    fn test_new_agent_full_battery() {
        let agent = Agent::new("TEST-01");
        assert_eq!(agent.battery(), 100);
    }
    
    #[test]
    fn test_new_agent_at_origin() {
        let agent = Agent::new("TEST-01");
        assert!(agent.position().is_origin());
    }
    
    // --- Тести переходів станів ---
    
    #[test]
    fn test_takeoff_from_grounded() {
        let mut agent = Agent::new("TEST-01");
        agent.takeoff();
        assert_eq!(agent.state().name(), "Airborne");
    }
    
    #[test]
    fn test_takeoff_consumes_battery() {
        let mut agent = Agent::new("TEST-01");
        agent.takeoff();
        assert_eq!(agent.battery(), 90);  // 100 - 10
    }
    
    #[test]
    fn test_cannot_takeoff_low_battery() {
        let mut agent = Agent::new("TEST-01");
        // Штучно зменшуємо батарею (потрібен метод або pub(crate) поле)
        for _ in 0..19 {
            agent.takeoff();
            if agent.state().name() == "Airborne" {
                agent.land();
            }
        }
        // Батарея < 10, зліт неможливий
        let state_before = String::from(agent.state().name());
        agent.takeoff();
        assert_eq!(agent.state().name(), state_before);
    }
    
    #[test]
    fn test_land_from_airborne() {
        let mut agent = Agent::new("TEST-01");
        agent.takeoff();
        agent.land();
        assert_eq!(agent.state().name(), "Grounded");
    }
    
    #[test]
    fn test_cannot_land_from_grounded() {
        let mut agent = Agent::new("TEST-01");
        agent.land();  // вже на землі
        assert_eq!(agent.state().name(), "Grounded");
    }
    
    // --- Тести інваріантів ---
    
    #[test]
    fn test_battery_never_exceeds_100() {
        let agent = Agent::new("TEST-01");
        assert!(agent.battery() <= 100);
    }
    
    #[test]
    fn test_emergency_from_any_state() {
        let mut agent = Agent::new("TEST-01");
        agent.emergency("тест");
        assert_eq!(agent.state().name(), "Emergency");
    }
}
```

Кожен тест має зрозуміле ім'я, що описує що перевіряється. `test_takeoff_from_grounded` — перевіряє, що зліт з Grounded переводить у Airborne. `test_cannot_takeoff_low_battery` — перевіряє, що зліт неможливий при низькій батареї. `test_emergency_from_any_state` — перевіряє, що аварійний режим доступний завжди.

### Запуск тестів

```bash
cargo test
```

Вивід:

```
running 9 tests
test tests::test_new_agent_is_grounded ... ok
test tests::test_new_agent_full_battery ... ok
test tests::test_new_agent_at_origin ... ok
test tests::test_takeoff_from_grounded ... ok
test tests::test_takeoff_consumes_battery ... ok
test tests::test_cannot_takeoff_low_battery ... ok
test tests::test_land_from_airborne ... ok
test tests::test_cannot_land_from_grounded ... ok
test tests::test_emergency_from_any_state ... ok

test result: ok. 9 passed; 0 failed; 0 ignored
```

Якщо хоча б один тест провалюється — cargo test показує деталі та exit code 1. Це зручно для CI/CD — автоматичної перевірки при кожному коміті.

### Фільтрація тестів

Запуск лише тестів, що містять "battery" у назві:

```bash
cargo test battery
```

Запуск тестів з виводом println (за замовчуванням вивід придушується):

```bash
cargo test -- --nocapture
```

## 19.6. Тест як специфікація поведінки

Тести — це не лише перевірка коду. Це формальна специфікація поведінки системи. Кожен тест — це твердження: "моя система поводиться так". Якщо хтось змінить код і зламає поведінку — тест це покаже.

Для БПЛА це критично. Ось список тверджень про агента, кожне з яких стає тестом:

"Новий агент завжди починає з Grounded та 100% батареї" — `test_new_agent_is_grounded`, `test_new_agent_full_battery`.

"Зліт можливий лише зі стану Grounded" — `test_takeoff_from_grounded`, `test_cannot_takeoff_from_airborne`, `test_cannot_takeoff_from_charging`.

"Зліт витрачає рівно 10% батареї" — `test_takeoff_consumes_battery`.

"Зліт неможливий, якщо батарея менше 10%" — `test_cannot_takeoff_low_battery`.

"Аварійний режим доступний з будь-якого стану" — `test_emergency_from_grounded`, `test_emergency_from_airborne`, `test_emergency_from_charging`.

"Батарея ніколи не перевищує 100% і не менше 0%" — `test_battery_never_exceeds_100`, `test_battery_never_below_zero`.

Кожне таке твердження — один тест. Якщо всі тести проходять — система відповідає специфікації. Якщо тест провалюється — ви знаєте точно, яке твердження порушено, і можете негайно це виправити.

### Workflow розробки з тестами

Ефективний підхід до розробки з тестами:

Крок 1: Напишіть тест для поведінки, яку хочете реалізувати. Тест провалюється (бо код ще не написаний).

Крок 2: Напишіть мінімальний код, що робить тест зеленим.

Крок 3: Рефакторинг — покращіть код, не ламаючи тести.

Крок 4: Повторіть для наступної поведінки.

Цей підхід називається TDD (Test-Driven Development). Ми не вимагаємо строгого TDD, але принцип "спершу тест, потім код" — корисна навичка.

### Скільки тестів достатньо

Правило: кожна публічна функція або метод повинна мати хоча б один тест. Для кожного методу, що змінює стан (`&mut self`) — три тести: нормальний випадок, граничний, невалідний. Для переходів стану — тест кожного дозволеного та кожного забороненого переходу.

Для Agent з Розділу 17 (4 стани, 5 методів) мінімальний набір — 15–20 тестів. Це не забагато: кожен тест — 3–5 рядків, загалом ~100 рядків тестового коду.

## 19.7. Integration tests

Unit-тести (у `#[cfg(test)]`) перевіряють окремі функції. Integration tests перевіряють взаємодію модулів — "чи працює система в цілому". Вони розміщуються в директорії `tests/` на рівні `src/`:

```
my_drone/
├── Cargo.toml
├── src/
│   ├── lib.rs          // бібліотечний crate
│   ├── agent.rs
│   └── position.rs
└── tests/
    └── mission_test.rs
```

```rust
// tests/mission_test.rs
use my_drone::Agent;  // імпорт з lib.rs

#[test]
fn test_full_mission() {
    let mut agent = Agent::new("MISSION-01");
    
    // Повна місія: зліт → рух × 3 → посадка
    agent.takeoff();
    for _ in 0..3 {
        agent.move_forward();
    }
    agent.land();
    
    assert_eq!(agent.state().name(), "Grounded");
    assert!(agent.battery() < 100);
}

#[test]
fn test_emergency_interrupts_mission() {
    let mut agent = Agent::new("MISSION-02");
    agent.takeoff();
    agent.move_forward();
    agent.emergency("втрата зв'язку");
    
    assert_eq!(agent.state().name(), "Emergency");
    // Після аварії агент більше не повинен рухатись
}
```

Ключова відмінність: integration tests бачать лише публічний API (`pub` елементи). Вони не мають доступу до приватних функцій та полів. Це перевіряє, що публічний інтерфейс працює правильно — так, як його бачить зовнішній користувач.

Для integration tests проєкт повинен мати `lib.rs` (бібліотечний crate). Для навчального проєкту з лише `main.rs` — використовуйте unit-тести.

### Коли unit, а коли integration

Unit-тести — для кожної функції окремо. "Чи правильно `distance_to` обчислює відстань?" "Чи `takeoff` змінює стан?"

Integration-тести — для сценаріїв. "Чи повна місія (зліт → рух → посадка) завершується коректно?" "Чи аварійний режим перериває місію?"

Обидва види потрібні: unit-тести ловлять помилки на рівні окремих функцій, integration-тести — на рівні взаємодії.

## 19.8. Тестові патерни та антипатерни

### Патерн: один тест — одне твердження

Кожен тест перевіряє рівно одну річ. Якщо тест провалюється — ви одразу знаєте, що саме зламалось:

```rust
// Добре: один тест — одне твердження
#[test]
fn test_takeoff_changes_state() {
    let mut a = Agent::new("T");
    a.takeoff();
    assert_eq!(a.state().name(), "Airborne");
}

#[test]
fn test_takeoff_reduces_battery() {
    let mut a = Agent::new("T");
    a.takeoff();
    assert_eq!(a.battery(), 90);
}
```

```rust
// Погано: кілька перевірок в одному тесті
#[test]
fn test_takeoff_everything() {
    let mut a = Agent::new("T");
    a.takeoff();
    assert_eq!(a.state().name(), "Airborne");  // якщо це впаде...
    assert_eq!(a.battery(), 90);                // ...це не запуститься
    assert!(!a.position().is_origin());         // ...і це теж
}
```

У другому варіанті, якщо перший assert впаде — решта не виконається. Ви не дізнаєтесь, чи батарея правильна. Окремі тести краще ізолюють проблему.

### Патерн: Arrange-Act-Assert (AAA)

Кожен тест має три фази. Arrange — підготовка (створення об'єктів, встановлення стану). Act — дія (виклик тестованого методу). Assert — перевірка (чи результат відповідає очікуванню):

```rust
#[test]
fn test_land_from_airborne() {
    // Arrange
    let mut agent = Agent::new("T");
    agent.takeoff();  // переводимо у Airborne
    
    // Act
    agent.land();
    
    // Assert
    assert_eq!(agent.state().name(), "Grounded");
}
```

### Антипатерн: тести без assert

```rust
#[test]
fn test_mission() {
    let mut a = Agent::new("T");
    a.takeoff();
    a.move_forward();
    a.land();
    // Де assert? Тест "проходить", навіть якщо все зламано!
}
```

Тест без assert завжди "зелений" — він нічого не перевіряє. Це хибне відчуття безпеки.

### Антипатерн: тестування приватних деталей реалізації

Тестуйте поведінку, а не реалізацію. "Після зльоту стан = Airborne" — це поведінка (тестувати). "Після зльоту внутрішній лічильник = 42" — це деталь реалізації (може змінитись при рефакторингу).

## Prompt Engineering: генерація тестів

### Промпт-шаблон

```
Ось мій код Agent з методами takeoff, land, move_forward:
```

```
    assert_eq!(a.battery(), 90);
}

    // Рекомендую додати pub(crate) fn set_battery_for_test
    ...
}

#[test]
```

### Як оцінити AI-тести

AI часто пропускає граничні випадки. Перевіряйте: чи покрита батарея рівно 10% (граничне значення)? Чи є тести для кожного стану, з якого не можна злетіти? Чи перевіряється, що дані (altitude, battery) коректні після операції, а не лише стан?

Додайте тести, які AI пропустив. Запишіть у промпт-журнал: що прийнято, що відхилено, що додано.

## Troubleshooting

### `cargo test` не знаходить тести

Перевірте: чи є `#[test]` перед кожною тестовою функцією? Чи тестовий модуль всередині `#[cfg(test)]`? Чи тестова функція не приймає параметрів? Чи функція не повертає значення (або повертає Result)?

### `assert_eq!` не працює для float

Порівняння f64 через `==` ненадійне через помилки округлення. `0.1 + 0.2` не дорівнює `0.3` у більшості мов програмування — це особливість IEEE 754:

```rust
// ПОГАНО: може провалитись через округлення
assert_eq!(0.1 + 0.2, 0.3);

// ДОБРЕ: перевірка з допуском (epsilon)
assert!((0.1 + 0.2 - 0.3).abs() < 1e-10);
```

Для відстаней та координат агента — завжди використовуйте допуск. Типове значення epsilon для f64 — 1e-9 або 1e-10.

### Тест провалюється, але код працює

Тест може бути написаний неправильно. Перевірте: чи правильні очікувані значення? Чи не плутаєте порядок аргументів у `assert_eq!`? В `assert_eq!(left, right)` — `left` зазвичай фактичне значення, `right` — очікуване. При провалі cargo test виводить обидва.

### Тести впливають один на одного

Кожен тест запускається в окремому потоці, але за замовчуванням — паралельно. Якщо тести змінюють глобальний стан (файли, змінні оточення) — вони можуть конфліктувати. Рішення: кожен тест створює свій екземпляр Agent; не використовуйте глобальні змінні у тестах; при потребі запускайте послідовно: `cargo test -- --test-threads=1`.

### `the trait Debug is not implemented`

`assert_eq!` вимагає, щоб тип реалізував Debug (для виводу при провалі):

```rust
#[derive(Debug)]  // додайте цей атрибут
struct Position { x: f64, y: f64 }
```

Без `#[derive(Debug)]` assert_eq не зможе вивести значення при провалі.

### `the trait PartialEq is not implemented`

`assert_eq!` порівнює через `==`, що потребує PartialEq:

```rust
#[derive(Debug, PartialEq)]  // додайте PartialEq
struct Position { x: f64, y: f64 }
```

Тепер `assert_eq!(pos_a, pos_b)` працюватиме.

## Додатково

### Тести як документація

Тести — найнадійніша документація. Коментарі застарівають, README забувають оновити. Але тести, що проходять — завжди актуальні. Побачивши `test_takeoff_from_grounded`, ви знаєте: зліт з Grounded працює. Побачивши `test_cannot_takeoff_low_battery` — зліт при низькій батареї блокується. Новий розробник може прочитати тести і зрозуміти, як система повинна поводитись — без документації.

### Чому тести критичні перед паралелізмом

У Частині IV ви додасте багатопотоковість: кілька агентів працюватимуть одночасно, ділячи дані через Arc та Mutex. Баги в паралельному коді — найскладніші для відтворення: вони з'являються випадково, залежать від таймінгу, і можуть зникати при дебагу. Тести, написані зараз, гарантують, що базова логіка правильна. Якщо паралельна версія ламає тест — проблема саме у паралелізмі, а не у логіці агента. Це звужує пошук бага від "де завгодно у коді" до "у паралельній обгортці".

### #[ignore] — тимчасово вимкнути тест

```rust
#[test]
#[ignore]
fn test_long_mission() {
    // Цей тест займає 30 секунд — ігноруємо при звичайному cargo test
    for _ in 0..10000 {
        // ... тривала симуляція ...
    }
}
```

Запуск ігнорованих тестів: `cargo test -- --ignored`. Запуск всіх тестів (включаючи ігноровані): `cargo test -- --include-ignored`.

### Організація тестів: конвенції іменування

Називайте тести за шаблоном `test_<що_тестується>_<очікуваний_результат>`:

- `test_new_agent_is_grounded` — конструктор створює агента в стані Grounded
- `test_takeoff_consumes_battery` — зліт зменшує батарею
- `test_cannot_takeoff_low_battery` — зліт неможливий при низькому заряді
- `test_distance_same_point_is_zero` — відстань до себе = 0

Ім'я тесту — це його документація. При провалі cargo test показує ім'я — воно повинно пояснювати, що саме зламалось.

### Корисні опції cargo test

`cargo test` — запустити всі тести. `cargo test battery` — тільки тести з "battery" у назві. `cargo test -- --nocapture` — показати вивід println у тестах. `cargo test -- --test-threads=1` — запускати тести послідовно (для тестів з побічними ефектами). `cargo test --doc` — запустити тести з документаційних коментарів.

## Контрольні запитання

### Рівень 1

1. Що робить `#[test]`?

Відповідь: позначає функцію як тест, що запускається через `cargo test`.

2. Чим `assert_eq!` відрізняється від `assert!`?

Відповідь: `assert_eq!(a, b)` перевіряє рівність та показує обидва значення при провалі. `assert!(condition)` перевіряє довільну умову.

### Рівень 2

3. Чому тести розміщують у `#[cfg(test)]`?

Відповідь: щоб тестовий код не потрапляв у фінальний бінарник. `#[cfg(test)]` компілюється лише при `cargo test`.

4. Як тестувати приватні функції?

Відповідь: тестовий модуль `#[cfg(test)] mod tests` — підмодуль поточного модуля. `use super::*` дає доступ до всіх елементів, включаючи приватні.

### Рівень 3

5. Напишіть тест, що перевіряє: "агент не може злетіти з батареєю менше 10%".

### Рівень 4

6. Чому тести називають "специфікацією поведінки"? Як це пов'язано з паралелізмом?

Відповідь: тест формалізує очікувану поведінку. Якщо тест проходить — поведінка відповідає специфікації. При додаванні паралелізму тести гарантують, що базова логіка не зламалась — будь-який провал вказує на проблему саме у паралельному коді.

## Резюме

`#[test]` позначає функцію як тест. `cargo test` запускає всі тести. `#[cfg(test)]` — тестовий код не потрапляє у бінарник.

Три макроси: `assert!(condition)`, `assert_eq!(left, right)`, `assert_ne!(left, right)`. Кастомні повідомлення — третім аргументом.

`#[should_panic]` — тест очікує паніку. Тест може повертати Result.

Тести у тому ж файлі (`#[cfg(test)] mod tests`) мають доступ до приватних функцій через `use super::*`.

Integration tests — у директорії `tests/`, бачать лише публічний API.

Тест — специфікація поведінки. Кожен інваріант, кожен перехід стану — окремий тест.

Для f64 — перевірка з допуском: `(a - b).abs() < epsilon`.

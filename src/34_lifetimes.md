# Розділ 34. Lifetimes: анотації часу життя

## Анотація

У Розділі 15 ви вивчили borrowing: &T та &mut T — посилання, що "позичають" дані без передачі ownership. Компілятор перевіряв, що посилання не переживає дані, на які вказує. У більшості випадків це працювало автоматично — без жодних анотацій з вашого боку. Але є ситуації, де компілятор не може визначити час життя посилання самостійно: функція приймає два посилання і повертає одне — яке з вхідних? Структура зберігає посилання — як довго воно валідне? Для таких випадків Rust має lifetime annotations — `'a`, `'b` — явні позначки, що зв'язують час життя посилань. Lifetimes — найскладніша тема для початківців у Rust, але після розуміння стають інтуїтивними. Головне: lifetime — це не "скільки живе змінна", а "як довго посилання гарантовано валідне".

## Цілі навчання

Після опрацювання цього розділу студент зможе:

1. Пояснити, що таке lifetime і чому компілятор іноді потребує анотацій.
2. Написати lifetime-анотації у функціях: `fn f<'a>(x: &'a str) -> &'a str`.
3. Пояснити три правила lifetime elision.
4. Створити struct з посиланнями: `struct View<'a> { data: &'a [f64] }`.
5. Пояснити `'static` та коли він потрібен.
6. Діагностувати типові lifetime-помилки компілятора.

## Ключові терміни

**Lifetime (час життя)** — область коду, протягом якої посилання гарантовано валідне. Позначається `'a`, `'b` тощо.

**Lifetime annotation (анотація часу життя)** — явна позначка `'a` у сигнатурі функції чи структури, що зв'язує час життя кількох посилань.

**Lifetime elision (виведення часу життя)** — набір правил, за якими компілятор виводить lifetimes автоматично, без явних анотацій.

**'static** — спеціальний lifetime, що означає "живе протягом усього виконання програми". Рядкові літерали ("hello") мають тип `&'static str`.

**Dangling reference (висяче посилання)** — посилання на дані, що вже звільнені. Rust запобігає цьому на етапі компіляції.

## Мотиваційний кейс

Агент БПЛА під час місії створює "вікно" (view) у свій стан — підмножину даних для передачі координатору. Координатор не потребує повного стану агента — лише позицію, батарею та список виявлених об'єктів. Копіювати ці дані — марнотратство (особливо список об'єктів). Посилання — ефективно, але як гарантувати, що координатор не тримає "вікно" довше, ніж живе стан агента? Якщо агент оновив стан або знищився — "вікно" стає dangling reference. Lifetime annotations — механізм Rust для формалізації цієї гарантії: "view живе не довше, ніж agent_state".

## 34.1. Проблема: коли компілятор не може визначити lifetime

У більшості коду lifetimes виводяться автоматично. Ви писали функції з посиланнями десятки разів без жодних `'a`:

```rust
fn first_word(s: &str) -> &str {
    let bytes = s.as_bytes();
    for (i, &byte) in bytes.iter().enumerate() {
        if byte == b' ' {
            return &s[0..i];
        }
    }
    s
}
```

Компілятор розуміє: вхід один (`&str`), вихід один (`&str`) — повернене посилання живе стільки, скільки живе вхідне. Все очевидно.

Але що коли входів два?

```rust
// НЕ компілюється без lifetime анотацій
fn longer(a: &str, b: &str) -> &str {
    if a.len() > b.len() { a } else { b }
}
```

```
error[E0106]: missing lifetime specifier
 --> src/main.rs:1:33
  |
1 | fn longer(a: &str, b: &str) -> &str {
  |              ----     ----      ^ expected named lifetime parameter
  |
help: this function's return type contains a borrowed value,
      but the signature does not say whether it is borrowed from `a` or `b`
```

Компілятор запитує: "повернене посилання — це частина `a` чи `b`? Від цього залежить, як довго воно валідне." Якщо a живе до рядка 10, а b — до рядка 20, і функція повертає b — посилання валідне до рядка 20. Якщо повертає a — лише до рядка 10. Але компілятор не знає, яку гілку if виконає — це залежить від runtime. Тому потрібна анотація.

### Ментальна модель: що таке lifetime

Перш ніж вивчати синтаксис, побудуємо правильну ментальну модель. Lifetime — це не властивість змінної і не "тривалість існування". Lifetime — це область коду (region), протягом якої посилання гарантовано вказує на валідні дані.

Кожна змінна має scope — від оголошення до кінця блоку. Кожне посилання має lifetime — від створення до останнього використання. Lifetime посилання повинен вкладатись у scope даних, на які воно вказує. Якщо посилання "виживає" після знищення даних — це dangling reference, і Rust забороняє це на етапі компіляції.

```rust
fn main() {
    let x = 5;            // ------+-- lifetime 'x починається
    let r = &x;           // --+-- | lifetime 'r починається
    println!("{}", r);     //   |  |
                           // --+  | 'r закінчується (останнє використання)
                           //      |
}                          // -----+ 'x закінчується (scope)
// 'r повністю вкладається у 'x — безпечно
```

А тепер небезпечний випадок:

```rust
fn main() {
    let r;                 // ------+-- 'r починається
    {                      //       |
        let x = 5;         // -+--- | 'x починається
        r = &x;            //  |   | r посилається на x
    }                      // -+   | 'x закінчується — x знищено!
    println!("{}", r);     //      | 'r ще живий, але дані вже немає
}                          // -----+
// 'r НЕ вкладається у 'x — ПОМИЛКА компіляції
```

Компілятор бачить: r використовується після `}`, де x вже знищено. Посилання вказує на звільнену пам'ять. Помилка: "borrowed value does not live long enough".

Ця модель — "lifetime як область, що повинна вкладатись" — ключ до розуміння всіх lifetime-помилок. Коли компілятор каже "lifetime 'a не живе достатньо довго" — він має на увазі: "область, де посилання використовується, виходить за межі області, де дані гарантовано існують".

Аналогія: бібліотечна картка. Ви берете книгу (дані) з бібліотеки і даєте другу картку з номером полиці (посилання). Поки книга на полиці — картка корисна. Якщо книгу повернули (дані знищені) — картка вказує на порожню полицю. Lifetime — це "термін дії картки": він не може перевищувати термін перебування книги на полиці.

## 34.2. Синтаксис lifetime-анотацій

Lifetime annotation — апостроф + ім'я: `'a`, `'b`, `'data`, `'input`. За конвенцією використовують короткі: `'a`, `'b`. Анотація ставиться після `&`:

```rust
fn longer<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}
```

`<'a>` після імені функції — оголошення lifetime параметра (аналогічно `<T>` для generic типу). `&'a str` — посилання з lifetime 'a. Сигнатура каже: "обидва вхідних посилання мають lifetime 'a, і повернене посилання теж має lifetime 'a".

Що це означає на практиці? Повернене посилання валідне протягом перетину (intersection) lifetimes a та b — тобто протягом коротшого з двох. Компілятор підставляє конкретний lifetime при кожному виклику:

```rust
fn main() {
    let result;
    let string1 = String::from("довгий рядок");
    {
        let string2 = String::from("xyz");
        result = longer(string1.as_str(), string2.as_str());
        println!("Довший: {}", result); // OK: обидва живі
    }
    // println!("{}", result); // ПОМИЛКА: string2 вже знищено,
    //                         // result може вказувати на неї
}
```

Компілятор бачить: string2 живе до кінця блоку `{}`. Lifetime 'a для цього виклику — перетин lifetime string1 та string2, тобто до кінця `{}`. Використання result після `}` — помилка, бо 'a вже завершився.

Важливе уточнення: lifetime-анотації не змінюють, як довго живуть дані. Вони лише описують зв'язки між посиланнями, щоб компілятор міг перевірити безпеку. `'a` — це не "зробити посилання довше", а "пообіцяти, що обидва живуть достатньо довго".

Найпоширеніша помилка початківців — думати, що `'a` "подовжує" lifetime. Ні. Якщо string2 живе до рядка 8, а ви написали `'a` — string2 все одно живе до рядка 8. Анотація лише каже компілятору: "перевір, що результат не використовується після рядка 8". Якщо використовується — помилка компіляції. Lifetime annotations — це не інструкції, а контракт: обіцянка програміста, яку компілятор перевіряє.

### Різні сценарії з lifetime

Не завжди обидва входи мають однаковий lifetime. Іноді функція завжди повертає перший аргумент:

```rust
fn with_default<'a>(value: &'a str, _default: &str) -> &'a str {
    if value.is_empty() {
        "N/A" // повертаємо літерал ('static), не _default
    } else {
        value // повертаємо value ('a)
    }
}
```

Тут результат завжди має lifetime 'a (від value). `_default` має свій, незв'язаний lifetime — він не впливає на вихід. Якщо функція ніколи не повертає частину `_default` — його lifetime не потрібно зв'язувати з виходом.

Інший сценарій — функція, що створює нові дані:

```rust
fn make_greeting(name: &str) -> String {
    format!("Привіт, {}!", name)
}
```

Тут жодних lifetime-анотацій. Функція повертає owned String — не посилання. String не "позичена" ні від кого, вона створена заново. Lifetimes потрібні лише коли функція повертає посилання.

Правило: lifetime-анотації потрібні лише у сигнатурах, де є вхідні та вихідні посилання, і компілятор не може визначити зв'язок через elision rules. Усередині тіла функції компілятор відстежує lifetimes автоматично.

## 34.3. Lifetime elision: коли анотації не потрібні

У ранніх версіях Rust кожне посилання потребувало явної анотації. Код виглядав так: `fn first_word<'a>(s: &'a str) -> &'a str`. Розробники помітили, що в 90% випадків анотації передбачувані — і додали правила elision, за якими компілятор виводить lifetimes автоматично. Це не "магія" — це три чітких правила, що застосовуються послідовно.

Правило 1: кожен параметр-посилання отримує свій власний lifetime. `fn f(x: &str, y: &str)` стає `fn f<'a, 'b>(x: &'a str, y: &'b str)`. Два параметри — два незалежних lifetime.

Правило 2: якщо є рівно один вхідний lifetime — він призначається всім вихідним. `fn f(x: &str) -> &str` стає `fn f<'a>(x: &'a str) -> &'a str`. Логіка: якщо вхід один, вихід може посилатися лише на нього.

Правило 3: якщо один із параметрів — `&self` або `&mut self`, його lifetime призначається всім вихідним. Логіка: метод зазвичай повертає частину self.

Після застосування трьох правил, якщо залишились невизначені вихідні lifetimes — компілятор вимагає явних анотацій.

Прослідкуємо покроково, як компілятор застосовує правила для кількох прикладів:

Приклад 1: `fn first_char(s: &str) -> &str`

Крок 1 (правило 1): один параметр-посилання → `fn first_char<'a>(s: &'a str) -> &str`. Крок 2 (правило 2): один вхідний lifetime → він же вихідний: `fn first_char<'a>(s: &'a str) -> &'a str`. Всі вихідні lifetimes визначені — анотації не потрібні.

Приклад 2: `fn longer(a: &str, b: &str) -> &str`

Крок 1 (правило 1): два параметри → `fn longer<'a, 'b>(a: &'a str, b: &'b str) -> &str`. Крок 2 (правило 2): два вхідних lifetime (не один) — правило не застосовується. Крок 3 (правило 3): жоден параметр не &self — правило не застосовується. Вихідний lifetime невизначений — компілятор вимагає анотацію.

Приклад 3: `fn name(&self) -> &str` (метод)

Крок 1 (правило 1): один параметр &self → `fn name<'a>(&'a self) -> &str`. Крок 2 (правило 2): один вхідний — але правило 3 має пріоритет для &self. Крок 3 (правило 3): є &self → його lifetime для виходу: `fn name<'a>(&'a self) -> &'a str`. Визначено — анотації не потрібні.

Приклади:

```rust
// Один вхід → правило 2 працює, анотації не потрібні
fn first_char(s: &str) -> &str { &s[0..1] }
// Компілятор виводить: fn first_char<'a>(s: &'a str) -> &'a str

// Метод з &self → правило 3 працює
impl Agent {
    fn name(&self) -> &str { &self.id }
    // Компілятор виводить: fn name<'a>(&'a self) -> &'a str
}

// Два входи, жоден не &self → правила не достатні, потрібна анотація
fn longer(a: &str, b: &str) -> &str { ... } // ПОМИЛКА
fn longer<'a>(a: &'a str, b: &'a str) -> &'a str { ... } // OK
```

Правило: якщо компілятор не скаржиться — lifetimes виведені автоматично. Додавайте анотації лише коли компілятор явно просить. Не додавайте "на всяк випадок" — зайві анотації ускладнюють читання.

## 34.4. Структури з посиланнями

Структура може зберігати посилання замість owned даних. Це ефективно (немає копіювання), але потребує lifetime-анотації — компілятор повинен знати, що посилання всередині структури живе достатньо довго.

Чому структура потребує анотації, а локальна змінна — ні? Тому що структура може бути передана у іншу функцію, повернута, збережена у колекції — компілятор повинен знати lifetime кожного посилання всередині, щоб перевірити всі ці операції. Для локальної змінної `let r = &x;` компілятор бачить і x, і r у тому самому scope і перевіряє безпосередньо.

`<'a>` після імені структури означає: "ця структура параметризована lifetime 'a — кожен екземпляр прив'язаний до конкретних даних і не може пережити їх".

```rust
#[derive(Debug)]
struct AgentView<'a> {
    id: &'a str,
    position: &'a (f64, f64),
    readings: &'a [f64],
}

impl<'a> AgentView<'a> {
    fn average_reading(&self) -> f64 {
        if self.readings.is_empty() { return 0.0; }
        let sum: f64 = self.readings.iter().sum();
        sum / self.readings.len() as f64
    }
}

fn main() {
    let id = String::from("SCOUT-01");
    let pos = (47.5, 35.2);
    let data = vec![10.0, 20.0, 15.0, 22.0, 18.0];
    
    let view = AgentView {
        id: &id,
        position: &pos,
        readings: &data,
    };
    
    println!("Агент {} на {:?}, avg={:.1}", 
             view.id, view.position, view.average_reading());
    
    // view не може пережити id, pos, data:
    // drop(id); // ПОМИЛКА, якщо view ще використовується
}
```

`AgentView<'a>` означає: "ця структура містить посилання з lifetime 'a, і сама структура не може жити довше, ніж 'a". Компілятор гарантує, що AgentView не переживе дані, на які вказує. Це "вікно" у стан агента — ефективне (нуль копіювання) та безпечне (compile-time гарантія).

`impl<'a> AgentView<'a>` — lifetime оголошується і для impl-блоку. Методи можуть повертати посилання з тим самим lifetime (правило 3 elision).

Зверніть увагу на поле `battery: u8` — якби воно було у AgentView, воно б не потребувало lifetime. Copy-типи (числа, bool, char) копіюються при створенні структури — вони не "позичені", а "скопійовані". Lifetime потрібен лише для полів-посилань (&str, &[T], &T).

Коли структура з посиланням, а коли з owned даними? Посилання — коли структура "тимчасова" (view, ітератор, результат пошуку) і не повинна переживати джерело. Типовий приклад: AgentView існує лише протягом обробки запиту координатора, потім знищується. Owned — коли структура "самостійна" (Agent, MissionPlan) і зберігається довго, незалежно від зовнішніх даних. Agent з `id: String` може жити скільки завгодно. Agent з `id: &str` прив'язаний до lifetime джерела рядка.

Загальна порада для початківців: починайте з owned типів (String, Vec, власні структури). Переходьте на посилання лише коли це дає вимірювану перевагу (уникнення копіювання великих даних) або необхідне архітектурно (view, ітератор).

## 34.5. 'static: найбільш непорозумілий lifetime

`'static` — спеціальний lifetime, що означає "живе протягом усього виконання програми". Це найдовший можливий lifetime. Але навколо `'static` існує стільки плутанини, що варто розібрати окремо кожне його значення.

Перше значення: `&'static T` — посилання, що живе "вічно". Найтиповіший приклад — рядкові літерали:

```rust
fn main() {
    let s: &'static str = "цей рядок живе вічно";
    println!("{}", s);
}
```

Літерал `"цей рядок..."` вшитий у бінарний файл програми. Він існує з моменту запуску до завершення — тому посилання на нього є `&'static str`. Функція може повернути `&'static str`, бо дані ніколи не знищуються.

Де ще зустрічається `&'static T`: константи (`const MAX_ALT: &str = "500"`), значення, ліковані через `Box::leak` (навмисний "витік" на купу, що живе до кінця програми — рідкісний патерн).

Друге значення: trait bound `T: 'static`. Це інше! Означає: "тип T не містить посилань з обмеженим lifetime — або owned тип, або тип із `&'static` посиланнями всередині". Це НЕ означає "T живе вічно". String є `T: 'static` — хоча String знищується при drop як будь-який інший тип. Суть у тому, що String не містить посилань на чужі дані із обмеженим lifetime — він повністю самодостатній.

Чому це важливо? `thread::spawn` вимагає `F: FnOnce() + Send + 'static`. Потік може жити довільно довго — можливо, довше за scope, де його створили. Якщо замикання захоплює `&local_variable` — це посилання стане dangling, коли scope завершиться, а потік ще працює. `'static` bound гарантує: "замикання не містить таких посилань":

```rust
fn main() {
    let name = String::from("SCOUT-01");
    
    // OK: move передає ownership String у потік
    // String: 'static (не містить non-static посилань)
    std::thread::spawn(move || {
        println!("{}", name);
    });
    
    // НЕ OK: &name — посилання з обмеженим lifetime
    // let r = &name;
    // std::thread::spawn(move || { println!("{}", r); }); // ПОМИЛКА
}
```

Поширена помилка — плутати `&'static str` та `T: 'static`. Таблиця для ясності:

| Запис | Що означає | Приклад |
|-------|-----------|---------|
| `&'static str` | Посилання, що живе вічно | Рядковий літерал "hello" |
| `T: 'static` | Тип не містить non-static посилань | String, Vec<i32>, i32 |
| `&'a str` | Посилання з lifetime 'a | Посилання на частину String |

String є `'static` (не містить посилань), але `&String` — ні (це посилання з обмеженим lifetime). `i32` є `'static`, `&i32` — ні.

Третя плутанина: "'static означає memory leak". Ні. `'static` для owned типів (String, Vec) — це лише trait bound, що каже "тип самодостатній". Значення все одно знищується нормально при drop. Memory leak відбувається лише при `Box::leak` — і це свідомий рідкісний патерн, а не наслідок `'static`.

## 34.6. Кілька lifetime параметрів

Іноді функція має посилання з різними lifetimes:

```rust
fn first_or_default<'a, 'b>(data: &'a [f64], default: &'b f64) -> &'a f64 {
    if data.is_empty() {
        // Тут проблема: хочемо повернути default ('b), 
        // але сигнатура обіцяє 'a
        // Потрібно або змінити сигнатуру, або повернути з data
        &0.0 // це не працює — літерал не має lifetime 'a
    } else {
        &data[0]
    }
}
```

Якщо функція може повернути посилання з будь-якого входу — всі повинні мати однаковий lifetime:

```rust
fn first_or_default<'a>(data: &'a [f64], default: &'a f64) -> &'a f64 {
    if data.is_empty() { default } else { &data[0] }
}

fn main() {
    let numbers = vec![10.0, 20.0, 30.0];
    let fallback = 0.0;
    
    let result = first_or_default(&numbers, &fallback);
    println!("Результат: {}", result);
}
```

Один lifetime 'a — означає "повернене посилання валідне протягом перетину lifetimes обох входів". Це коректно: якщо будь-який з входів знищується — результат теж невалідний, бо він може посилатися на будь-який із них.

## 34.7. Практика: AgentView для БПЛА

```rust
use std::collections::HashMap;

struct AgentState {
    id: String,
    position: (f64, f64),
    battery: u8,
    altitude: f64,
    history: Vec<(f64, f64)>,
    objects: HashMap<String, (f64, f64)>,
}

// View — "вікно" у стан агента, без копіювання
struct AgentView<'a> {
    id: &'a str,
    position: &'a (f64, f64),
    battery: u8,  // Copy-тип, можна копіювати
    recent_positions: &'a [(f64, f64)],
}

impl AgentState {
    fn create_view(&self, history_depth: usize) -> AgentView<'_> {
        let start = if history_depth >= self.history.len() { 
            0 
        } else { 
            self.history.len() - history_depth 
        };
        
        AgentView {
            id: &self.id,
            position: &self.position,
            battery: self.battery,
            recent_positions: &self.history[start..],
        }
    }
}

impl<'a> AgentView<'a> {
    fn summary(&self) -> String {
        format!("[{}] на {:?}, bat={}%, history={} точок",
                self.id, self.position, self.battery, self.recent_positions.len())
    }
}

fn report_to_coordinator(view: &AgentView) {
    // Функція працює з view — не потребує доступу до повного AgentState
    println!("Звіт: {}", view.summary());
}

fn main() {
    let mut state = AgentState {
        id: String::from("SCOUT-01"),
        position: (47.5, 35.2),
        battery: 85,
        altitude: 150.0,
        history: vec![(0.0, 0.0), (10.0, 5.0), (20.0, 15.0), (30.0, 25.0), (47.5, 35.2)],
        objects: HashMap::new(),
    };
    
    // Створюємо view — легковагий "знімок"
    let view = state.create_view(3);
    report_to_coordinator(&view);
    
    // view не може пережити state:
    // drop(state); // ПОМИЛКА тут, якщо view ще використовується
}
```

`AgentView<'_>` у поверненні create_view — `'_` означає "виведи lifetime автоматично" (elision). Компілятор виводить: lifetime повернення = lifetime &self. Тобто view живе не довше, ніж state.

Чому view, а не clone? AgentState може мати Vec з тисячами точок та HashMap з сотнями об'єктів. Clone скопіює все — це мілісекунди та кілобайти. AgentView — нуль копіювання, лише кілька посилань (байти). Для telemetry з тисячами оновлень на секунду — різниця суттєва.

## Troubleshooting

### Помилка 1: `missing lifetime specifier`

Функція повертає посилання, але компілятор не може визначити lifetime:

```rust
fn longer(a: &str, b: &str) -> &str { ... }
```

Виправлення: додати `'a`: `fn longer<'a>(a: &'a str, b: &'a str) -> &'a str`.

### Помилка 2: `borrowed value does not live long enough`

Посилання використовується після знищення даних:

```rust
fn main() {
    let r;
    {
        let s = String::from("hello");
        r = &s; // позичання
    } // s знищено
    println!("{}", r); // ПОМИЛКА: r — dangling reference
}
```

Виправлення: перемістити s у зовнішній scope, або клонувати замість посилання.

### Помилка 3: `cannot return reference to local variable`

```rust
fn make_greeting() -> &str {
    let s = String::from("hello");
    &s // ПОМИЛКА: s знищується при поверненні
}
```

Виправлення: повернути String (owned), не &str. Або повернути &'static str (літерал).

### Помилка 4: `lifetime may not live long enough` у struct

```rust
struct View { data: &str } // ПОМИЛКА: missing lifetime
```

Виправлення: `struct View<'a> { data: &'a str }`.

## Додатково

### Типові хибні уявлення про lifetimes

Хибне уявлення 1: "Lifetime annotation робить дані довше живучими". Ні. Annotation — це опис, не інструкція. `'a` каже компілятору "перевір, що дані живуть достатньо довго", а не "зроби їх довше живучими". Якщо дані знищуються на рядку 10, жодна анотація не продовжить їхнє існування.

Хибне уявлення 2: "Кожне посилання потребує lifetime annotation". Ні. Більшість lifetimes виводяться автоматично через elision. Додавайте анотації лише коли компілятор просить. Зайві анотації ускладнюють код без користі.

Хибне уявлення 3: "Lifetimes — це runtime механізм". Ні. Lifetimes існують лише при компіляції. У бінарному файлі жодних lifetime перевірок немає. Це pure compile-time гарантія — zero runtime cost.

Хибне уявлення 4: "Якщо я не розумію lifetimes — потрібно клонувати все". Clone — легітимне рішення, коли lifetime-архітектура занадто складна. Але розуміння lifetimes відкриває можливість писати ефективніший код (zero-copy views, ітератори, parser combinators).

### Lifetime subtyping

Lifetime 'a є "підтипом" 'b, якщо 'a живе не менше ніж 'b. Тобто `'long: 'short` — 'long є підтипом 'short. Це означає: де очікується посилання з коротким lifetime, можна передати посилання з довшим. Це інтуїтивно: якщо щось живе 10 секунд, його можна безпечно використовувати там, де потрібно 5 секунд.

### Lifetime та closures

Замикання, що захоплюють посилання, мають lifetime пов'язаний із захопленими даними. `move` closure забирає ownership — lifetime не обмежений. Non-move closure з &T — обмежений lifetime T. Саме тому thread::spawn вимагає `'static` — потік може жити довше за будь-який конкретний lifetime.

## Контрольні запитання

### Рівень 1

1. Що таке lifetime annotation?

Відповідь: явна позначка `'a`, що зв'язує час життя кількох посилань. Каже компілятору: "ці посилання живуть однаково довго (або результат живе не довше за входи)".

2. Коли lifetime-анотації не потрібні?

Відповідь: коли компілятор може вивести їх через elision rules: один вхідний lifetime → він же вихідний; &self → lifetime self для виходу.

### Рівень 2

3. Перелічіть три правила lifetime elision.

Відповідь: (1) кожен вхідний параметр-посилання отримує свій lifetime. (2) Якщо один вхідний lifetime — він призначається всім вихідним. (3) Якщо є &self — lifetime self призначається всім вихідним.

4. Чим `'static` відрізняється від інших lifetimes?

Відповідь: `'static` означає "живе протягом усього виконання програми". Для `&'static str` — рядковий літерал у бінарнику. Для `T: 'static` — тип не містить non-static посилань (owned типи або &'static).

### Рівень 3

5. Чому цей код не компілюється та як виправити?

```rust
fn first_or<'a>(data: &'a [i32], default: &i32) -> &'a i32 {
    if data.is_empty() { default } else { &data[0] }
}
```

Відповідь: default має інший (неоголошений) lifetime, але функція повертає його як 'a. Виправлення: `fn first_or<'a>(data: &'a [i32], default: &'a i32) -> &'a i32` — обидва входи один lifetime.

### Рівень 4

6. Поясніть різницю між `&'static str` та `String` і коли обирати що.

Відповідь: `&'static str` — посилання на рядок, вшитий у бінарник (літерал). Не виділяє пам'ять на купі, не можна змінювати, живе вічно. String — owned рядок на купі, можна змінювати, звільняється при drop. `&'static str` — для констант та літералів. String — для динамічних рядків.

## Резюме

Lifetime — область коду, де посилання гарантовано валідне. Анотації `'a` зв'язують lifetimes кількох посилань: "результат живе не довше за входи". Анотації не змінюють час життя — лише описують зв'язки для перевірки компілятором.

Elision rules виводять lifetimes автоматично у більшості випадків: один вхід → він же вихід, &self → lifetime self. Явні анотації потрібні, коли кілька вхідних посилань і компілятор не може визначити, якому належить вихід.

Структури з посиланнями (`AgentView<'a>`) — ефективні "вікна" у дані без копіювання. Struct не може пережити дані, на які посилається.

`'static` — живе вічно. Для літералів (`&'static str`) та як bound (`T: 'static` — тип без non-static посилань).

Правило: не додавайте lifetime-анотації "на всяк випадок". Пишіть без них. Коли компілятор попросить — додайте мінімально необхідні.

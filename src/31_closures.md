# Розділ 31. Closures: замикання

## Анотація

У Розділі 23 ви вже використовували замикання: `|x| x * 2` у map, `|&&x| x > 0.0` у filter, `|acc, &x| acc + x` у fold. Тоді ми не заглиблювались у механізм — достатньо було синтаксису. Тепер час зрозуміти замикання глибоко: як вони захоплюють змінні з оточення, які три trait (Fn, FnMut, FnOnce) їх класифікують, як `move` передає ownership у замикання, і як замикання стануть фундаментом для потоків та асинхронного коду.

## Ключові терміни

**Closure (замикання)** — анонімна функція, що може захоплювати змінні з оточуючого scope. Синтаксис: `|параметри| тіло`.

**Capture (захоплення)** — процес, при якому замикання "бере" змінну з оточення — за посиланням, за мутабельним посиланням, або за значенням.

**Fn** — trait для замикань, що захоплюють оточення за іммутабельним посиланням. Можна викликати скільки завгодно разів.

**FnMut** — trait для замикань, що захоплюють за мутабельним посиланням. Можна викликати скільки завгодно, але потребує `&mut`.

**FnOnce** — trait для замикань, що захоплюють за значенням (move). Можна викликати лише один раз — після виклику захоплені значення спожиті.

**move closure** — замикання з ключовим словом `move`, що примусово захоплює всі змінні за значенням (ownership), незалежно від того, як вони використовуються.

## Мотиваційний кейс

Система керування роєм БПЛА потребує гнучких стратегій поведінки. Один агент патрулює за маршрутом, інший — слідує за ціллю, третій — уникає перешкод. Стратегія — це функція, що приймає стан агента і повертає дію. Але стратегія може залежати від контексту: "слідувати за ціллю з координатами (X, Y)" — тут X та Y є параметрами стратегії, а не аргументами функції. Замикання ідеально для цього: `|state| navigate_to(state, target_x, target_y)` — замикання "запам'ятовує" target_x та target_y з моменту створення.

## 31.1. Замикання vs функція: в чому різниця

Функція у Rust (fn) не може звертатись до локальних змінних зовнішнього scope. Кожна функція — ізольований блок з доступом лише до своїх параметрів та глобальних констант:

```rust
fn main() {
    let threshold = 50.0;
    
    fn is_above(value: f64) -> bool {
        value > threshold // ПОМИЛКА: can't capture dynamic environment
    }
}
```

Замикання може:

```rust
fn main() {
    let threshold = 50.0;
    
    let is_above = |value: f64| -> bool {
        value > threshold // OK: замикання захоплює threshold
    };
    
    println!("{}", is_above(60.0)); // true
    println!("{}", is_above(30.0)); // false
}
```

Замикання `|value| value > threshold` "бачить" змінну `threshold` з зовнішнього scope і використовує її. Це і є "замикання" (closure): функція, що "замикається" навколо свого оточення, захоплюючи потрібні змінні.

Синтаксичні відмінності: параметри у вертикальних рисках `|x, y|` замість дужок `(x, y)`. Типи параметрів зазвичай виводяться автоматично (не потрібно писати `|x: f64|`, достатньо `|x|`). Тип повернення виводиться з тіла. Для однорядкових замикань фігурні дужки необов'язкові: `|x| x * 2` замість `|x| { x * 2 }`.

Замикання — це не "спеціальний синтаксис для коротких функцій". Це принципово інший механізм: замикання може зберігати стан через захоплені змінні. Функція — stateless: кожен виклик починає з чистого аркуша. Замикання — potentially stateful: кожен виклик може залежати від захопленого контексту.

## 31.2. Три режими захоплення

Rust автоматично обирає мінімально необхідний рівень захоплення для кожної змінної. Це тісно пов'язано з ownership та borrowing з Частини II.

Захоплення за іммутабельним посиланням (`&T`) — замикання лише читає змінну:

```rust
fn main() {
    let name = String::from("SCOUT-01");
    
    let greet = || println!("Агент: {}", name); // захоплює &name
    
    greet();
    greet(); // можна викликати скільки завгодно
    println!("name доступне: {}", name); // name не переміщене
}
```

Замикання використовує `name` лише для читання — Rust автоматично захоплює за `&String`. Оригінальна змінна залишається доступною.

Захоплення за мутабельним посиланням (`&mut T`) — замикання модифікує змінну:

```rust
fn main() {
    let mut counter = 0;
    
    let mut increment = || {
        counter += 1; // захоплює &mut counter
        println!("Лічильник: {}", counter);
    };
    
    increment(); // 1
    increment(); // 2
    increment(); // 3
    
    // counter тут доступний знову (мутабельне позичання завершилось)
    println!("Фінал: {}", counter); // 3
}
```

Замикання модифікує `counter` — Rust захоплює за `&mut i32`. Зверніть увагу: саме замикання оголошене як `let mut increment` — бо виклик замикання змінює його внутрішній стан (захоплене мутабельне посилання). Поки замикання "живе" і використовується — counter позичений як &mut, і ви не можете використати counter напряму.

Захоплення за значенням (move) — замикання забирає ownership:

```rust
fn main() {
    let data = vec![1, 2, 3];
    
    let consume = move || {
        println!("Дані: {:?}", data); // data переміщено у замикання
    };
    
    consume();
    // println!("{:?}", data); // ПОМИЛКА: data переміщено
}
```

`move` перед замиканням примушує захопити всі змінні за значенням. data переміщується у замикання і більше не доступна у зовнішньому scope. Це критично для потоків (Розділ 36): потік може пережити scope, де створено замикання, тому посилання на локальні змінні неприпустимі — лише ownership.

Без `move` Rust обирає автоматично — мінімальний рівень. Якщо замикання лише читає — &T. Якщо модифікує — &mut T. `move` перевизначає це рішення, завжди забираючи ownership. Для Copy-типів (i32, f64, bool) move означає копіювання значення у замикання — оригінал залишається доступним.

## 31.3. Fn, FnMut, FnOnce: три trait замикань

Компілятор автоматично визначає, який trait реалізує замикання, на основі того, як воно використовує захоплені змінні:

`Fn` — замикання не модифікує та не споживає захоплене. Можна викликати скільки завгодно разів через &self. Методи map, filter, any, all, find приймають Fn.

`FnMut` — замикання модифікує захоплене через &mut. Можна викликати скільки завгодно, але потребує мутабельного доступу до замикання. Метод for_each приймає FnMut.

`FnOnce` — замикання споживає (move) захоплене. Можна викликати лише один раз, бо після виклику захоплені значення знищені. `thread::spawn` приймає FnOnce.

```rust
fn apply_fn<F: Fn(f64) -> f64>(f: &F, value: f64) -> f64 {
    f(value) // можна викликати скільки завгодно
}

fn apply_fn_mut<F: FnMut()>(f: &mut F) {
    f(); // потребує &mut
    f(); // можна викликати знову
}

fn apply_fn_once<F: FnOnce() -> Vec<i32>>(f: F) -> Vec<i32> {
    f() // тільки один раз — f спожито
    // f() // ПОМИЛКА: f вже спожито
}

fn main() {
    // Fn: лише читає threshold
    let threshold = 100.0;
    let check = |alt: f64| alt > threshold;
    println!("{}", apply_fn(&check, 150.0)); // true
    println!("{}", apply_fn(&check, 50.0));  // false
    
    // FnMut: модифікує count
    let mut count = 0;
    let mut counter = || { count += 1; println!("count = {}", count); };
    apply_fn_mut(&mut counter); // count = 1, count = 2
    
    // FnOnce: споживає data
    let data = vec![1, 2, 3];
    let producer = move || data; // move — забирає data
    let result = apply_fn_once(producer);
    println!("{:?}", result); // [1, 2, 3]
}
```

Ієрархія: кожне Fn є також FnMut, кожне FnMut є також FnOnce. Тобто якщо функція приймає `FnOnce` — їй можна передати і Fn, і FnMut. Якщо приймає `Fn` — лише Fn (найобмеженіший). Тому бібліотечні функції зазвичай запитують мінімально необхідний trait: map запитує FnMut (хоча зазвичай отримує Fn — це працює, бо Fn : FnMut).

## 31.4. Замикання як параметри та повернення

Замикання як параметр функції — ви вже це бачили з ітераторами. Два синтаксиси:

```rust
// impl Fn — static dispatch, monomorphization
fn apply(value: f64, transform: impl Fn(f64) -> f64) -> f64 {
    transform(value)
}

// Generic — еквівалент, більш гнучкий
fn apply_generic<F: Fn(f64) -> f64>(value: f64, transform: F) -> f64 {
    transform(value)
}

fn main() {
    let result = apply(100.0, |x| x * 1.1);
    println!("{}", result); // 110.0
    
    let factor = 2.5;
    let scaled = apply(100.0, |x| x * factor); // замикання захоплює factor
    println!("{}", scaled); // 250.0
}
```

Замикання як повернення функції — для цього потрібен Box<dyn Fn> або impl Fn:

```rust
fn make_threshold_checker(threshold: f64) -> impl Fn(f64) -> bool {
    move |value| value > threshold // move — бо threshold має пережити функцію
}

fn main() {
    let is_high = make_threshold_checker(200.0);
    let is_low = make_threshold_checker(50.0);
    
    println!("300 > 200? {}", is_high(300.0)); // true
    println!("300 > 50? {}", is_low(300.0));   // true
    println!("100 > 200? {}", is_high(100.0)); // false
}
```

`move` тут обов'язковий: без нього замикання захопило б `&threshold`, але threshold — локальна змінна make_threshold_checker, яка знищується при поверненні. Посилання на знищену змінну — dangling reference. `move` переміщує threshold у замикання, і воно живе стільки, скільки живе замикання.

Це фабрика замикань: `make_threshold_checker(200.0)` створює замикання з "зашитим" порогом 200. Кожен виклик фабрики — нове замикання зі своїм порогом. Потужний патерн для стратегій БПЛА.

## 31.5. Практика: стратегії поведінки БПЛА як замикання

Замикання ідеальні для моделювання стратегій, бо стратегія — це "дія + контекст":

```rust
type Strategy = Box<dyn Fn((f64, f64), u8) -> String>;

fn patrol_strategy(waypoints: Vec<(f64, f64)>) -> Strategy {
    let mut current = 0;
    Box::new(move |pos: (f64, f64), battery: u8| {
        if battery < 20 {
            return String::from("RETURN_TO_BASE");
        }
        let target = waypoints[current % waypoints.len()];
        // current += 1; // не працює — Fn, не FnMut. Спрощуємо:
        format!("MOVE_TO ({:.0}, {:.0})", target.0, target.1)
    })
}

fn follow_target_strategy(target: (f64, f64)) -> Strategy {
    Box::new(move |pos: (f64, f64), battery: u8| {
        if battery < 15 {
            return String::from("RETURN_TO_BASE");
        }
        let dx = target.0 - pos.0;
        let dy = target.1 - pos.1;
        format!("FOLLOW ({:.0}, {:.0})", dx.signum(), dy.signum())
    })
}

fn execute_strategy(strategy: &Strategy, pos: (f64, f64), battery: u8) {
    let action = strategy(pos, battery);
    println!("Позиція {:?}, батарея {}% → {}", pos, battery, action);
}

fn main() {
    let patrol = patrol_strategy(vec![(0.0, 0.0), (10.0, 0.0), (10.0, 10.0)]);
    let follow = follow_target_strategy((50.0, 50.0));
    
    println!("--- Патруль ---");
    execute_strategy(&patrol, (0.0, 0.0), 80);
    execute_strategy(&patrol, (5.0, 0.0), 45);
    execute_strategy(&patrol, (8.0, 0.0), 10); // low battery
    
    println!("--- Слідування ---");
    execute_strategy(&follow, (10.0, 20.0), 90);
    execute_strategy(&follow, (30.0, 40.0), 12); // low battery
}
```

`type Strategy = Box<dyn Fn(...)>` — алías для trait object замикання. Кожна фабрика (`patrol_strategy`, `follow_target_strategy`) повертає замикання з "зашитим" контекстом (waypoints або target). Функція `execute_strategy` не знає, яка стратегія всередині — вона працює з абстракцією.

Цей патерн — strategy pattern через замикання — легший за повноцінні trait objects з Розділу 28. Для простих стратегій (одна функція з контекстом) замикання зручніші. Для складних (кілька методів, внутрішній стан) — trait objects кращі.

## Troubleshooting

### Помилка 1: `closure may outlive the current function`

Замикання захоплює посилання на локальну змінну, але повертається з функції:

```rust
fn make_greeter(name: &str) -> impl Fn() {
    || println!("Hello, {}", name) // ПОМИЛКА: name не живе достатньо
}
```

Виправлення: `move` та owned тип:

```rust
fn make_greeter(name: String) -> impl Fn() {
    move || println!("Hello, {}", name)
}
```

Або якщо параметр &str — clone перед move:

```rust
fn make_greeter(name: &str) -> impl Fn() {
    let owned = name.to_string();
    move || println!("Hello, {}", owned)
}
```

### Помилка 2: `cannot borrow as mutable — closure is Fn, not FnMut`

Замикання модифікує захоплену змінну, але параметр вимагає Fn:

```rust
fn apply(f: impl Fn()) { f(); f(); }

fn main() {
    let mut count = 0;
    apply(|| { count += 1; }); // ПОМИЛКА: closure is FnMut, apply вимагає Fn
}
```

Виправлення: змінити bound на FnMut: `fn apply(mut f: impl FnMut())`.

### Помилка 3: `use of moved value` після move closure

```rust
fn main() {
    let data = vec![1, 2, 3];
    let closure = move || println!("{:?}", data);
    println!("{:?}", data); // ПОМИЛКА: data переміщено у closure
}
```

Виправлення: clone перед move (`let data_clone = data.clone(); let closure = move || println!("{:?}", data_clone);`), або переструктурувати код, щоб data не потрібна після замикання.

### Помилка 4: `expected closure, found fn`

Намагаєтесь повернути різні замикання з різних гілок:

```rust
fn choose(flag: bool) -> impl Fn(i32) -> i32 {
    if flag {
        |x| x + 1
    } else {
        |x| x * 2 // ПОМИЛКА: різні замикання — різні типи
    }
}
```

Виправлення: `Box<dyn Fn(i32) -> i32>` замість `impl Fn`:

```rust
fn choose(flag: bool) -> Box<dyn Fn(i32) -> i32> {
    if flag {
        Box::new(|x| x + 1)
    } else {
        Box::new(|x| x * 2)
    }
}
```

## Додатково

### Замикання під капотом: anonymous struct

Компілятор перетворює кожне замикання на анонімну struct, що зберігає захоплені змінні, та реалізує відповідний Fn/FnMut/FnOnce trait. Замикання `|x| x + threshold` приблизно еквівалентне:

```rust
struct AnonymousClosure {
    threshold: f64, // захоплена змінна
}

impl Fn<(f64,)> for AnonymousClosure {
    fn call(&self, (x,): (f64,)) -> f64 {
        x + self.threshold
    }
}
```

Кожне замикання — унікальний тип. Два `|x| x + 1` — два різних типи, навіть якщо ідентичні за кодом. Тому `impl Fn(...)` у поверненні функції може повернути лише одне конкретне замикання, а `Box<dyn Fn(...)>` — будь-яке.

### Зв'язок з потоками

`std::thread::spawn` приймає замикання `FnOnce() -> T + Send + 'static`. FnOnce — бо потік запускається один раз. Send — бо замикання передається в інший потік. `'static` — бо потік може пережити scope створення. Це пояснює, чому `move` майже завжди потрібен з thread::spawn: без move замикання захоплює посилання на локальні змінні, а посилання не є 'static. Детально — у Розділі 36.

## Контрольні запитання

### Рівень 1

1. Чим замикання відрізняється від функції?

Відповідь: замикання може захоплювати змінні з оточуючого scope, функція — ні. Замикання має анонімний тип. Функція — іменований тип fn.

2. Що означає `move` перед замиканням?

Відповідь: примусове захоплення всіх змінних за значенням (ownership). Без move Rust обирає мінімальний режим: &T, &mut T, або move — автоматично.

### Рівень 2

3. Чим Fn відрізняється від FnMut та FnOnce?

Відповідь: Fn — замикання лише читає захоплене (&self), можна викликати скільки завгодно. FnMut — модифікує (&mut self), можна викликати скільки завгодно. FnOnce — споживає (self), лише один раз. Ієрархія: Fn : FnMut : FnOnce — кожне Fn є FnMut, кожне FnMut є FnOnce.

4. Чому `move` потрібний при поверненні замикання з функції?

Відповідь: без move замикання захоплює &T на локальну змінну. При поверненні з функції змінна знищується, а замикання матиме dangling reference. move переміщує значення у замикання — воно живе стільки, скільки живе замикання.

### Рівень 3

5. Напишіть фабрику: `fn make_adder(n: i32) -> impl Fn(i32) -> i32`.

Відповідь:

```rust
fn make_adder(n: i32) -> impl Fn(i32) -> i32 {
    move |x| x + n
}
```

### Рівень 4

6. Чому кожне замикання має унікальний тип, навіть якщо код ідентичний?

Відповідь: компілятор генерує анонімну struct для кожного замикання. Навіть два `|x| x + 1` — різні struct (хоча без захоплених полів). Це дозволяє monomorphization: компілятор знає точний тип замикання при генерації generic-коду, і може інлайнити виклик. Якби всі замикання мали один тип — потрібен dynamic dispatch, як з dyn Fn.

## Резюме

Замикання — анонімні функції, що захоплюють змінні з оточення. Три режими захоплення: за &T (читання), за &mut T (модифікація), за значенням/move (ownership). Rust обирає мінімальний автоматично; `move` примушує ownership.

Три trait: Fn (лише читає, скільки завгодно викликів), FnMut (модифікує, скільки завгодно), FnOnce (споживає, один виклик). Ієрархія: Fn : FnMut : FnOnce.

Замикання як параметри: `impl Fn(...)` або generic `F: Fn(...)` — static dispatch. Замикання як повернення: `impl Fn(...)` для одного типу, `Box<dyn Fn(...)>` для динамічного вибору.

Кожне замикання — унікальний анонімний тип. Це дозволяє monomorphization та інлайнінг без runtime overhead.

move closures критичні для потоків: потік може пережити scope створення, тому потребує ownership над даними, а не посилання.

---

## Що далі

Замикання дали "мозок" для потоків — фрагмент коду зі своїм контекстом. Але деякі структури даних потребують явного розміщення на купі: рекурсивні типи (дерево рішень агента), великі об'єкти, trait objects. У Розділі 32 ви вивчите `Box<T>` — найпростіший smart pointer, що володіє даними на купі. Box<dyn Trait> ви вже використовували для trait objects — тепер зрозумієте, як він працює під капотом. А рекурсивні типи (дерево, зв'язний список) стануть можливими завдяки Box, що розриває нескінченну рекурсію розміру.

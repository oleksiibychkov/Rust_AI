# Розділ 43. Rayon: паралелізм даних

## Анотація

Tokio оптимізований для IO-bound конкурентності: тисячі задач, що більшість часу чекають мережу чи диск. Але деякі задачі — CPU-bound: обчислення маршрутів для 100 агентів, аналіз зображень з камер, статистичний аналіз великих наборів даних сенсорів. Для таких задач потрібен паралелізм даних: розділити дані на частини, обробити кожну частину на окремому ядрі, зібрати результати. Rayon — бібліотека, що робить це тривіальним: замініть `.iter()` на `.par_iter()`, і ваш код працює паралельно на всіх ядрах. Rayon використовує work-stealing scheduler — ядра, що завершили свою частину, "крадуть" роботу у зайнятих. Результат — автоматичне балансування навантаження без ручного розподілу.

---

## Цілі навчання

Після опрацювання цього розділу студент зможе:

1. Пояснити різницю між паралелізмом даних (Rayon) та конкурентністю задач (Tokio).
2. Використати `par_iter()`, `par_iter_mut()`, `into_par_iter()` для паралельної обробки.
3. Використати `par_chunks()` для обробки блоками.
4. Використати `join()` для паралельного виконання двох обчислень.
5. Пояснити work-stealing та чому він ефективний.
6. Обрати між Rayon та Tokio для конкретної задачі.

## Ключові терміни

**Data parallelism (паралелізм даних)** — один алгоритм застосовується до різних частин даних одночасно. "Обчислити відстань для кожного з 1000 агентів — 250 на кожне з 4 ядер."

**Task parallelism (паралелізм задач)** — різні задачі виконуються одночасно. "Один потік сканує, інший аналізує, третій логує."

**Work stealing** — стратегія балансування: потік, що завершив свою роботу, "краде" задачі з черги іншого потоку. Автоматичне балансування без ручного розподілу.

**par_iter()** — паралельний ітератор Rayon. API ідентичний звичайному iter(), але обробка розподілена по ядрах.

---

## Мотиваційний кейс

Координатор рою повинен обчислити оптимальний маршрут для кожного зі 100 агентів. Один маршрут — 10мс CPU. Послідовно: 100 * 10мс = 1 секунда. На 8 ядрах з Rayon: ~130мс (не ідеальні 125мс через overhead). Зміна у коді: `.iter().map(calculate_route)` → `.par_iter().map(calculate_route)`. Одне слово (`par_`) — 8x прискорення.

## 43.1. Rayon vs Tokio: два види паралелізму

Tokio та Rayon вирішують різні задачі, і їх часто плутають.

Tokio — для IO-bound конкурентності. Тисячі задач, що чекають мережу/диск/таймер. Задачі легковагі (кілобайти). Потоків мало (= кількість ядер). Ефективність — у тому, що потік не простоює при очікуванні. Використовує async/await.

Rayon — для CPU-bound паралелізму. Десятки-тисячі обчислень, кожне навантажує процесор. Розподіляє роботу по ядрах через work-stealing. Потоків = кількість ядер. Ефективність — у тому, що всі ядра зайняті корисною роботою. Використовує звичайний sync код.

Ключова помилка: використовувати Rayon для IO (мережеві запити) або Tokio для CPU (обчислення без .await). Rayon для IO — потоки блокуються на очікуванні, марнотратство. Tokio для CPU — тривале обчислення блокує async runtime, інші задачі не виконуються.

У реальній системі обидва працюють разом: Tokio для мережевої комунікації рою, Rayon для обчислень маршрутів та аналізу даних. Інтеграція: `tokio::task::spawn_blocking(|| rayon_computation())`.

## 43.2. Підключення та par_iter

```toml
[dependencies]
rayon = "1.10"
```

Найпростіший приклад — паралельна обробка вектора:

```rust
use rayon::prelude::*;

fn main() {
    let data: Vec<u64> = (1..=10_000_000).collect();
    
    // Послідовно
    let start = std::time::Instant::now();
    let sum_seq: u64 = data.iter().map(|&x| x * x).sum();
    let seq_time = start.elapsed();
    
    // Паралельно
    let start = std::time::Instant::now();
    let sum_par: u64 = data.par_iter().map(|&x| x * x).sum();
    let par_time = start.elapsed();
    
    assert_eq!(sum_seq, sum_par);
    println!("Послідовно: {:?}", seq_time);
    println!("Паралельно: {:?}", par_time);
    println!("Прискорення: {:.1}x", seq_time.as_secs_f64() / par_time.as_secs_f64());
}
```

`use rayon::prelude::*` імпортує trait `ParallelIterator` та метод `par_iter()`. API паралельного ітератора майже ідентичний звичайному: `map`, `filter`, `sum`, `collect`, `fold`, `any`, `all`, `find_any`, `for_each`. Замініть `iter()` → `par_iter()`, і обробка розподіляється по ядрах.

Не кожна операція дає прискорення. Якщо обробка одного елемента — наносекунди (просте додавання) — overhead розподілу по потоках перевищує виграш. Rayon окупається коли обробка елемента — мікросекунди або більше, або коли елементів дуже багато (мільйони).

Три варіанти паралельного ітератора — аналогічно звичайним:

```rust
use rayon::prelude::*;

fn main() {
    let data = vec![1, 2, 3, 4, 5];
    
    // par_iter(): &T — читання
    let sum: i32 = data.par_iter().map(|&x| x * x).sum();
    
    // par_iter_mut(): &mut T — модифікація на місці
    let mut mutable = vec![1.0, 2.0, 3.0, 4.0];
    mutable.par_iter_mut().for_each(|x| *x = x.sqrt());
    
    // into_par_iter(): T — споживання
    let strings: Vec<String> = (0..100)
        .into_par_iter()
        .map(|i| format!("item-{}", i))
        .collect();
}
```

## 43.3. Work stealing: чому Rayon ефективний

Наївний підхід до паралелізму: розділити N елементів на K частин (по кількості ядер), кожному ядру — своя частина. Проблема: якщо одна частина обробляється довше (елементи різної складності) — одне ядро працює, інші чекають. Ефективність падає.

Rayon використовує work-stealing scheduler. Кожен потік має свою deque (двосторонню чергу) задач. Потік бере задачі зі свого кінця. Коли його deque порожній — він "краде" задачу з протилежного кінця deque іншого потоку. Результат: поки є незавершена робота — всі ядра зайняті. Балансування — автоматичне, без участі програміста.

Rayon рекурсивно розбиває роботу: вектор з мільйона елементів → дві половини → кожна ще на дві → і так до порогу (зазвичай кілька тисяч елементів на chunk). Кожен рівень розбиття — задача у deque. Потоки забирають задачі та обробляють. Якщо один потік швидко обробив свої chunks — він "краде" у сусіда.

Вам не потрібно налаштовувати work stealing — Rayon робить це автоматично. Достатньо знати: "Rayon ефективно розподіляє нерівномірну роботу по ядрах".

## 43.4. join: паралельне виконання двох задач

`rayon::join(task_a, task_b)` — запустити дві функції паралельно та чекати обидві:

```rust
use rayon;

fn expensive_compute_a() -> u64 {
    (1..=50_000_000_u64).sum()
}

fn expensive_compute_b() -> f64 {
    (1..=10_000_000).map(|i| (i as f64).sqrt()).sum()
}

fn main() {
    let (result_a, result_b) = rayon::join(
        || expensive_compute_a(),
        || expensive_compute_b(),
    );
    
    println!("A = {}, B = {:.2}", result_a, result_b);
}
```

join — для task parallelism (два різних обчислення). par_iter — для data parallelism (одне обчислення на багатьох даних). join рекурсивний: всередині task_a можна викликати ще один join — Rayon ефективно розподілить по ядрах.

## 43.5. Практика: паралельні обчислення для рою БПЛА

Перша задача — паралельний розрахунок маршрутів:

```rust
use rayon::prelude::*;

fn calculate_route(from: (f64, f64), to: (f64, f64)) -> Vec<(f64, f64)> {
    // Імітація складного обчислення
    let steps = 100;
    (0..=steps).map(|i| {
        let t = i as f64 / steps as f64;
        (
            from.0 + (to.0 - from.0) * t,
            from.1 + (to.1 - from.1) * t,
        )
    }).collect()
}

fn main() {
    let agents: Vec<((f64, f64), (f64, f64))> = (0..100)
        .map(|i| ((i as f64, 0.0), (50.0 + i as f64, 100.0)))
        .collect();
    
    let start = std::time::Instant::now();
    
    // Паралельно обчислюємо маршрути для всіх 100 агентів
    let routes: Vec<Vec<(f64, f64)>> = agents.par_iter()
        .map(|&(from, to)| calculate_route(from, to))
        .collect();
    
    println!("{} маршрутів за {:?}", routes.len(), start.elapsed());
    println!("Перший маршрут: {} точок", routes[0].len());
}
```

Друга задача — паралельний аналіз даних сенсорів:

```rust
use rayon::prelude::*;

fn analyze_readings(readings: &[f64]) -> (f64, f64, f64) {
    if readings.is_empty() { return (0.0, 0.0, 0.0); }
    let sum: f64 = readings.iter().sum();
    let avg = sum / readings.len() as f64;
    let min = readings.iter().cloned().fold(f64::INFINITY, f64::min);
    let max = readings.iter().cloned().fold(f64::NEG_INFINITY, f64::max);
    (avg, min, max)
}

fn main() {
    // 50 агентів, кожен з 10000 показань
    let all_readings: Vec<Vec<f64>> = (0..50)
        .map(|i| (0..10000).map(|j| (i * 1000 + j) as f64 * 0.01).collect())
        .collect();
    
    // Паралельний аналіз: кожен агент — окреме ядро
    let stats: Vec<(f64, f64, f64)> = all_readings.par_iter()
        .map(|readings| analyze_readings(readings))
        .collect();
    
    println!("Статистика {} агентів:", stats.len());
    for (i, (avg, min, max)) in stats.iter().enumerate().take(5) {
        println!("  Agent-{}: avg={:.1}, min={:.1}, max={:.1}", i, avg, min, max);
    }
}
```

par_iter розподіляє 50 аналізів по ядрах. Кожен analyze_readings — CPU-bound (sum, min, max по 10000 елементів). На 8 ядрах — ~6x прискорення.

## 43.6. Інтеграція Rayon з Tokio

CPU-bound обчислення не можна виконувати безпосередньо в async коді — вони блокують runtime. Рішення: `spawn_blocking` + Rayon:

```rust
use rayon::prelude::*;

#[tokio::main]
async fn main() {
    // Async: отримуємо дані (імітація мережі)
    let data: Vec<f64> = tokio::task::spawn_blocking(|| {
        // CPU-bound: паралельна обробка через Rayon
        (0..1_000_000)
            .into_par_iter()
            .map(|i| (i as f64).sqrt())
            .collect()
    }).await.unwrap();
    
    println!("Оброблено {} елементів", data.len());
    
    // Async: відправляємо результат (імітація мережі)
    // ...
}
```

`spawn_blocking` виносить замикання на окремий потік. Всередині — Rayon паралелізм. Tokio runtime не блокується. Це стандартний патерн для комбінування async IO та CPU-обчислень.

## Troubleshooting

### Помилка 1: `the trait ParallelIterator is not implemented`

Забули `use rayon::prelude::*`.

### Помилка 2: par_iter повільніший за iter

Обробка елемента занадто дешева — overhead розподілу перевищує виграш. Rayon окупається при мікросекундах+ на елемент або мільйонах елементів.

### Помилка 3: `cannot borrow as mutable` у par_iter closure

Спроба модифікувати спільну змінну з паралельних потоків:

```rust
let mut sum = 0;
data.par_iter().for_each(|&x| sum += x); // ПОМИЛКА: data race
```

Виправлення: використати `.sum()`, `.reduce()`, або atomic types. par_iter closure не може мутувати зовнішні змінні — це data race.

## Додатково

### Коли Rayon, коли threads, коли Tokio

| Задача | Інструмент |
|--------|-----------|
| Обробка великого масиву даних | Rayon par_iter |
| Кілька незалежних CPU-обчислень | Rayon join або threads |
| Мережеві запити, IO | Tokio async |
| Background обчислення в async | Tokio spawn_blocking + Rayon |
| Один довгий CPU-потік | std::thread |

### Rayon ThreadPool

За замовчуванням Rayon створює глобальний thread pool із кількістю потоків = кількість ядер. Можна налаштувати:

```rust
rayon::ThreadPoolBuilder::new()
    .num_threads(4)
    .build_global()
    .unwrap();
```

Або створити ізольований pool для конкретної задачі:

```rust
let pool = rayon::ThreadPoolBuilder::new().num_threads(2).build().unwrap();
pool.install(|| {
    // Код всередині використовує цей pool, не глобальний
    let sum: u64 = (0..1000000).into_par_iter().sum();
});
```

## Контрольні запитання

### Рівень 1

1. Чим Rayon відрізняється від Tokio?

Відповідь: Rayon — для CPU-bound паралелізму (розподіл обчислень по ядрах). Tokio — для IO-bound конкурентності (тисячі задач, що чекають мережу/файли). Rayon використовує sync код, Tokio — async/await.

2. Як зробити iter() паралельним?

Відповідь: замінити `.iter()` на `.par_iter()` (потрібен `use rayon::prelude::*`). API адаптерів (map, filter, collect, sum) ідентичний.

### Рівень 2

3. Що таке work stealing?

Відповідь: стратегія балансування навантаження. Кожен потік має свою чергу задач. Коли потік завершив свої — він "краде" задачу з черги іншого потоку. Результат: всі ядра зайняті, поки є робота, без ручного розподілу.

4. Чому par_iter може бути повільнішим за iter?

Відповідь: overhead розподілу (створення задач, синхронізація, збір результатів) перевищує виграш від паралелізму, коли обробка одного елемента дуже дешева. Rayon окупається при мікросекундах+ на елемент.

### Рівень 3

5. Як використати Rayon всередині Tokio async?

Відповідь: `tokio::task::spawn_blocking(|| { rayon_code })` виносить CPU-bound код на окремий потік. Rayon працює всередині spawn_blocking, Tokio runtime не блокується.

### Рівень 4

6. Порівняйте три підходи для обчислення маршрутів 100 агентів: один потік, 100 std::thread, Rayon par_iter.

Відповідь: один потік — 1с (послідовно). 100 threads — ~130мс але 800МБ стеків і overhead створення/знищення потоків. Rayon — ~130мс, 8 потоків (= ядра), мінімальний overhead, work stealing балансує нерівномірні обчислення. Rayon найефективніший: менше пам'яті, менше overhead, автоматичне балансування.

---

## Резюме

Rayon — паралелізм даних для CPU-bound задач. `par_iter()` замість `iter()` — і обробка розподіляється по ядрах. API ідентичний звичайним ітераторам.

Work stealing забезпечує автоматичне балансування: потоки, що завершили свою частину, "крадуть" роботу у зайнятих.

`join(task_a, task_b)` — паралельне виконання двох задач. par_iter — data parallelism (одна операція на багатьох даних).

Rayon для CPU-bound, Tokio для IO-bound. Комбінація: `spawn_blocking(|| rayon_code)` всередині async.

Rayon окупується при мікросекундах+ на елемент. Для тривіальних операцій — overhead перевищує виграш.

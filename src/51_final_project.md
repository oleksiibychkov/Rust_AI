# Розділ 51. Фінальний проєкт: Автономний рій БПЛА

## Анотація

Цей розділ формулює вимоги до фінального проєкту, надає архітектурні рекомендації та пояснює процес розробки. Проєкт інтегрує всі концепції курсу: від ownership до async actors, від базових типів до мережевої комунікації. Студент має повну свободу в архітектурних рішеннях — єдине обмеження: обов'язкові компоненти та промпт-журнал.

---

## 51.1. Вимоги до проєкту

### Обов'язкові компоненти

1. Рій з N агентів-акторів на Tokio (N >= 5).
2. Координатор-актор із обробкою звітів та видачею команд.
3. Машина станів для кожного агента (мінімум 4 стани: Idle, Active, Returning, Emergency).
4. Дерево рішень або стратегія поведінки (closures або enum-based).
5. Комунікація через async channels (мінімум 2 типи: mpsc + broadcast або watch).
6. Серіалізація стану через serde (збереження/відновлення checkpoint).
7. Спільні ресурси через Arc (мінімум один: спільна карта або конфігурація).
8. Обробка помилок через thiserror (ієрархія AgentError, NavigationError тощо).
9. Тести: мінімум 15 (unit + integration).
10. Логування через tracing (#[instrument], рівні, spans).
11. Документація: rustdoc для публічних типів, README.md з описом архітектури.
12. Модульна архітектура: мінімум 5 модулів (agent, coordinator, messages, error, config).

### Рекомендовані (бонусні) компоненти

- Rayon для CPU-bound обчислень (розрахунок маршрутів).
- TCP-комунікація (координатор та агенти — окремі процеси).
- Макрос для генерації boilerplate акторів.
- Graceful shutdown через CancellationToken.
- Метрики: кількість повідомлень, час виконання, покриття зони.
- Візуалізація (текстова карта зони у терміналі).

## 51.2. Архітектурні рекомендації

### Структура проєкту

```
swarm_project/
├── Cargo.toml
├── README.md
├── src/
│   ├── main.rs              // ініціалізація, запуск акторів
│   ├── config.rs             // MissionConfig, завантаження з TOML
│   ├── error.rs              // AgentError, NavigationError, ...
│   ├── messages.rs           // AgentReport, Command, CoordinatorMessage
│   ├── coordinator/
│   │   ├── mod.rs
│   │   ├── actor.rs          // CoordinatorActor
│   │   └── handle.rs         // CoordinatorHandle
│   ├── agent/
│   │   ├── mod.rs
│   │   ├── actor.rs          // AgentActor
│   │   ├── handle.rs         // AgentHandle
│   │   ├── state_machine.rs  // AgentState transitions
│   │   └── strategy.rs       // Decision tree / closures
│   ├── world/
│   │   ├── mod.rs
│   │   ├── map.rs            // SharedMap: Arc<RwLock<HashMap>>
│   │   └── position.rs       // Position, HasPosition
│   └── telemetry/
│       ├── mod.rs
│       ├── checkpoint.rs     // save/load стану через serde
│       └── stats.rs          // агрегація метрик
├── tests/
│   ├── agent_tests.rs
│   ├── coordinator_tests.rs
│   └── integration_tests.rs
└── config/
    └── mission.toml
```

## 51.3. Процес розробки (рекомендований)

### Тиждень 1: Проєктування

1. Визначити архітектуру: які актори, які канали, які повідомлення.
2. Створити enum для всіх типів повідомлень.
3. Визначити машину станів агента.
4. Написати README.md з описом архітектури.
5. PE: попросити AI review архітектури, зафіксувати у промпт-журналі.

### Тиждень 2: Каркас

1. Створити модулі, struct-и, Handle-и.
2. async fn run() з порожнім match для кожного актора.
3. main.rs з ініціалізацією та запуском.
4. Перевірити: компілюється, актори запускаються, канали працюють.
5. PE: попросити AI згенерувати scaffolding, адаптувати під свою архітектуру.

### Тиждень 3: Логіка

1. Заповнити handle_message для кожного актора.
2. Реалізувати state machine transitions.
3. Реалізувати стратегію/дерево рішень.
4. Додати serde checkpoint (save/load).
5. PE: попросити AI review коду, дебаг помилок.

### Тиждень 4: Якість

1. Написати тести (unit + integration).
2. Додати tracing (#[instrument], рівні, spans).
3. Написати rustdoc документацію.
4. Запустити clippy, виправити warnings.
5. PE: попросити AI згенерувати тести та документацію, оцінити якість.

## 51.4. Обов'язкова PE-складова

### Промпт-журнал

Мінімум 10 записів протягом розробки. Для кожного:

| Поле | Опис |
|------|------|
| Дата | Коли |
| Етап | Проєктування / Каркас / Логіка / Тести / Документація |
| Задача | Що потрібно було зробити |
| Промпт | Повний текст промпту до AI |
| Відповідь AI | Ключові моменти (не повний текст) |
| Рішення | Прийнято / Відхилено / Модифіковано |
| Обґрунтування | Чому саме це рішення |
| Що навчився | Інсайт від взаємодії з AI |

### Фінальний аналіз AI-ефективності

На основі промпт-журналу:
- Де AI допоміг найбільше? (зазвичай: дебаг, boilerplate, документація)
- Де AI нашкодив або помилився? (зазвичай: архітектурні рішення без контексту, галюцинації)
- Яка стратегія промптингу найефективніша для вас?
- Як би ви змінили свій workflow з AI на наступному проєкті?

## 51.5. Критерії оцінювання

| Критерій | Вага | Що оцінюється |
|----------|------|---------------|
| Архітектура та якість коду | 25% | Модульність, actors, channels, error handling, idiomatic Rust |
| Функціональність | 20% | Рій працює, стани переключаються, комунікація, checkpoint |
| Тести та документація | 20% | 15+ тестів, rustdoc, README, clippy clean |
| Якість промптів | 20% | Різноманітність (архітектура, дебаг, review, тести), ефективність |
| Промпт-журнал | 10% | 10+ записів, аналіз прийнятих/відхилених, фінальний аналіз |
| Захист | 5% | Пояснення рішень, відповіді на запитання |

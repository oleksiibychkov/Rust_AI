# Додаток A. Prompt Cookbook: повна бібліотека промпт-шаблонів

Цей додаток — систематизована колекція промпт-шаблонів для кожного етапу розробки на Rust з AI-асистентом. Кожен шаблон перевірений на практиці та оптимізований для отримання якісних відповідей.

Як користуватись: знайдіть категорію за вашою поточною задачею, оберіть відповідний шаблон, замініть [плейсхолдери] на ваш код/контекст, відправте AI. Оцініть відповідь критично — AI може помилятись.

---

## Категорія 1: Дебаг помилок компілятора

### 1.1. Загальний дебаг-шаблон

```
Мій код:
[вставити код]

Помилка компілятора:
[вставити ПОВНИЙ текст помилки, включаючи номери рядків]

1. Поясни простими словами, що ця помилка означає.
2. Покажи конкретний рядок, де проблема.
3. Поясни ЧОМУ це помилка (яке правило Rust порушено).
4. Покажи виправлений код.
5. Поясни, як уникнути цієї помилки у майбутньому.
```

Коли: будь-яка помилка компілятора, яку ви не розумієте.
Порада: завжди вставляйте ПОВНИЙ текст помилки, не скорочуйте. Номери рядків та "help:" підказки критичні для AI.

### 1.2. Помилки borrow checker

```
Мій код не компілюється через borrow checker:
[код]

Помилка: [помилка]

Покроково:
1. Покажи діаграму lifetimes: яка змінна створюється коли,
   яке посилання на що вказує, коли що знищується.
2. Покажи точку конфлікту: де два позичання перетинаються.
3. Запропонуй 2-3 варіанти виправлення з trade-offs кожного:
   - clone (простий, але overhead)
   - реструктуризація (оптимальний, але більше змін)
   - інший підхід (Arc/Rc якщо потрібно)
```

Коли: "cannot borrow as mutable because also borrowed as immutable", "borrowed value does not live long enough", "use of moved value".

### 1.3. Помилки lifetime

```
Помилка lifetime у моєму коді:
[код]

Помилка: [помилка]

1. Покажи, які посилання тут є і який lifetime кожне має.
2. Застосуй три правила lifetime elision — чому вони 
   НЕ вирішують цю ситуацію.
3. Які lifetime-анотації потрібно додати і чому.
4. Покажи виправлений код з мінімальними анотаціями.
5. Чи можна переписати без посилань (owned типи)?
```

Коли: "missing lifetime specifier", "lifetime may not live long enough".

### 1.4. Помилки типів та traits

```
Помилка типу:
[код]

Помилка: [помилка]

1. Який тип очікується і який фактичний?
2. Чому конвертація не відбувається автоматично?
3. Як виправити: From/Into, as, явна конвертація?
4. Якщо "trait not implemented" — чи можна derive, 
   чи потрібно impl вручну?
```

Коли: "expected X, found Y", "the trait X is not implemented for Y", "no method named X found".

### 1.5. Помилки async

```
Помилка у моєму async коді:
[код]

Помилка: [помилка]

1. Чи пов'язана помилка з Send/Sync, lifetime, чи логікою async?
2. Якщо Send — який тип не Send і чому? Яка альтернатива?
3. Якщо lifetime — чому посилання не може пережити .await?
4. Покажи виправлення з поясненням.
```

Коли: "future cannot be sent between threads safely", "borrowed value does not live long enough" в async fn, "there is no reactor running".

### 1.6. Runtime panic

```
Мій код компілюється, але панікує при запуску:

Код: [код]
Panic: [повний текст panic, включаючи backtrace якщо є]
Вхідні дані: [які дані спричинили паніку]

1. Який рядок спричинив паніку?
2. Чому саме ці вхідні дані проблемні?
3. Як виправити: перевірка перед операцією, Option/Result, 
   або зміна логіки?
4. Чи є інші місця у коді з такою ж вразливістю?
```

Коли: "index out of bounds", "called unwrap() on None/Err", "already borrowed: BorrowMutError".

## Категорія 2: Генерація коду за специфікацією

### 2.1. Struct з методами

```
Створи Rust struct:

Назва: [назва]
Поля: 
  - [поле1]: [тип] — [опис]
  - [поле2]: [тип] — [опис]
  ...

Методи:
  - new([параметри]) -> Self
  - [метод1]([параметри]) -> [повернення] — [опис поведінки]
  - [метод2]([параметри]) -> [повернення] — [опис поведінки]

Derive: Debug, Clone, [інші якщо потрібні]
Display: [формат виводу]

Додай:
- Валідацію у new() (повертає Result)
- Документацію (/// для кожного методу)
- 3 тести
```

Коли: створення нового типу з нуля.

### 2.2. Enum з варіантами

```
Створи Rust enum:

Назва: [назва]
Варіанти:
  - [Variant1] — unit (без даних)
  - [Variant2]([тип]) — tuple 
  - [Variant3] { [поле]: [тип], [поле]: [тип] } — struct
  ...

Реалізуй:
- Display (зрозумілий текст для кожного варіанту)
- thiserror::Error (якщо це enum помилок)
- From для автоматичної конвертації з [тип помилки]
- Метод [назва](&self) -> [тип] через match
```

Коли: створення enum для станів, повідомлень, або помилок.

### 2.3. Scaffold актора (Actor Model)

```
Створи actor на Tokio:

Назва: [назва]Actor
Стан актора:
  - [поле1]: [тип]
  - [поле2]: [тип]

Типи повідомлень (enum [назва]Message):
  - [Variant1] { [поля] } — [опис що робити]
  - [Variant2] { [поля] } — [опис що робити]
  - GetStatus { reply: oneshot::Sender<[тип]> }
  - Shutdown

Handle: [назва]Handle з методами:
  - [method1]([параметри]) — відправляє [Variant1]
  - [method2]([параметри]) — відправляє [Variant2]
  - get_status() -> [тип] — request-response через oneshot
  - shutdown()

Додай:
- async fn run() з while let Some(msg) = receiver.recv().await
- #[instrument] для tracing
- Обробку кожного варіанту у match
```

Коли: додавання нового актора у систему.

### 2.4. Trait з реалізаціями

```
Створи trait:

Назва: [назва]
Методи:
  - [method1](&self) -> [тип] — обов'язковий
  - [method2](&self) -> [тип] — обов'язковий  
  - [default_method](&self) -> [тип] — default реалізація через method1/method2

Реалізуй для 2 типів:
  1. [Тип1] з полями [список] — [опис логіки]
  2. [Тип2] з полями [список] — [опис логіки]

Покажи:
- Використання через &dyn Trait
- Використання через generics <T: Trait>
- Тести для обох реалізацій
```

Коли: створення абстракції з кількома реалізаціями.

### 2.5. Функція з обробкою помилок

```
Напиши функцію:

Сигнатура: fn [назва]([параметри]) -> Result<[тип], [ErrorType]>
Поведінка: [опис кроків]

Можливі помилки:
  - [коли1] → [ErrorVariant1]
  - [коли2] → [ErrorVariant2]
  - [коли3] → [ErrorVariant3]

Використовуй:
- Оператор ? для пропагації
- map_err для додавання контексту
- Жоден unwrap
- Інформативні повідомлення у кожному Err
```

Коли: будь-яка функція, що може провалитись.

### 2.6. CLI-програма

```
Створи CLI-програму на Rust:

Опис: [що робить програма]
Аргументи:
  - [arg1]: [тип] — [опис]
  - [arg2]: [тип] — [опис] (опціональний, default: [значення])

  1. [крок 1]

  3. [крок 3]

Використовуй clap для парсингу аргументів.

```

Коли: створення утиліти командного рядка.

## Категорія 3: Рефакторинг

### 3.1. Цикли → ітератори

```
Перепиши цей цикл через ітератори:
[код з for/while та мутабельними змінними]

Для кожної трансформації:
1. Який адаптер ітератора замінює цей крок 
   (map, filter, fold, flat_map, enumerate, zip, etc.)?
2. Чому саме цей адаптер?
3. Покажи проміжний та фінальний варіант.

Не використовуй collect() де можна обійтись 
без створення нової колекції.
```

### 3.2. unwrap → Result

```
Ось код з unwrap():
[код]

Для кожного unwrap():
1. Що може піти не так (коли буде None/Err)?
2. Який метод обрати: 
   - unwrap_or(default) — фіксоване значення
   - unwrap_or_else(|| ...) — ледаче обчислення
   - ? — пропагація нагору
   - match/if let — обробка на місці
   - ok_or("msg") — Option → Result
3. Чому саме цей метод для цього випадку?

Покажи фінальний код без жодного unwrap.
```

### 3.3. Дублювання → generic

```
Ось 2-3 функції з дуже схожою логікою:
[функція 1]
[функція 2]
[функція 3]

Узагальни через одну generic-функцію:
1. Що спільного між функціями?
2. Чим вони відрізняються? Це параметр типу чи замикання?
3. Які trait bounds потрібні? Мінімальні!
4. Покажи generic-версію.
5. Покажи, що вона працює для всіх трьох випадків.
```

### 3.4. Вкладений match → комбінатори

```
Перепиши вкладений match/if через комбінатори:
[код з 3+ рівнями вкладеності]

Використовуй:
- map, and_then, or_else, filter для Option/Result
- ? для пропагації
- Ланцюжок комбінаторів (pipeline)

Покажи покрокову трансформацію:
- Крок 1: зовнішній match → ?
- Крок 2: внутрішній match → and_then
- ... 
- Фінальний: один ланцюжок
```

### 3.5. String → thiserror enum

```
Ось код де помилки — String:
[код з Result<T, String>]

Рефактор:
1. Визнач всі категорії помилок (які String-повідомлення є).
2. Створи enum з thiserror для кожної категорії.
3. Додай #[from] де можливо.
4. Заміни всі Err(String::from(...)) на Err(MyError::Variant).
5. Покажи, як caller тепер може match на конкретні помилки.
```

### 3.6. Struct → Actor

```
Ось struct з мутабельними методами:
[struct з impl, методи приймають &mut self]

Рефактор у actor на Tokio:
1. Кожен pub метод → варіант enum Message.
2. &mut self методи → обробка у match всередині run().
3. Публічний API → Handle з async методами.
4. Методи з поверненням → oneshot у Message.
5. Покажи before/after та поясни переваги.
```

### 3.7. Послідовний → паралельний

```
Ось послідовний код:
[код з циклом, де кожна ітерація незалежна]

Зроби паралельним:
1. Чи це CPU-bound (Rayon) чи IO-bound (Tokio)?
2. Які дані потрібно clone/move?
3. Як зібрати результати (collect, join, channel)?
4. Покажи паралельну версію.
5. Оціни очікуваний speedup на N ядрах.
```

## Категорія 4: Тестування

### 4.1. Повний набір тестів для функції

```
Ось функція:
[код функції]

Згенеруй тести:
1. Happy path: типові вхідні дані, очікуваний результат.
2. Edge cases:
   - Порожній вхід (порожній вектор, порожній рядок, None)
   - Одиничний елемент
   - Максимальне/мінімальне значення (MAX, MIN, 0, -1)
   - Граничні значення (рівно на межі: == threshold)
3. Error cases: кожен варіант Err з перевіркою типу помилки.
4. Для кожного тесту: назва, що перевіряє, assert.

Використовуй #[test], assert_eq!, assert!(matches!(...)).
```

### 4.2. Тести для актора

```
Ось actor + handle:
[код Actor та Handle]

Згенеруй тести:
1. Створення актора: new() повертає handle + actor.
2. Базова операція: відправити повідомлення → перевірити стан 
   через get_status (oneshot request-response).
3. Послідовність: кілька повідомлень → перевірити кумулятивний ефект.
4. Shutdown: відправити Shutdown → actor task завершується.
5. Канал закритий: actor завершується коли всі handles drop.

Використовуй #[tokio::test] для async тестів.
```

### 4.3. Пошук пропущених edge cases

```
Ось мої тести:
[існуючі тести]

Для функції:
[код функції]

Які edge cases я пропустив?
Для кожного пропущеного:
1. Опис сценарію
2. Чому це може зламатись
3. Готовий тест
```

### 4.4. Property-based тести

```
Ось функція:
[код]

Згенеруй property-based тести з proptest:
1. Визнач інваріанти (що ЗАВЖДИ true для будь-якого вхідного):
   - Наприклад: sort(v).len() == v.len()
   - Наприклад: parse(serialize(x)) == x (roundtrip)
2. Для кожного інваріанту — proptest macro
3. Визнач стратегії генерації вхідних даних

Формат:
proptest! {
    #[test]
    fn prop_[назва](input in [стратегія]) {
        prop_assert!([інваріант]);
    }
}
```

### 4.5. Benchmark

```
Ось функція, яку хочу бенчмаркувати:
[код]

Створи benchmark з criterion:
1. Базовий benchmark з типовими даними.
2. Benchmark з різними розмірами вхідних (10, 100, 1000, 10000).
3. Порівняння двох реалізацій (наприклад, iter vs par_iter).

Cargo.toml:
[[bench]]
name = "my_benchmark"
harness = false

[dev-dependencies]
criterion = { version = "0.5", features = ["html_reports"] }
```

## Категорія 5: Code Review

### 5.1. Повний review модуля

```
Переглянь цей Rust-модуль:
[код модуля]

Для кожної знайденої проблеми:
- Категорія: Bug / Performance / ErrorHandling / Style / Safety
- Серйозність: Critical / Warning / Info
- Рядок(и): де проблема
- Опис: що не так
- Fix: конкретний виправлений код

Перевір конкретно:
- Чи є unwrap/expect у non-test коді?
- Чи є потенційні паніки (індексація без перевірки)?
- Чи коректна обробка помилок (чи не ковтаються Err)?
- Чи є зайві clone (де достатньо &)?
- Чи ідіоматичний стиль (ітератори де доречно)?
- Чи є зайві mut (змінна оголошена mut, але не мутується)?
```

### 5.2. Review async коду

```
Переглянь цей async код на Tokio:
[код]

Перевір конкретно:
1. Чи є блокуючі операції (std::thread::sleep, sync IO, 
   тривалі CPU-обчислення без spawn_blocking)?
2. Чи є потенційні deadlocks (кілька lock у різному порядку)?
3. Чи мінімальний scope lock (Mutex/RwLock тримається через .await)?
4. Чи всі канали правильно закриваються (drop sender)?
5. Чи є потенційний memory leak (unbounded channel з швидким producer)?
6. Чи правильно використаний select! (чи всі гілки коректні)?
7. std::sync::Mutex vs tokio::sync::Mutex — правильний вибір?
```

### 5.3. Review обробки помилок

```
Переглянь обробку помилок:
[код]

Перевір:
1. Чи є unwrap/expect у production коді? Замінити на що?
2. Чи інформативні повідомлення у Err? Чи можна зрозуміти 
   причину помилки без дебагера?
3. Чи правильна ієрархія помилок (модуль → агент → місія)?
4. Чи є #[from] де потрібно (для автоматичної конвертації через ?)?
5. Чи не ковтаються помилки (let _ = result, .ok() без обробки)?
6. Чи є map_err з контекстом для низькорівневих помилок?
```

### 5.4. Review на ідіоматичність

```
Чи ідіоматичний цей Rust-код?
[код]

Перевір по категоріях:
- Naming: snake_case для fn/var, CamelCase для types, SCREAMING для const?
- Ітератори: де цикл можна замінити на iter().map().filter().collect()?
- Pattern matching: де if/else можна замінити на match чи if let?
- Ownership: де clone можна уникнути через &?
- Error handling: idiomatic (?/map_err/thiserror) чи ad-hoc (String, unwrap)?
- Derives: чи всі корисні traits derive (Debug, Clone, PartialEq)?
- Публічність: чи мінімальний pub surface (що не потрібно зовні — private)?

Для кожного покращення — before/after код.
```

### 5.5. Security review

```
Переглянь на безпеку:
[код]

Перевір:
1. Чи є unsafe? Якщо так — чи обґрунтований, чи є # Safety, 
   чи мінімальний scope?
2. Чи є panic paths у production коді?
3. Чи є integer overflow без checked/saturating?
4. Чи валідуються зовнішні дані (user input, network, file)?
5. Чи є потенційні DoS-вектори (unbounded allocation, 
   нескінченні цикли при певному вході)?
6. Чи не логуються чутливі дані (паролі, токени)?
```

## Поради щодо промптингу

Загальні принципи ефективного промптингу для Rust:

Будьте конкретними. "Напиши функцію" — поганий промпт. "Напиши функцію fn parse_waypoint(line: &str) -> Result<(f64, f64), ParseError>, що парсить рядок 'x,y' з валідацією координат [-90, 90] для широти" — добрий промпт.

Давайте контекст. AI не бачить ваш проєкт. Вставляйте: код, помилки, типи, що використовуються, обмеження, мету.

Просіть пояснення. Не лише код — а чому саме такий код. "Поясни кожне рішення" змушує AI обґрунтовувати, і ви побачите, якщо логіка хибна.

Ітеруйте. Перший промпт рідко дає ідеальний результат. "Добре, але зміни X на Y" або "Це не враховує Z — доповни" — нормальний workflow.

Перевіряйте. Завжди компілюйте та тестуйте код від AI. Навіть ідеально виглядаючий код може не компілюватись або мати тонкі баги. AI — помічник, не оракул.
